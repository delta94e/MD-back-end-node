```
┌────────────────────────────────────────────────────┐
│                  REDIS DATA STRUCTURES              │
│                                                    │
│  String     "hello"                 ← cache, counters
│  List       [a, b, c, d]           ← queues, recent items
│  Set        {a, b, c}              ← unique items, tags
│  Sorted Set {a:1, b:2, c:3}       ← leaderboards, rankings
│  Hash       {field1:v1, field2:v2} ← objects, user session
│  Stream     event log              ← message queue (like Kafka)
│  Bitmap     01101001               ← presence, feature flags
│  HyperLogLog                       ← cardinality estimation
│  Geo        lat/long               ← nearby locations
└────────────────────────────────────────────────────┘
```

```bash
# String — cache, counters
SET user:1 '{"name":"Alice"}' EX 3600    # set with 1h TTL
INCR page:views                           # atomic increment
SETNX lock:resource "owner"               # set if NOT exists (distributed lock!)

# Sorted Set — leaderboard
ZADD leaderboard 1000 "player:1"
ZADD leaderboard 2500 "player:2"
ZREVRANGE leaderboard 0 9 WITHSCORES     # top 10 players

# Hash — user session
HSET session:abc user_id 42 role "admin"
HGETALL session:abc

# Stream — event/message queue
XADD events * user_id 42 action "login"  # append event
XREAD COUNT 10 STREAMS events 0          # read events
```

## 2. Caching Patterns

```
Cache-Aside (Lazy Loading):
  App ──▶ Cache HIT? ──▶ Return
          Cache MISS? ──▶ DB ──▶ Store in Cache ──▶ Return
  ✅ Chỉ cache data thực sự dùng
  ❌ Cache miss = cold start chậm
  ❌ Data có thể stale

Write-Through:
  App ──▶ Write Cache ──▶ Write DB (đồng thời)
  ✅ Cache luôn consistent
  ❌ Write latency tăng (write 2 nơi)
  ❌ Cache data có thể không bao giờ đọc

Write-Behind (Write-Back):
  App ──▶ Write Cache ──▶ Return (async write DB later)
  ✅ Write cực nhanh
  ❌ Data loss nếu cache crash trước khi flush

Read-Through:
  App ──▶ Cache (cache tự query DB nếu miss)
  ✅ Logic đơn giản từ app perspective
```

```
Cache Invalidation (KHÓ NHẤT!):

  TTL-based: SET key value EX 300 (5 min)
  Event-based: khi DB update → delete cache key

  ⚠️ Thundering Herd: key expire → 1000 requests cùng lúc → DB overload
  Fix: lock (chỉ 1 request rebuild cache), stale-while-revalidate

  ⚠️ Cache Stampede: hot key expire → spike DB queries
  Fix: probabilistic early expiration, lock, background refresh
```

## 3. Persistence

```
RDB (Snapshot):
  - Point-in-time snapshot → disk (binary format)
  - SAVE (blocking) vs BGSAVE (background fork)
  - ✅ Compact, fast recovery
  - ❌ Data loss: changes between snapshots lost

AOF (Append-Only File):
  - Log EVERY write operation → disk
  - Policies: always (safest), everysec (balanced), no (fastest)
  - ✅ Ít data loss (max 1 sec with everysec)
  - ❌ File lớn, recovery chậm hơn

Hybrid (Redis 4.0+):
  - RDB snapshot + AOF for operations since snapshot
  - ✅ Fast recovery + minimal data loss
```

## 4. Cluster & Sentinel

```
Redis Cluster:
  16384 hash slots distributed across nodes

  Node A: slots 0-5460
  Node B: slots 5461-10922
  Node C: slots 10923-16383

  Key → CRC16(key) mod 16384 → slot → node

  Each master has replica(s) for failover
  Resharding: move slots between nodes live

Redis Sentinel:
  Monitoring + automatic failover for non-cluster setup
  Sentinel watches Primary → Primary down → promote Replica → notify clients
```

## 5. Advanced Patterns

```bash
# Distributed Lock (Redlock simplified)
SET lock:order:123 "worker-1" NX EX 30    # acquire (NX = set if not exists)
# ... do work ...
DEL lock:order:123                         # release
# ⚠️ Redlock controversy: Martin Kleppmann argues unsafe for correctness
# → Use for efficiency (dedup), NOT for correctness-critical ops

# Rate Limiting (Sliding Window)
# INCR rate:user:42:1709300000 → count requests in current window
# EXPIRE rate:user:42:1709300000 60 → TTL 60s

# Pub/Sub
SUBSCRIBE channel:notifications           # subscriber
PUBLISH channel:notifications "new order"  # publisher
# ⚠️ Fire-and-forget: if subscriber offline → messages LOST
# → Use Redis Streams for persistence
```

## 6. Câu Hỏi Phỏng Vấn

**Q: Cache invalidation strategies? Thundering herd?**

> A: TTL-based (simple), event-based (DB update → delete cache). Thundering herd: popular key expires → 1000 concurrent requests hit DB. Fix: (1) mutex lock — 1 request rebuilds, others wait. (2) Stale-while-revalidate — serve stale, refresh async. (3) Probabilistic early expiration.

**Q: Redis single-threaded — tại sao nhanh?**

> A: (1) In-memory (RAM: ~100ns vs SSD: ~100μs). (2) Single-threaded = no locks, no context switching. (3) I/O multiplexing (epoll). (4) Efficient data structures (ziplist, skiplist). (5) Since Redis 6: I/O threads for network but commands still single-threaded.
