# §3.5 — Data Patterns

> **Mục tiêu**: Hiểu các patterns quan trọng cho production database — Connection Pooling, Read Replicas, CQRS, Event Sourcing, Zero-downtime Migrations.

---

## 1. Connection Pooling

```
Vấn đề: mỗi DB connection = 1 process (PostgreSQL)
  100 connections × ~5MB = 500MB RAM chỉ cho connections!
  Max connections default: 100

Solution: Connection Pool
  App ──▶ Pool (reuse connections) ──▶ Database

  PgBouncer (external pooler):
    Transaction mode: connection released SAU mỗi transaction
    Session mode: connection held toàn session
    Statement mode: released SAU mỗi statement

  Pool size formula (rule of thumb):
    connections = (core_count × 2) + effective_spindle_count
    8 cores + SSD → ~17 connections
    ⚠️ KHÔNG phải càng nhiều càng tốt — context switching!
```

## 2. Read Replicas

```
┌──────────┐  writes  ┌──────────┐
│   App    │─────────▶│ PRIMARY  │
│          │          │ (write)  │
│          │  reads   │          │──replication──▶ REPLICA 1
│          │─────────▶│          │──replication──▶ REPLICA 2
└──────────┘          └──────────┘

Read/Write Splitting:
  INSERT/UPDATE/DELETE → Primary
  SELECT → Replica (load balanced)

⚠️ Replication Lag: replica có thể TRỄ ~ms-seconds
  Problem: user creates order → redirect → reads → order NOT FOUND!

Fix: "Read Your Own Writes"
  1. After write → read from PRIMARY (within session)
  2. Client timestamp → if (now - writeTime < 5s) → read Primary
  3. Synchronous replication (trade-off: latency)
```

## 3. CQRS (Command Query Responsibility Segregation)

```
Traditional:
  Same model for Read AND Write → compromise both

CQRS:
  ┌───────────────────┐     ┌───────────────────┐
  │  COMMAND (Write)  │     │   QUERY (Read)     │
  │  - Normalized     │     │  - Denormalized    │
  │  - Validation     │     │  - Optimized views │
  │  - Business rules │     │  - Materialized    │
  │  Write DB         │────▶│  Read DB           │
  │  (PostgreSQL)     │sync │  (Elasticsearch    │
  └───────────────────┘     │   or Redis)        │
                            └───────────────────┘

Khi nào dùng CQRS?
  ✅ Read patterns rất khác Write patterns
  ✅ High read volume cần optimization riêng
  ✅ Different scaling cho reads vs writes
  ❌ Simple CRUD apps → OVERKILL!
```

## 4. Event Sourcing

```
Traditional: store CURRENT STATE
  users table: {id: 1, balance: 150}  ← chỉ biết balance hiện tại

Event Sourcing: store EVENTS (history)
  events: [
    {type: "AccountCreated", data: {userId: 1, balance: 0}},
    {type: "Deposited",      data: {userId: 1, amount: 200}},
    {type: "Withdrawn",      data: {userId: 1, amount: 50}},
  ]
  → Replay events: 0 + 200 - 50 = 150 ✓

✅ Full audit trail, time travel, debug
✅ Rebuild state from events (replayable)
❌ Complex, eventual consistency
❌ Event schema evolution (versioning)
❌ Query current state = replay ALL events → slow!
   Fix: Snapshots (periodically save current state)
```

## 5. Database Migrations — Zero Downtime

```
⚠️ Backward-compatible migrations:

✅ SAFE:
  1. ADD column (nullable or with default)
  2. ADD index CONCURRENTLY
  3. ADD table

❌ UNSAFE (breaks running code):
  1. DROP column — old code still references it!
  2. RENAME column — old code uses old name!
  3. Change column type — old code expects old type!

Strategy: Expand-Contract Pattern
  Phase 1 (Expand): ADD new column, backfill, deploy code that writes BOTH
  Phase 2 (Migrate): backfill old data to new column
  Phase 3 (Contract): deploy code using NEW column only, DROP old column

This takes 3 deploys but ZERO DOWNTIME!
```
