## 1. Document Model & Schema Design

```javascript
// Document = JSON-like (BSON internally)
{
  _id: ObjectId("..."),
  name: "Alice",
  email: "alice@example.com",
  address: {
    city: "HCM",         // EMBEDDED document
    zip: "70000"
  },
  orders: [               // EMBEDDED array
    { product: "Widget", qty: 2, total: 50 },
    { product: "Gadget", qty: 1, total: 100 }
  ]
}

// Schema Design: EMBEDDING vs REFERENCING
// Embedding (denormalized):
//   ✅ 1 query lấy tất cả (locality)
//   ✅ Atomic updates (1 document)
//   ❌ Document size limit 16MB
//   ❌ Data duplication
//   → Dùng khi: 1-to-few, data luôn đọc cùng nhau

// Referencing (normalized):
//   ✅ No duplication, no size limit
//   ❌ Multiple queries ($lookup = JOIN)
//   ❌ No atomic cross-document (trước v4.0)
//   → Dùng khi: 1-to-many, many-to-many, data truy cập riêng
```

## 2. Indexing

```javascript
// Single field
db.users.createIndex({ email: 1 }); // ascending

// Compound
db.orders.createIndex({ user_id: 1, created_at: -1 });
// ESR Rule: Equality → Sort → Range (column order!)

// Multikey (array fields)
db.products.createIndex({ tags: 1 }); // index each element

// Text index (full-text search)
db.articles.createIndex({ title: "text", body: "text" });
db.articles.find({ $text: { $search: "mongodb performance" } });

// TTL index (auto-delete expired documents)
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 });

// Explain
db.orders.find({ user_id: 42 }).explain("executionStats");
// COLLSCAN = no index (bad!) vs IXSCAN = index scan (good!)
```

## 3. Aggregation Pipeline

```javascript
// Aggregation = pipeline of stages (like unix pipes)
db.orders.aggregate([
  { $match: { status: "completed" } }, // filter first!
  {
    $lookup: {
      // JOIN
      from: "users",
      localField: "user_id",
      foreignField: "_id",
      as: "user",
    },
  },
  { $unwind: "$user" }, // flatten array
  {
    $group: {
      // GROUP BY
      _id: "$user.country",
      totalRevenue: { $sum: "$total" },
      orderCount: { $sum: 1 },
      avgOrder: { $avg: "$total" },
    },
  },
  { $sort: { totalRevenue: -1 } }, // ORDER BY
  { $limit: 10 }, // LIMIT
  {
    $project: {
      // SELECT fields
      country: "$_id",
      totalRevenue: 1,
      orderCount: 1,
      avgOrder: { $round: ["$avgOrder", 2] },
    },
  },
]);

// $facet: multiple pipelines trên cùng data (pagination!)
db.products.aggregate([
  {
    $facet: {
      data: [{ $skip: 20 }, { $limit: 10 }],
      total: [{ $count: "count" }],
    },
  },
]);
```

## 4. Replication & Sharding

```
Replica Set:
  ┌────────────┐
  │  PRIMARY   │ ← writes
  │  (vote)    │
  └─────┬──────┘
        │ oplog replication
  ┌─────▼──────┐  ┌────────────┐
  │ SECONDARY  │  │ SECONDARY  │  ← reads (optional)
  │  (vote)    │  │  (vote)    │
  └────────────┘  └────────────┘
  Primary down → election → new Primary (majority votes needed)

Sharding:
  Distribute data across multiple servers

  ┌────────────┐
  │  mongos    │  ← router
  │  (proxy)   │
  └─────┬──────┘
   ┌────┼────┐
   ▼    ▼    ▼
 Shard1 Shard2 Shard3

  Shard Key: determines data distribution
  ✅ High cardinality, evenly distributed
  ❌ Monotonic (ObjectId, timestamp) → all writes to 1 shard!
  ✅ Hashed shard key: even distribution
```

## 5. Transactions

```javascript
// Multi-document transactions (v4.0+)
const session = client.startSession();
session.startTransaction();
try {
  await Account.updateOne(
    { _id: from },
    { $inc: { balance: -100 } },
    { session },
  );
  await Account.updateOne({ _id: to }, { $inc: { balance: 100 } }, { session });
  await session.commitTransaction();
} catch (err) {
  await session.abortTransaction();
} finally {
  session.endSession();
}
// ⚠️ Performance impact: locks acquired, WiredTiger snapshots
// → Minimize transaction duration, prefer embedding when possible
```

## 6. MongoDB vs PostgreSQL

| Feature    | PostgreSQL                     | MongoDB                          |
| ---------- | ------------------------------ | -------------------------------- |
| Data Model | Relational (tables)            | Document (JSON)                  |
| Schema     | Strict (migrations)            | Flexible (schemaless)            |
| JOINs      | Native, efficient              | $lookup (slower)                 |
| ACID       | Full                           | Full (since v4.0)                |
| Scaling    | Vertical + read replicas       | **Horizontal (sharding)**        |
| Best for   | Complex queries, relationships | Rapid iteration, flexible schema |
| JSONB      | ✅ Supported                   | Native                           |

**Khi nào MongoDB?**: Schema thay đổi nhanh, embedded/nested data, horizontal scaling, real-time analytics, IoT/logging.

**Khi nào PostgreSQL?**: Complex relationships, strict data integrity, advanced SQL, financial data, existing SQL expertise.

## 7. Câu Hỏi Phỏng Vấn

**Q: Embedding vs Referencing — khi nào?**

> A: Embed khi: (1) data luôn đọc cùng nhau, (2) 1-to-few, (3) ít update embedded data. Reference khi: (1) many-to-many, (2) data truy cập độc lập, (3) document sẽ lớn (>16MB limit). Hybrid: embed frequently-accessed data, reference the rest.
