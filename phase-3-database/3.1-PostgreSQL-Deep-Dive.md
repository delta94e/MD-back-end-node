## 1. SQL Mastery

### 1.1 JOINs

```sql
-- Setup
-- users: {id, name}
-- orders: {id, user_id, total}

-- INNER JOIN — chỉ rows match CẢ HAI bảng
SELECT u.name, o.total
FROM users u INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN — TẤT CẢ users, kể cả chưa có orders (NULL)
SELECT u.name, COALESCE(o.total, 0) as total
FROM users u LEFT JOIN orders o ON u.id = o.user_id;

-- RIGHT JOIN = LEFT JOIN đảo ngược (hiếm dùng)
-- FULL OUTER JOIN — tất cả từ CẢ HAI bảng
-- CROSS JOIN — mọi combination (Cartesian product) — cẩn thận!

-- SELF JOIN — join bảng với chính nó
SELECT e.name AS employee, m.name AS manager
FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;
```

### 1.2 Window Functions

```sql
-- Window Functions = tính toán TRÊN nhóm rows mà KHÔNG gom lại (khác GROUP BY!)

-- ROW_NUMBER, RANK, DENSE_RANK
SELECT name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rank,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rank
FROM employees;

-- salary: 100, 100, 80, 70
-- ROW_NUMBER: 1, 2, 3, 4 (luôn unique)
-- RANK:       1, 1, 3, 4 (skip rank 2)
-- DENSE_RANK: 1, 1, 2, 3 (không skip)

-- LAG / LEAD — access row trước/sau
SELECT date, revenue,
  revenue - LAG(revenue) OVER (ORDER BY date) AS daily_change,
  LEAD(revenue) OVER (ORDER BY date) AS next_day_revenue
FROM sales;

-- Running total
SELECT date, amount,
  SUM(amount) OVER (ORDER BY date) AS running_total
FROM transactions;

-- Top N per group (classic interview question!)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn
  FROM employees
) sub WHERE rn <= 3; -- Top 3 salary per department
```

### 1.3 CTEs vs Subqueries

```sql
-- CTE (Common Table Expression) — đọc dễ hơn, recursive support
WITH active_users AS (
  SELECT * FROM users WHERE last_login > NOW() - INTERVAL '30 days'
),
user_orders AS (
  SELECT user_id, COUNT(*) as order_count
  FROM orders GROUP BY user_id
)
SELECT au.name, uo.order_count
FROM active_users au JOIN user_orders uo ON au.id = uo.user_id;

-- Recursive CTE — tree traversal, hierarchy
WITH RECURSIVE org_tree AS (
  SELECT id, name, manager_id, 0 AS level
  FROM employees WHERE manager_id IS NULL  -- root (CEO)
  UNION ALL
  SELECT e.id, e.name, e.manager_id, ot.level + 1
  FROM employees e JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT * FROM org_tree ORDER BY level;
```

## 2. Schema Design

### 2.1 Normalization

```
1NF: No repeating groups, atomic values
  ❌ tags: "js,python,go"
  ✅ Separate tags table

2NF: 1NF + no partial dependencies (composite key)
3NF: 2NF + no transitive dependencies
BCNF: 3NF + every determinant is a candidate key

Thực tế: normalize đến 3NF, denormalize có chủ đích cho performance
```

### 2.2 Many-to-Many & Polymorphic

```sql
-- Many-to-Many: junction table
CREATE TABLE user_roles (
  user_id INT REFERENCES users(id),
  role_id INT REFERENCES roles(id),
  PRIMARY KEY (user_id, role_id)
);

-- Polymorphic Association
-- Comments có thể thuộc Post HOẶC Video
CREATE TABLE comments (
  id SERIAL PRIMARY KEY,
  body TEXT,
  commentable_type VARCHAR(50),  -- 'Post' hoặc 'Video'
  commentable_id INT
);
-- ⚠️ Không thể FK constraint! Fix: separate tables hoặc check constraint
```

## 3. Indexing Deep Dive

```
Index Types trong PostgreSQL:

┌─────────────────────────────────────────────────┐
│  B-tree (DEFAULT)                                │
│  - Equality (=) + Range (<, >, BETWEEN)          │
│  - ORDER BY, GROUP BY                            │
│  - LIKE 'prefix%' (NOT '%suffix')               │
│  - Phù hợp: hầu hết queries                     │
├─────────────────────────────────────────────────┤
│  Hash                                            │
│  - Chỉ equality (=)                              │
│  - Nhẹ hơn B-tree cho equality-only              │
├─────────────────────────────────────────────────┤
│  GIN (Generalized Inverted Index)                │
│  - Full-text search, JSONB, Arrays               │
│  - Contains (@>), Overlap (&&)                   │
│  - Chậm insert, nhanh search                     │
├─────────────────────────────────────────────────┤
│  GiST (Generalized Search Tree)                  │
│  - Geometric, range types, full-text             │
│  - Nearest neighbor, containment                  │
├─────────────────────────────────────────────────┤
│  BRIN (Block Range Index)                         │
│  - Very large tables, physically ordered data     │
│  - Timestamp columns (append-only data)          │
│  - Rất nhỏ, rất nhanh scan                       │
└─────────────────────────────────────────────────┘
```

```sql
-- Composite index: COLUMN ORDER MATTERS!
CREATE INDEX idx_user_status_date ON orders(status, created_at);
-- ✅ WHERE status = 'active' AND created_at > '2024-01-01'
-- ✅ WHERE status = 'active' (left prefix)
-- ❌ WHERE created_at > '2024-01-01' (skip first column!)

-- Partial index: chỉ index subset
CREATE INDEX idx_active_users ON users(email) WHERE active = true;
-- Nhỏ hơn, nhanh hơn full index (chỉ ~10% active users)

-- Expression index
CREATE INDEX idx_lower_email ON users(LOWER(email));
-- ✅ WHERE LOWER(email) = 'alice@example.com'

-- Index-only scan (covering index)
CREATE INDEX idx_orders_cover ON orders(user_id, status) INCLUDE (total);
-- Query chỉ cần data trong index → KHÔNG cần đọc table (heap)!
```

## 4. Query Optimization

### EXPLAIN ANALYZE

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;

-- Output giải thích:
-- Seq Scan on orders  (cost=0.00..1234.00 rows=100 width=48)
--                      (actual time=0.5..12.3 rows=95 loops=1)
--   Filter: (user_id = 42)
--   Rows Removed by Filter: 9905
-- Planning Time: 0.1 ms
-- Execution Time: 12.5 ms

-- cost: estimated (startup..total)
-- rows: estimated vs actual → lệch nhiều = outdated statistics!
-- Fix: ANALYZE orders; (update statistics)
```

### Scan Types

```
Sequential Scan: đọc TOÀN BỘ bảng (no index)
  → OK cho small tables hoặc SELECT phần lớn rows

Index Scan: dùng index tìm rows → đọc heap (table)
  → Tốt khi trả ít rows (<~5-10% of table)

Bitmap Index Scan: index → bitmap → heap
  → Tốt khi trả nhiều rows hơn Index Scan nhưng ít hơn Seq Scan

Index Only Scan: chỉ đọc index (covering index)
  → Nhanh nhất! Không cần đọc heap
```

### Join Algorithms

```
Nested Loop: O(N × M) — tốt khi 1 bảng nhỏ + có index
Hash Join: O(N + M) — build hash table từ bảng nhỏ, probe bảng lớn
Merge Join: O(N log N + M log M) — sort cả 2 + merge. Tốt khi đã sorted
```

## 5. Transactions & MVCC

### 5.1 ACID

```
Atomicity:    ALL or NOTHING — transaction hoàn thành hoàn toàn hoặc rollback
Consistency:  DB luôn ở trạng thái hợp lệ (constraints, triggers)
Isolation:    Transactions không thấy partial changes của nhau
Durability:   Committed data KHÔNG BAO GIỜ mất (WAL → disk)
```

### 5.2 Isolation Levels & Anomalies

```
                    Dirty    Non-repeatable  Phantom   Write
                    Read     Read            Read      Skew
Read Uncommitted    ✅ có     ✅ có           ✅ có     ✅ có
Read Committed      ❌ không  ✅ có           ✅ có     ✅ có
Repeatable Read     ❌        ❌ không        ✅ có*    ✅ có*
Serializable        ❌        ❌              ❌ không  ❌ không

* PostgreSQL RR thực tế dùng Snapshot Isolation → ngăn Phantom
  nhưng KHÔNG ngăn Write Skew (khác MySQL!)

Dirty Read: đọc data CHƯA committed (transaction khác chưa xong)
Non-repeatable Read: đọc 2 lần cùng row → kết quả khác (bị UPDATE)
Phantom Read: đọc 2 lần cùng query → thêm/bớt rows (bị INSERT/DELETE)
Write Skew: 2 transactions đọc overlapping data → cả 2 write → vi phạm constraint
```

### 5.3 MVCC

```
PostgreSQL MVCC (Multi-Version Concurrency Control):
  Mỗi row có: xmin (transaction ID tạo), xmax (transaction ID xóa/update)

  Transaction 100: INSERT INTO users VALUES (1, 'Alice')
  Row: {data: 'Alice', xmin: 100, xmax: null}

  Transaction 200: UPDATE users SET name = 'Bob' WHERE id = 1
  Old row: {data: 'Alice', xmin: 100, xmax: 200}  ← marked "dead"
  New row: {data: 'Bob',   xmin: 200, xmax: null}  ← new version

  Transaction 150 (started before 200):
  → Sees OLD row (xmin=100 < 150, xmax=200 > 150) → 'Alice'

  Transaction 250 (started after 200):
  → Sees NEW row (xmin=200 < 250) → 'Bob'

  → Readers NEVER block Writers! Writers NEVER block Readers!
  → Nhưng: dead tuples tích lũy → cần VACUUM!

  VACUUM: xóa dead tuples, reclaim disk space
  AUTOVACUUM: chạy tự động (configure: autovacuum_vacuum_threshold)
```

## 6. Locks

```sql
-- Row-level locks
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;      -- exclusive lock
SELECT * FROM accounts WHERE id = 1 FOR SHARE;       -- shared lock (read OK)
SELECT * FROM accounts WHERE id = 1 FOR UPDATE SKIP LOCKED; -- skip if locked!
-- SKIP LOCKED: dùng cho job queues! Workers skip locked jobs

-- Advisory Locks (application-level)
SELECT pg_advisory_lock(12345);    -- acquire lock number 12345
-- ... do exclusive work ...
SELECT pg_advisory_unlock(12345);  -- release

-- Deadlock: T1 lock A → wait B, T2 lock B → wait A
-- PostgreSQL detects → abort 1 transaction → retry
-- Prevention: luôn lock theo THỨ TỰ NHẤT QUÁN!
```

## 7. Partitioning

```sql
-- Range Partitioning (phổ biến nhất — time-series data)
CREATE TABLE events (
  id BIGSERIAL,
  created_at TIMESTAMPTZ NOT NULL,
  data JSONB
) PARTITION BY RANGE (created_at);

CREATE TABLE events_2024_q1 PARTITION OF events
  FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
CREATE TABLE events_2024_q2 PARTITION OF events
  FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

-- Partition Pruning: query chỉ scan partitions liên quan
SELECT * FROM events WHERE created_at > '2024-03-01';
-- → Chỉ scan events_2024_q1! ← nhanh hơn nhiều so với full table

-- Khi nào partition?
-- ✅ Table > 100GB, time-series, archive old data
-- ❌ Small tables, random access patterns
```

## 8. Replication

```
┌──────────────────┐
│   PRIMARY         │  ← ALL writes go here
│   (Read/Write)    │
│                  │
│  WAL (Write-     │──── Streaming ────▶ ┌──────────────┐
│   Ahead Log)     │   Replication       │   REPLICA 1   │
│                  │                     │  (Read Only)  │
│                  │──── Streaming ────▶ ┌──────────────┐
│                  │   Replication       │   REPLICA 2   │
└──────────────────┘                     │  (Read Only)  │
                                         └──────────────┘

Streaming Replication: WAL records stream → replica applies
  Synchronous: primary WAITS until replica confirms → no data loss, higher latency
  Asynchronous: primary doesn't wait → lower latency, possible data loss on failover

Logical Replication: replicate specific tables, cross-version, selective
  → Dùng cho: zero-downtime migration, data warehouse sync

Read Replicas: route reads → replicas, writes → primary
  ⚠️ Replication lag: replica có thể TRỄ vài ms-seconds
  → "read-your-own-writes" pattern: after write → read from primary
```

## 9. Advanced Features

```sql
-- JSONB operations
SELECT data->>'name' AS name FROM users;                    -- extract text
SELECT * FROM products WHERE data @> '{"category": "electronics"}'; -- containment
CREATE INDEX idx_data_gin ON products USING GIN (data);     -- GIN index cho JSONB

-- Full-text search
SELECT * FROM articles
WHERE to_tsvector('english', title || ' ' || body)
  @@ to_tsquery('english', 'postgresql & performance');

-- Materialized Views — cached query results
CREATE MATERIALIZED VIEW monthly_revenue AS
  SELECT DATE_TRUNC('month', created_at) AS month, SUM(total) AS revenue
  FROM orders GROUP BY 1;
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;
-- CONCURRENTLY: không block reads during refresh

-- Triggers
CREATE FUNCTION update_timestamp() RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_timestamp BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();
```

## 10. Câu Hỏi Phỏng Vấn

**Q: MVCC hoạt động thế nào? Vacuum là gì?**

> A: Mỗi row có xmin/xmax (transaction IDs). UPDATE tạo row MỚI (xmin = current txid), đánh dấu row cũ (xmax = current txid). Transactions chỉ thấy rows visible theo snapshot. Readers KHÔNG block writers. Dead tuples tích lũy → VACUUM xóa dead tuples, reclaim space. AUTOVACUUM chạy tự động.

**Q: Query chậm — debug thế nào?**

> A: (1) `EXPLAIN ANALYZE` — xem actual plan. (2) Check Seq Scan → thiếu index? (3) Estimated vs actual rows lệch → `ANALYZE table`. (4) Join algorithm phù hợp? (5) `pg_stat_statements` — top slow queries. (6) Check index usage: `pg_stat_user_indexes`. (7) Connection pooling (PgBouncer).
