## 1. CAP Theorem & PACELC

```
CAP Theorem (Eric Brewer, 2000):
  Trong distributed system, khi có NETWORK PARTITION (P),
  bạn chỉ chọn được 1 trong 2:

  ┌──────────────────────────────┐
  │        CAP THEOREM           │
  │                              │
  │    C ─────────── A           │
  │    │Consistency│Availability │
  │    │           │             │
  │    └─────┬─────┘             │
  │          │                   │
  │          P                   │
  │   Partition Tolerance        │
  │   (network failures happen!) │
  └──────────────────────────────┘

  CP (Consistency + Partition Tolerance):
    → Sacrifice Availability
    → Ví dụ: ZooKeeper, etcd, HBase, MongoDB (default)
    → Khi partition: reject requests thay vì return stale data
    → Dùng khi: financial data, inventory count = PHẢI ĐÚNG

  AP (Availability + Partition Tolerance):
    → Sacrifice Consistency (eventual consistency)
    → Ví dụ: Cassandra, DynamoDB, Couchbase
    → Khi partition: vẫn serve requests, có thể return stale data
    → Dùng khi: social media feed, DNS, shopping cart

  ⚠️ P luôn xảy ra trong distributed system!
  → Thực tế: bạn chọn giữa C và A khi có partition

PACELC Theorem (mở rộng CAP):
  IF Partition → choose A or C (giống CAP)
  ELSE (no partition) → choose Latency or Consistency

  Ví dụ:
  DynamoDB: PA/EL (partition → available, else → low latency)
  ACID DB:  PC/EC (partition → consistent, else → consistent)
  Cassandra: PA/EL (tunable consistency per query!)
```

## 2. Consistency Models

```
                    STRONG
                      │
  Linearizability ────┤  Reads luôn thấy latest write
                      │  (real-time ordering)
                      │  Ví dụ: single-node DB, ZooKeeper
                      │
  Sequential ─────────┤  Tất cả clients thấy CÙNG thứ tự
  Consistency         │  (nhưng có thể không phải real-time)
                      │
  Causal ─────────────┤  Nếu A causes B → mọi client thấy A trước B
  Consistency         │  Unrelated events: thứ tự tùy ý
                      │  Ví dụ: reply luôn sau comment gốc
                      │
  Eventual ───────────┤  Cuối cùng TẤT CẢ replicas sẽ converge
  Consistency         │  Nhưng tạm thời: clients có thể thấy stale data
                      │  Ví dụ: DNS, S3, social media likes
                    WEAK

⚠️ Strong consistency = đắt (latency, throughput)
   Eventual consistency = rẻ nhưng complex application logic
   → Chọn theo USE CASE, không phải "mạnh nhất"
```

## 3. Distributed Consensus — Raft

```
Raft = consensus algorithm để các nodes ĐỒNG Ý trên cùng giá trị
(Raft dễ hiểu hơn Paxos, dùng trong etcd, CockroachDB, TiKV)

3 Roles:
  Leader:    nhận writes, replicate to followers
  Follower:  replicate data từ leader
  Candidate: đang election

Election Process:
  1. Leader gửi heartbeat mỗi N ms
  2. Follower không nhận heartbeat → timeout → trở thành Candidate
  3. Candidate tăng term, vote cho chính mình, gửi RequestVote
  4. Nhận majority votes → trở thành Leader
  5. Nếu 2 candidates → split vote → new election (random timeout)

Log Replication:
  ┌────────┐  Write "x=5"  ┌────────┐
  │ Client │──────────────▶│ Leader │
  └────────┘               └───┬────┘
                               │ AppendEntries RPC
                    ┌──────────┼──────────┐
                    ▼          ▼          ▼
              ┌──────────┐ ┌──────────┐ ┌──────────┐
              │Follower 1│ │Follower 2│ │Follower 3│
              └──────────┘ └──────────┘ └──────────┘
                    │          │          │
                    └──────ACK─┼──────ACK─┘
                               ▼
                  Majority ACK → COMMIT → respond to client

  → Write committed khi MAJORITY (>50%) followers ACK
  → Nếu leader crash → new election → new leader có committed log
```

## 4. Clocks & Ordering

```
Vấn đề: physical clocks KHÔNG synchronized giữa các machines!
  Machine A: 10:00:00.000
  Machine B: 10:00:00.050 (lệch 50ms)
  → Event trên B "trước" A theo physical clock nhưng thực tế SAU!

Lamport Timestamps (Logical Clock):
  Rules:
    1. Process increment counter trước mỗi event
    2. Gửi message: attach counter
    3. Nhận message: max(local, received) + 1

  Limitation: Lamport(A) < Lamport(B) KHÔNG có nghĩa A xảy ra trước B
  → Chỉ biết: nếu A causally-before B → Lamport(A) < Lamport(B)

Vector Clocks:
  Mỗi process giữ array [P1_count, P2_count, P3_count]
  → Detect concurrent events (Lamport không thể!)
  → Dùng trong: DynamoDB (conflict detection), Riak

  VC(A) = [2, 1, 0]
  VC(B) = [1, 2, 0]
  → A2 > B1 nhưng A1 < B2 → CONCURRENT!
  → Cần conflict resolution (LWW, merge, user resolve)

Hybrid Logical Clocks (HLC):
  Combine physical time + logical counter
  → Best of both: real-time ordering + causality
  → Dùng trong CockroachDB, TiDB
```

## 5. Failure Modes

```
CRASH-STOP:
  Node crash → NGỪNG hoàn toàn, không gửi sai data
  → Dễ handle: heartbeat → detect → failover
  → Hầu hết real-world failures

CRASH-RECOVERY:
  Node crash → restart → resume (có thể mất in-memory state)
  → WAL (Write-Ahead Log) để recover

BYZANTINE FAILURES:
  Node GỬI SAI data (bug, hacked, malicious)
  → Rất khó handle (cần >2/3 honest nodes)
  → Blockchain dùng Byzantine Fault Tolerance (BFT)
  → Hầu hết internal systems: trust nodes → KHÔNG cần BFT

NETWORK PARTITION:
  ┌──────┐          ┌──────┐
  │Node A│  ╳ ╳ ╳   │Node C│
  │Node B│  network  │Node D│
  └──────┘  down     └──────┘

  AB nghĩ CD chết, CD nghĩ AB chết
  → SPLIT BRAIN: cả 2 sides hoạt động độc lập
  → Fix: quorum (majority voting), fencing tokens
```

## 6. Distributed Transactions

```
2PC (Two-Phase Commit):
  Coordinator → Prepare → All participants vote YES/NO
  If ALL yes → Coordinator → Commit → All commit
  If ANY no  → Coordinator → Abort → All rollback

  ┌─────────────┐    Prepare?    ┌──────────┐
  │ Coordinator │───────────────▶│ Partici. │
  │             │◀──── YES ──────│    A     │
  │             │                └──────────┘
  │             │    Prepare?    ┌──────────┐
  │             │───────────────▶│ Partici. │
  │             │◀──── YES ──────│    B     │
  └──────┬──────┘                └──────────┘
         │
    All YES → COMMIT (phase 2)
    Any NO  → ABORT

  ⚠️ Blocking protocol: nếu coordinator crash sau Prepare
  → participants STUCK (đã lock resources, chờ Commit/Abort!)
  → Fix: 3PC (thêm pre-commit phase, non-blocking)
         Saga pattern (no distributed lock)

Saga Pattern (xem §4.4 chi tiết):
  Local transactions + compensation
  Choreography: events
  Orchestration: central coordinator
  → Preferred cho microservices (no distributed locking!)
```

## 7. Câu Hỏi Phỏng Vấn

**Q: CAP Theorem — giải thích với ví dụ?**

> A: Khi network partition xảy ra (P luôn có), chọn C hoặc A. **CP**: banking system — reject request thay vì show sai balance. etcd, ZooKeeper. **AP**: social media likes — show stale count OK, eventually consistent. Cassandra, DynamoDB. PACELC mở rộng: khi KHÔNG partition, chọn Latency vs Consistency.

**Q: 2PC vs Saga — khi nào dùng?**

> A: **2PC**: strong consistency, short-lived transactions, database-level (PostgreSQL across shard). Nhưng: blocking, coordinator SPOF. **Saga**: long-lived business transactions, microservices. No distributed lock. Compensation logic phức tạp hơn nhưng scalable + non-blocking. Microservices → hầu như luôn Saga.

**Q: Eventual consistency — challenges?**

> A: (1) Stale reads. (2) Conflict resolution (LWW, vector clocks, merge). (3) Read-your-own-writes (session consistency). (4) Application complexity tăng. Trade-off: strong consistency = higher latency + lower availability.
