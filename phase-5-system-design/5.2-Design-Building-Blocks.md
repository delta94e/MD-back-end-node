## 1. Load Balancer

```
L4 (Transport Layer — TCP):
  Nhìn: IP + Port
  Nhanh (no packet inspection)
  ❌ Không biết HTTP content

L7 (Application Layer — HTTP):
  Nhìn: URL path, headers, cookies, body
  ✅ Content-based routing
  ❌ Chậm hơn L4 (phải parse HTTP)

Algorithms:
  Round Robin:         A → B → C → A → B → C
  Weighted Round Robin: A(3) → B(1) → server A nhận 3x traffic
  Least Connections:   route đến server ÍT connections nhất
  IP Hash:             hash(client_ip) → same server (sticky)
  Consistent Hashing:  distribute evenly, minimal redistribution

Health Checks:
  Active: LB ping server mỗi N giây (/health endpoint)
  Passive: track errors → mark unhealthy after N failures
  → Unhealthy server → remove from pool → check periodically → add back
```

## 2. Reverse Proxy & CDN

```
Reverse Proxy (Nginx, HAProxy):
  Client → Reverse Proxy → Backend Servers

  Chức năng:
    SSL/TLS termination (decrypt HTTPS ở proxy)
    Load balancing
    Caching static content
    Compression (gzip, brotli)
    Rate limiting
    DDoS protection

CDN (Content Delivery Network):
  ┌────────┐     ┌──────────┐     ┌──────────┐
  │ User   │────▶│ CDN Edge │     │  Origin  │
  │ (VN)   │     │ (SG)     │────▶│  Server  │
  └────────┘     └──────────┘     │  (US)    │
                   Cache HIT:      └──────────┘
                   serve from edge    Cache MISS:
                   (50ms)             fetch origin
                                     (200ms)

  Pull CDN: CDN pull from origin on first request + cache
  Push CDN: bạn upload to CDN proactively

  Cache Invalidation:
    TTL-based (e.g., max-age: 3600)
    Purge API (immediate invalidation)
    Versioned URLs: /js/app.v2.js (change URL = bypass cache)
```

## 3. Cache Strategies

```
Multi-level Caching:
  Browser Cache → CDN → API Gateway → App Cache (Redis) → DB

  ┌──────┐  ┌─────┐  ┌─────────┐  ┌───────┐  ┌────┐
  │Browser│→│ CDN │→│ Gateway │→│ Redis │→│ DB │
  │Cache  │  │     │  │ Cache   │  │       │  │    │
  └──────┘  └─────┘  └─────────┘  └───────┘  └────┘
  Fastest ─────────────────────────────── Slowest

Eviction Policies:
  LRU (Least Recently Used):  remove oldest accessed ← phổ biến nhất
  LFU (Least Frequently Used): remove least accessed
  FIFO: remove oldest inserted
  TTL: remove after time expires
  Random: remove random entry

Cache Invalidation Problems:
  ⚠️ Thundering Herd: hot key expires → 1000 requests hit DB
  Fix: mutex lock, stale-while-revalidate

  ⚠️ Cache Stampede: many keys expire simultaneously
  Fix: jittered TTL (TTL = 3600 + random(0, 300))

  ⚠️ Cache Penetration: requests for NON-EXISTENT data → always miss
  Fix: cache null values (TTL=60s), Bloom Filter
```

## 4. API Gateway & Service Discovery

```
API Gateway:
  Client → Gateway → Microservices

  Responsibilities:
    Authentication/Authorization
    Rate Limiting
    Request/Response transformation
    Circuit Breaker
    Load Balancing
    API versioning
    Logging/Monitoring

  Tools: Kong, AWS API Gateway, Nginx, Traefik

Service Discovery:
  Problem: microservices scale up/down → IP changes!

  Client-side:
    Service → Register to Registry (Consul, etcd)
    Client → Query Registry → Get service address → Direct call
    ✅ No extra hop
    ❌ Client complexity

  Server-side:
    Client → Load Balancer → Query Registry → Route to service
    ✅ Simple client
    ❌ Extra hop (LB)
```

## 5. Consistent Hashing

```
Problem: hash(key) % N → khi thêm/bớt server → HẦU HẾT keys remap!
  3 servers: key1→S1, key2→S2, key3→S0
  Add S3:    key1→S0(!), key2→S3(!), key3→S3(!) → cache miss storm!

Consistent Hashing:
  1. Hash ring: 0 ────────────────▶ 2^32

     Place servers on ring:
     ┌─── S1 (hash=100) ─── S2 (hash=300) ─── S3 (hash=600) ───┐
     │                                                            │
     └────────────────────────────────────────────────────────────┘

  2. Hash key → find NEXT server clockwise
     key_a (hash=150) → S2 (next clockwise after 150)
     key_b (hash=400) → S3

  3. Add S4 (hash=200):
     ONLY keys between S1(100) and S4(200) remap!
     → Minimal redistribution (~K/N keys move)

  Virtual Nodes:
    Each server → multiple points on ring
    S1 → [S1_0, S1_1, S1_2, ...] (spread evenly)
    → More balanced distribution

  Used in: DynamoDB, Cassandra, Memcached, CDN
```

## 6. Bloom Filter

```
Bloom Filter = probabilistic data structure
  "Is X in set?" →  YES (maybe) / NO (definitely)

  ✅ No false negatives: nếu nói NO → CHẮC CHẮN không có
  ⚠️ False positives: nếu nói YES → CÓ THỂ sai (nhỏ, configurable)
  ✅ Space efficient: 1 billion items ≈ 1.7GB (vs set: ~8GB+)

  Insert "hello":
    hash1("hello") = 3 → set bit[3] = 1
    hash2("hello") = 7 → set bit[7] = 1
    hash3("hello") = 1 → set bit[1] = 1

  Check "hello": bit[3]=1, bit[7]=1, bit[1]=1 → PROBABLY YES
  Check "world": bit[5]=0 → DEFINITELY NO

Use cases:
  ✅ Cache penetration: check if key EXISTS before hitting DB
  ✅ Spam detection: is this email known spam?
  ✅ Duplicate detection: crawled this URL before?
  ✅ Recommendation: already shown this item to user?
```

## 7. Elasticsearch

```
Inverted Index (cốt lõi):
  Document 1: "Node.js is fast"
  Document 2: "Node.js and Redis are fast"

  Index:
    "node.js" → [doc1, doc2]
    "fast"    → [doc1, doc2]
    "redis"   → [doc2]
    "is"      → [doc1]
    "and"     → [doc2]
    "are"     → [doc2]

  Search "redis fast" → doc2 (matches both terms)
  → Relevance scoring: TF-IDF, BM25

Architecture:
  Cluster → Nodes → Indices → Shards → Documents
  Primary shard + Replica shards (fault tolerance)

Use cases backend:
  Full-text search, log analysis (ELK stack), analytics
```
