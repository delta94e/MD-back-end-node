# §5.4 — System Design Interview Practice

> **Mục tiêu**: Framework cho system design interview + outline cho 14 bài design phổ biến nhất. Mỗi bài phải design được trong 35-45 phút.

---

## Interview Framework (4 Steps)

```
Step 1: REQUIREMENTS (5 min)
  Functional: user stories, core features
  Non-functional: scale (QPS, storage), latency, availability, consistency
  Back-of-envelope: traffic, storage, bandwidth estimation

Step 2: HIGH-LEVEL DESIGN (10 min)
  Draw core components: Client → LB → API → Service → DB/Cache
  Data model: key entities, relationships
  API design: endpoints, request/response

Step 3: DEEP DIVE (20 min)
  Interviewer chọn component để đào sâu
  Database schema, indexing, sharding
  Caching strategy, consistency
  Scaling bottlenecks, solutions

Step 4: WRAP UP (5 min)
  Trade-offs: giải thích decisions
  Future improvements: monitoring, analytics
  Edge cases: failure modes, data loss
```

## Design Outlines

### 1. URL Shortener (TinyURL)

```
Core: long URL → short code (Base62: a-z, A-Z, 0-9)
  https://example.com/very/long/path → tinyurl.com/abc123

Write: hash(URL) → Base62(first 7 chars) → store mapping
Read:  GET /abc123 → lookup → 301 Redirect

Storage: key-value (short_code → long_url)
Scale:   read-heavy (100:1 ratio) → Cache (Redis) + DB replicas
Collision: counter-based ID → Base62 (no collision)
Analytics: click tracking, geo, referrer → async (Kafka → analytics DB)
```

### 2. Twitter/News Feed

```
Core question: Fan-out on Write vs Fan-out on Read?

Fan-out on Write (Push):
  User posts → immediately push to ALL followers' feeds
  ✅ Read is O(1) — feed pre-computed
  ❌ Celebrity problem: 100M followers = 100M writes!
  → Dùng cho: normal users (<10K followers)

Fan-out on Read (Pull):
  User opens feed → fetch posts from ALL followed users
  ✅ Write is O(1)
  ❌ Read is slow (merge from many sources)
  → Dùng cho: celebrities

Hybrid: push for normal users, pull for celebrities
```

### 3. Chat System (WhatsApp)

```
WebSocket for real-time
Message ordering: timestamp + sequence number per conversation
Offline support: store messages → deliver when online
Group chat: fan-out to group members

Key components:
  Chat Service (WebSocket) → Message Queue → Storage
  Presence Service (online/offline/last seen)
  Notification Service (push notifications)
```

### 4. YouTube/Netflix

```
Upload: video → chunk → transcode (multiple resolutions) → CDN
  Transcoding: worker queue (CPU intensive, async)

Streaming: adaptive bitrate (ABR)
  Client measures bandwidth → request appropriate quality
  HLS/DASH protocols → chunks served from nearest CDN edge

CDN: 80%+ traffic served from edge caches
  Popular videos: replicated to ALL edges
  Unpopular: origin server only
```

### 5. Ride Sharing (Uber)

```
Geospatial indexing: Geohash or QuadTree
  Geohash: encode lat/lng → string (precision = prefix length)
  "u4pruydqqvj" → smaller prefix = larger area

Matching: find nearby drivers in same geohash + adjacent cells
Real-time: driver location updates every 3-5 seconds
  → In-memory store (Redis Geo) → persistent store (background)

ETA: graph-based shortest path (Dijkstra + traffic data)
```

### 6. Payment System

```
Key: IDEMPOTENCY + double-entry bookkeeping

  POST /payments (Idempotency-Key: uuid)
  1. Check idempotency key → return cached if exists
  2. Debit buyer account, Credit seller account (same transaction!)
  3. Store result with idempotency key

  Double-entry: every transaction = debit + credit (sum = 0)
  Reconciliation: periodically verify with payment provider
  PCI compliance: NEVER store raw card numbers
```

### 7. Distributed Cache

```
Consistent hashing → distribute keys across cache nodes
Replication: each key replicated to N next nodes on ring
Eviction: LRU per node
Hot key: replicate to multiple nodes, client-side cache
Failure: node down → keys remap to next node → cold start → DB load
  Fix: warm cache from replica before failover
```

### 8. Notification System

```
Multiple channels: push, SMS, email, in-app
Priority queue: urgent (OTP) > normal (marketing)
Rate limiting per user (don't spam!)
Template engine: "Hello {name}, your order {id} is ready"
Delivery tracking: sent, delivered, read receipts

Architecture:
  Event → Notification Service → Priority Queue →
  Channel-specific workers (push/SMS/email) → External APIs
```

## Back-of-the-Envelope Numbers

```
Latency:
  L1 cache:           0.5 ns
  L2 cache:           7 ns
  RAM:                100 ns
  SSD random read:    150 μs
  HDD random read:    10 ms
  Network (same DC):  0.5 ms
  Network (US→EU):    150 ms

Throughput:
  SSD sequential:     1 GB/s
  Network (1Gbps):    125 MB/s
  DB query:           1-10 ms
  Redis GET:          0.1 ms

Scale:
  1 server:           ~10K-50K QPS (simple API, keep-alive)
  1M daily users:     ~12 QPS average, ~120 QPS peak (10x)
  1B daily users:     ~12K QPS avg, ~120K QPS peak
  1 tweet (140B) × 500M/day = 70 GB/day = ~25 TB/year
```
