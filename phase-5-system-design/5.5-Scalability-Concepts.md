## 1. Horizontal vs Vertical Scaling

```
Vertical (Scale Up):
  Add more CPU, RAM, SSD to SAME machine
  ✅ Simple, no code change
  ❌ Hardware limit, expensive, single point of failure
  → Dùng cho: database (primary), small apps

Horizontal (Scale Out):
  Add MORE machines
  ✅ "Infinite" scale, fault tolerance
  ❌ Complexity (state sync, load balancing, CAP)
  → Dùng cho: stateless services, web servers

  ⚠️ Key requirement: STATELESS services!
  → Session: store in Redis (not in-memory!)
  → Files: store in S3 (not local disk!)
  → Cache: centralized (Redis, not process-local!)
```

## 2. Database Scaling

```
1. Read Replicas:
   Primary (write) → Replica 1 (read) → Replica 2 (read)
   → Scale reads horizontally

2. Vertical Partitioning (by feature):
   Users DB, Orders DB, Products DB
   → Each DB only handles its domain

3. Horizontal Sharding (by data):
   User 1-1M → Shard 1
   User 1M-2M → Shard 2

   Shard key strategies:
   Hash-based: hash(user_id) % N → even distribution
   Range-based: user_id 1-1M → shard 1 (hot spots risk!)
   Directory-based: lookup table → flexible but extra hop

   ⚠️ Cross-shard queries = EXPENSIVE (scatter-gather)
   ⚠️ Rebalancing = COMPLEX (data migration)
   → Shard as LATE as possible!

4. Denormalization:
   Duplicate data to avoid JOINs across shards
   ✅ Read performance
   ❌ Write complexity, data inconsistency risk
```

## 3. Multi-Level Caching

```
┌────────┐ ┌─────┐ ┌─────────┐ ┌────────┐ ┌────┐
│Browser │→│ CDN │→│ Gateway │→│ Redis  │→│ DB │
│ Cache  │ │     │ │ Cache   │ │ (L1/L2)│ │    │
│ 1ms    │ │10ms │ │ 1ms     │ │ 0.1ms  │ │5ms │
└────────┘ └─────┘ └─────────┘ └────────┘ └────┘

Browser:   Cache-Control headers (max-age, ETag)
CDN:       Static assets, geo-distributed
Gateway:   API response caching (short TTL)
Redis L1:  Hot data, session, rate limit counters
Redis L2:  Computed results, aggregations
DB:        Query cache, materialized views

Cache-aside pattern (most common):
  1. Check cache → hit? return
  2. Miss → query DB → store in cache → return
  3. Write → update DB → invalidate cache (NOT update!)

  ⚠️ Why invalidate not update?
  Race condition: 2 concurrent writes → cache may have OLDER value
  Invalidate + lazy load → always consistent
```

## 4. Resilience Patterns

```
CIRCUIT BREAKER:
  ┌──────┐    ┌──────────┐    ┌──────────┐
  │CLOSED│───▶│  OPEN    │───▶│HALF-OPEN │
  │(normal)   │(reject   │    │(test 1   │
  │      │    │ all)     │    │ request) │
  └──────┘    └──────────┘    └──────────┘
       ▲           │               │
       │           │ timeout       │ success → CLOSED
       │           └───────────────┘ failure → OPEN
       └───────────────────────────┘

  Failure threshold: 5 failures in 10s → OPEN
  Open state: reject immediately (fast fail, no waiting)
  Half-open: after cooldown, try 1 request → success? close : reopen

RETRY with Exponential Backoff + Jitter:
  Attempt 1: wait 1s
  Attempt 2: wait 2s
  Attempt 3: wait 4s
  + Jitter: wait * random(0.5, 1.5)
  → Avoid thundering herd (all clients retry at same time!)

TIMEOUT:
  Set timeout cho EVERY external call
  → Prevent cascade: 1 slow service → block all callers
  → Combine with circuit breaker

BULKHEAD:
  Isolate resources per dependency
  Service A pool: max 20 connections
  Service B pool: max 10 connections
  → Service B slow → only its pool exhausted, NOT entire app!

FALLBACK:
  Primary failed → serve from cache (stale OK)
  OR serve degraded response (e.g., recommendation → show popular)
```

## 5. Observability

```
3 Pillars:

1. LOGGING (what happened):
   Structured logs (JSON, not text!)
   {"level":"error", "service":"order", "userId":42,
    "error":"payment failed", "traceId":"abc-123"}

   ELK Stack: Elasticsearch + Logstash + Kibana
   Alternatives: Loki + Grafana

2. METRICS (how is the system):
   Prometheus: collect time-series metrics
   Grafana: dashboards + alerting

   Key metrics (RED):
   Rate:     requests/second
   Errors:   error rate (5xx %)
   Duration: latency (p50, p95, p99)

   p99 latency = 99% of requests complete within X ms
   → p99 = 200ms means 1% of requests slower than 200ms

3. TRACING (where did time go):
   OpenTelemetry → Jaeger / Zipkin

   Request → Service A (50ms) → Service B (30ms) → DB (20ms)
   TraceID: abc-123 (propagated through ALL services)
   → Find bottleneck: Service A took 50ms, why?
   → Drill down into spans
```

## 6. Câu Hỏi Phỏng Vấn

**Q: Circuit Breaker — giải thích?**

> A: Pattern ngăn cascade failures. 3 states: **Closed** (normal), **Open** (reject all — fast fail), **Half-Open** (test 1 request). Closed→Open khi failure threshold exceeded. Open→Half-Open after timeout. Half-Open: success→Closed, failure→Open. Combine với retry + timeout + fallback cho resilient system.

**Q: p99 latency — tại sao quan trọng hơn average?**

> A: Average hides outliers. Average 50ms nhưng p99 = 5s → 1% users wait 5s! SLA thường define p99: "99% requests < 200ms". Monitor p50 (median), p95, p99. High p99 = investigate: GC pauses, slow DB queries, network issues.
