## 1. Monolith vs Microservices

```
MONOLITH:
  ┌──────────────────────────┐
  │      Single Codebase     │
  │  ┌─────┐ ┌─────┐ ┌────┐│
  │  │Users│ │Order│ │Pay ││
  │  └─────┘ └─────┘ └────┘│
  │    Single Database       │
  │    Single Deployment     │
  └──────────────────────────┘

MICROSERVICES:
  ┌──────┐  ┌──────┐  ┌──────┐
  │ User │  │Order │  │ Pay  │
  │ Svc  │  │ Svc  │  │ Svc  │
  │ DB   │  │ DB   │  │ DB   │
  └──┬───┘  └──┬───┘  └──┬───┘
     └─────API/Events─────┘
```

| Aspect      | Monolith         | Microservices                  |
| ----------- | ---------------- | ------------------------------ |
| Deployment  | 1 unit           | Independent per service        |
| Scaling     | Scale everything | Scale specific services        |
| Data        | Shared DB        | DB per service                 |
| Consistency | ACID (easy)      | Saga/eventual (hard)           |
| Debugging   | Stack trace      | Distributed tracing            |
| Team        | 1 team           | Conway's Law: team per service |
| Best for    | Startup, MVP     | Large teams, complex domains   |

```
MODULAR MONOLITH (best of both):
  Monolith codebase nhưng organized into MODULES
  Each module: own domain, own DB schema
  → Dễ refactor thành microservices SAU
  → Shopify, Basecamp dùng pattern này!
```

## 2. Event-Driven Architecture

```
3 Patterns (xem §3.5, §4.4 chi tiết):

1. Event Notification:      "OrderCreated" (minimal data)
2. Event-Carried State:     "OrderCreated" + full order data
3. Event Sourcing:          Store ALL events, derive state

Benefits:
  ✅ Loose coupling (services không biết nhau)
  ✅ Async processing (non-blocking)
  ✅ Audit trail (event log)
  ✅ Temporal decoupling (producer/consumer independent)

Challenges:
  ❌ Eventual consistency
  ❌ Event ordering
  ❌ Debugging complexity
  ❌ Idempotency handling
```

## 3. Serverless

```
Function-as-a-Service (FaaS):
  AWS Lambda, Google Cloud Functions, Azure Functions

  Event → Function executes → Return → Shutdown

  ✅ No server management
  ✅ Pay per invocation (cost effective for sporadic traffic)
  ✅ Auto-scaling (0 to thousands instantly)

  ❌ Cold Start: first invocation = slow (download + init)
     Node.js: ~200-500ms cold start
     Java:    ~3-10s cold start!
  ❌ Execution limit (15 min AWS Lambda)
  ❌ No persistent connections (WebSocket, DB pools)
  ❌ Vendor lock-in

Best for:
  ✅ Webhooks, image processing, scheduled tasks
  ✅ Unpredictable/bursty traffic
  ❌ Long-running processes, heavy computation
```

## 4. Hexagonal (Ports & Adapters)

```
┌───────────────────────────────────────────────┐
│                APPLICATION CORE                │
│                                               │
│  ┌─────────────────────────────────────────┐  │
│  │           DOMAIN LOGIC                   │  │
│  │  (business rules, entities, services)    │  │
│  └──────────────┬──────────────────────────┘  │
│                 │                              │
│  ┌──────────────▼──────────────────────────┐  │
│  │              PORTS                       │  │
│  │  (interfaces — what the app NEEDS)       │  │
│  │  UserRepository, PaymentGateway, etc.    │  │
│  └──────────────┬──────────────────────────┘  │
└─────────────────┼─────────────────────────────┘
                  │
     ┌────────────┼────────────┐
     ▼            ▼            ▼
  ADAPTERS    ADAPTERS     ADAPTERS
  PostgreSQL   Stripe       REST API
  Repository   Payment      Controller
  (outbound)   (outbound)   (inbound)

Core logic KHÔNG phụ thuộc vào framework, DB, external services
→ Swap PostgreSQL → MongoDB = chỉ thay adapter!
→ Test: mock ports → test domain logic isolated
```

## 5. Domain-Driven Design (DDD)

```
Key Concepts:

Bounded Context:
  Mỗi context = 1 area of business
  "Order" trong Catalog ≠ "Order" trong Shipping
  → Mỗi context có model RIÊNG

Ubiquitous Language:
  Code dùng CÙNG ngôn ngữ với Business
  class Order, function placeOrder, Payment.authorize()
  → Developer + Business hiểu nhau

Aggregate:
  Cluster of entities với 1 ROOT entity

  OrderAggregate:
    Order (root) → OrderItems, ShippingInfo
    Tất cả changes qua Order root
    → 1 transaction = 1 aggregate = consistency boundary

  ⚠️ Cross-aggregate = eventual consistency (Saga/Events)

Domain Events:
  OrderPlaced, PaymentCompleted, ItemShipped
  → Communicate giữa bounded contexts
  → Event-driven architecture natural fit
```

## 6. Câu Hỏi Phỏng Vấn

**Q: Monolith vs Microservices — khi nào dùng?**

> A: **Monolith**: startup, small team (<10 devs), new product, rapid iteration. **Microservices**: large team, complex domain, different scaling needs, technology heterogeneity. **Best practice**: start monolith → modular → extract microservices khi CẦN (Conway's Law). Premature microservices = distributed monolith nightmare.

**Q: DDD Bounded Context — tại sao quan trọng?**

> A: Avoid "God objects" — User entity khác nhau trong Auth context vs Billing context vs Shipping context. Mỗi context own model, own database, own team. Communication qua Domain Events hoặc API. Boundaries = team boundaries (Conway's Law). Thiếu boundaries → tightly-coupled monolith disguised as microservices.
