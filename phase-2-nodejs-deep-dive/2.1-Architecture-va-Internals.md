## 1. Node.js Architecture Overview

```
┌──────────────────────────────────────────────────────┐
│                  YOUR NODE.JS CODE                    │
│              (JavaScript / TypeScript)                │
│  const server = http.createServer(...)               │
└──────────────────┬───────────────────────────────────┘
                   │
      ┌────────────▼────────────┐
      │     NODE.JS BINDINGS    │     C++ layer
      │   (C++ glue code)       │     connects JS to C/C++
      └────┬──────────────┬─────┘
           │              │
   ┌───────▼──────┐  ┌───▼────────────────────────────┐
   │     V8       │  │           LIBUV                 │
   │  (JS Engine) │  │    (Async I/O library, C)       │
   │              │  │                                 │
   │  Parse JS    │  │  ┌──────────────┐               │
   │  Compile     │  │  │  Event Loop  │               │
   │  Execute     │  │  │  (single     │               │
   │  GC          │  │  │   thread)    │               │
   │              │  │  └──────┬───────┘               │
   │              │  │         │                       │
   │              │  │  ┌──────▼───────┐               │
   │              │  │  │ Thread Pool  │               │
   │              │  │  │ (4 threads)  │               │
   │              │  │  │ fs, dns,     │               │
   │              │  │  │ crypto, zlib │               │
   │              │  │  └──────────────┘               │
   └──────────────┘  │                                 │
                     │  Kernel async I/O:              │
                     │  epoll (Linux)                  │
                     │  kqueue (macOS)                 │
                     │  IOCP (Windows)                 │
                     └─────────────────────────────────┘

Tóm tắt:
  V8         = chạy JavaScript
  libuv      = async I/O, event loop, thread pool
  Bindings   = nối JS methods với C++ implementations
```

### Tại sao Single-Threaded mà handle 10,000 requests?

```
Traditional (Apache, PHP):
  1 request = 1 thread
  10,000 requests = 10,000 threads!
  Mỗi thread ~2MB stack → 20GB RAM chỉ cho threads!
  Thread context switching = overhead lớn

Node.js:
  1 thread (event loop) + kernel async I/O
  10,000 requests = 10,000 file descriptors (rất nhẹ!)

  Flow chi tiết:
  1. Request đến → callback đăng ký với libuv
  2. I/O operation (DB query, file read) → delegate cho kernel (epoll)
     HOẶC → delegate cho Thread Pool (fs, dns.lookup, crypto)
  3. Event Loop KHÔNG chờ → xử lý request khác ngay
  4. I/O hoàn thành → kernel/thread pool notify libuv
  5. Callback pushed vào event loop queue
  6. Event Loop chạy callback → trả response

  → Main thread KHÔNG BAO GIỜ bị block bởi I/O!
  → "Single-threaded" chỉ có JavaScript code
  → I/O thực sự chạy ở kernel threads hoặc libuv thread pool
```

## 2. libuv Event Loop — 6 Phases

```
Node.js Event Loop (chi tiết hơn §1.1):

   ┌───────────────────────────┐
┌─▶│       ① TIMERS            │  setTimeout, setInterval callbacks
│  │  (check if timer expired) │  ← chạy callbacks đã hết hạn
│  └───────────┬───────────────┘
│              │ nextTick + microtask drain
│  ┌───────────▼───────────────┐
│  │   ② PENDING CALLBACKS     │  System callbacks (TCP errors, etc.)
│  │  (I/O callbacks deferred  │  ← hiếm khi bạn thấy trực tiếp
│  │   from previous iteration)│
│  └───────────┬───────────────┘
│              │ nextTick + microtask drain
│  ┌───────────▼───────────────┐
│  │   ③ IDLE, PREPARE         │  Internal use by libuv
│  │                           │  ← bạn không cần quan tâm
│  └───────────┬───────────────┘
│              │ nextTick + microtask drain
│  ┌───────────▼───────────────┐
│  │        ④ POLL              │  ← QUAN TRỌNG NHẤT!
│  │                           │
│  │  1. Tính timeout (bao lâu │
│  │     nên block ở đây)      │
│  │  2. Process I/O events    │
│  │     trong poll queue      │
│  │                           │
│  │  Nếu poll queue rỗng:     │
│  │  - có setImmediate? → ⑤   │
│  │  - có timer hết hạn? → ①  │
│  │  - else: block chờ I/O    │
│  └───────────┬───────────────┘
│              │ nextTick + microtask drain
│  ┌───────────▼───────────────┐
│  │       ⑤ CHECK              │  setImmediate callbacks
│  │                           │  ← luôn chạy SAU poll phase
│  └───────────┬───────────────┘
│              │ nextTick + microtask drain
│  ┌───────────▼───────────────┐
│  │   ⑥ CLOSE CALLBACKS       │  socket.on('close'), etc.
│  │                           │
│  └───────────┬───────────────┘
└──────────────┘

GIỮA MỌI PHASE:
  1. Drain process.nextTick queue
  2. Drain microtask queue (Promise.then, queueMicrotask)
```

### setTimeout(0) vs setImmediate vs process.nextTick

```javascript
// CÂU HỎI KINH ĐIỂN: thứ tự output?

// Script chạy lần đầu (main module):
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// Thứ tự KHÔNG XÁC ĐỊNH! (phụ thuộc system performance)
// setTimeout(0) thực tế = setTimeout(1ms min)
// Nếu event loop đến timers phase < 1ms → timeout chưa ready → immediate trước
// Nếu > 1ms → timeout trước

// NHƯNG trong I/O callback:
const fs = require("fs");
fs.readFile("file.txt", () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
});
// setImmediate LUÔN trước! (I/O callback chạy ở poll → check phase tiếp theo)

// process.nextTick: LUÔN chạy trước tất cả
process.nextTick(() => console.log("nextTick"));
Promise.resolve().then(() => console.log("promise"));
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// Output: nextTick → promise → timeout/immediate
```

### nextTick starvation

```javascript
// ⚠️ NGUY HIỂM: nextTick CÓ THỂ gây starvation!
function recursiveNextTick() {
  process.nextTick(recursiveNextTick); // VĨNH VIỄN chạy nextTick
}
recursiveNextTick();
// Event Loop KHÔNG BAO GIỜ đến timers phase!
// setTimeout, setImmediate, I/O callbacks KHÔNG BAO GIỜ chạy!
// → "I/O starvation"

// Khi nào dùng nextTick?
// ✅ Đảm bảo callback chạy TRƯỚC mọi I/O (event ordering)
// ✅ Emit events sau constructor hoàn thành
// ❌ KHÔNG dùng cho recursion/loop → dùng setImmediate thay!
```

## 3. Thread Pool

```
┌──────────────────────────────────────────┐
│              LIBUV THREAD POOL            │
│         (Default: 4 threads)              │
│         UV_THREADPOOL_SIZE=4              │
│                                          │
│  Thread 1: [fs.readFile callback #1]     │
│  Thread 2: [dns.lookup callback]         │
│  Thread 3: [crypto.pbkdf2 callback]      │
│  Thread 4: [zlib.gzip callback]          │
│                                          │
│  Queue: [fs.readFile #2, #3, ..., #96]   │
│          ↑ 100 concurrent reads,         │
│            nhưng chỉ 4 threads!          │
│            96 tasks phải CHỜ!            │
└──────────────────────────────────────────┘

Những gì DÙNG thread pool:
  ✅ fs.* (tất cả file system operations)
  ✅ dns.lookup() (⚠️ KHÔNG phải dns.resolve())
  ✅ crypto.pbkdf2, crypto.scrypt, crypto.randomFill
  ✅ zlib.* (compression)

Những gì KHÔNG dùng thread pool (dùng kernel async):
  ❌ Network I/O (TCP, UDP, HTTP) → epoll/kqueue/IOCP
  ❌ dns.resolve() → c-ares library (async DNS)
  ❌ Pipes, signals

Tăng thread pool:
  UV_THREADPOOL_SIZE=16 node app.js
  Max: 1024 (nhưng tăng quá = context switching overhead)
```

```javascript
// 100 concurrent fs.readFile → chỉ 4 chạy cùng lúc!
const fs = require("fs");
for (let i = 0; i < 100; i++) {
  fs.readFile("data.txt", (err, data) => {
    console.log(`File ${i} read`);
  });
}
// 4 files đọc song song, 96 files queue chờ
// → Bottleneck! Fix: tăng UV_THREADPOOL_SIZE
//                  hoặc dùng streaming (fs.createReadStream)
```

## 4. Worker Threads

```javascript
// Worker Threads = TRUE multi-threading cho CPU-intensive tasks
// (khác Cluster: Cluster = multi-process)

// main.js
const {
  Worker,
  isMainThread,
  workerData,
  parentPort,
} = require("worker_threads");

if (isMainThread) {
  // MAIN THREAD — quản lý workers
  const worker = new Worker(__filename, { workerData: 1000000000 });
  worker.on("message", (result) => console.log("Result:", result));
  worker.on("error", (err) => console.error(err));

  console.log("Main thread continues..."); // KHÔNG bị block!
} else {
  // WORKER THREAD — CPU-intensive work
  let sum = 0;
  for (let i = 0; i < workerData; i++) sum += i;
  parentPort.postMessage(sum);
}
```

### SharedArrayBuffer & Atomics

```javascript
// SharedArrayBuffer: memory CHIA SẺ giữa threads (không copy!)
const { Worker } = require("worker_threads");

const shared = new SharedArrayBuffer(4); // 4 bytes
const arr = new Int32Array(shared);
arr[0] = 0;

const worker = new Worker(
  `
  const { workerData } = require("worker_threads");
  const arr = new Int32Array(workerData);
  Atomics.add(arr, 0, 42); // thread-safe increment
`,
  { eval: true, workerData: shared },
);

worker.on("exit", () => {
  console.log(arr[0]); // 42 — shared memory!
});
```

### Worker Threads vs Cluster

| Feature       | Worker Threads                | Cluster                         |
| ------------- | ----------------------------- | ------------------------------- |
| Model         | Multi-thread (1 process)      | Multi-process                   |
| Memory        | Shared (SharedArrayBuffer)    | Separate (copy)                 |
| Communication | MessageChannel, shared memory | IPC (serialized JSON)           |
| Use case      | CPU-intensive tasks           | HTTP server scaling             |
| Overhead      | Thấp                          | Cao hơn (mỗi process = full V8) |
| Isolation     | Thread crash = process crash  | Process crash = chỉ 1 worker    |

## 5. Cluster Module

```javascript
const cluster = require("cluster");
const http = require("http");
const numCPUs = require("os").cpus().length;

if (cluster.isPrimary) {
  console.log(`Primary ${process.pid} – forking ${numCPUs} workers`);
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork(); // Tạo worker process (copy of this script)
  }
  cluster.on("exit", (worker, code) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork(); // Auto-restart!
  });
} else {
  http
    .createServer((req, res) => {
      res.writeHead(200);
      res.end(`Worker ${process.pid}\n`);
    })
    .listen(8000);
  // Tất cả workers SHARE cùng port 8000!
  // OS load balance requests giữa workers (round-robin default trên Linux)
}
```

```
┌─────────────────────────────────────────────┐
│                PRIMARY PROCESS               │
│  - Quản lý workers                          │
│  - Fork, restart, monitor                   │
│  - IPC communication                        │
│                                             │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐       │
│  │  W1 │  │  W2 │  │  W3 │  │  W4 │       │
│  │:8000│  │:8000│  │:8000│  │:8000│       │
│  └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘       │
```

## 6. Child Process

```javascript
const { spawn, exec, execFile, fork } = require("child_process");

// spawn — stream-based, cho commands dài/output lớn
const ls = spawn("ls", ["-la", "/tmp"]);
ls.stdout.on("data", (data) => console.log(`stdout: ${data}`));
ls.stderr.on("data", (data) => console.error(`stderr: ${data}`));
ls.on("close", (code) => console.log(`exited with code ${code}`));
// ✅ Không buffer toàn bộ output vào memory

// exec — buffer-based, cho commands ngắn
exec("ls -la /tmp", (err, stdout, stderr) => {
  console.log(stdout); // toàn bộ output 1 lần
});
// ⚠️ Buffer giới hạn (default 1MB → maxBuffer option)

// execFile — như exec nhưng KHÔNG qua shell
execFile("node", ["--version"], (err, stdout) => {
  console.log(stdout); // v20.x.x
});
// ✅ An toàn hơn exec (không shell injection)

// fork — đặc biệt cho Node.js scripts, có IPC channel!
const child = fork("worker.js");
child.send({ type: "START", data: [1, 2, 3] }); // IPC
child.on("message", (msg) => console.log("From child:", msg));
```

| Method     | Shell | Buffer | Stream | IPC | Best for                     |
| ---------- | :---: | :----: | :----: | :-: | ---------------------------- |
| `spawn`    |  ❌   |   ❌   |   ✅   | ❌  | Long-running, large output   |
| `exec`     |  ✅   |   ✅   |   ❌   | ❌  | Short commands, small output |
| `execFile` |  ❌   |   ✅   |   ❌   | ❌  | No shell, safer              |
| `fork`     |  ❌   |   ❌   |   ✅   | ✅  | Node.js child processes      |

## 7. Câu Hỏi Phỏng Vấn

**Q: Node.js single-threaded — handle 10,000 concurrent requests thế nào?**

> A: "Single-threaded" chỉ có JavaScript. I/O KHÔNG chạy trên main thread: (1) Network I/O dùng kernel async (epoll/kqueue) — zero threads. (2) fs/crypto/zlib dùng libuv thread pool (default 4). (3) Event Loop: request đến → đăng ký callback → delegate I/O → xử lý request khác → I/O done → callback queued → event loop chạy callback. Mỗi connection chỉ là 1 file descriptor (~KB), không phải 1 thread (~MB).

**Q: process.nextTick vs setImmediate — khác gì? Khi nào dùng?**

> A: `nextTick` chạy **trước khi event loop chuyển phase** (giữa mỗi phase). `setImmediate` chạy ở **check phase** (sau poll). nextTick ưu tiên cao hơn nhưng CÓ THỂ gây I/O starvation nếu recursive. Best practice: dùng setImmediate cho deferred execution, nextTick chỉ khi cần ensure-before-I/O.

**Q: Thread pool 4 threads + 100 concurrent fs.readFile — chuyện gì xảy ra?**

> A: 4 files đọc song song, 96 files queue chờ trong libuv task queue. Performance bị bottleneck. Fix: (1) `UV_THREADPOOL_SIZE=16` (tăng lên 16-64 threads). (2) Dùng streaming `fs.createReadStream`. (3) Batch processing (đọc theo nhóm). ⚠️ dns.lookup() CŨNG dùng thread pool → nếu DNS chậm + nhiều fs operations → thread pool exhaustion!
