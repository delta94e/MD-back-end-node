## 1. RabbitMQ

```
AMQP (Advanced Message Queuing Protocol):

  Producer → Exchange ─── Binding ───▶ Queue → Consumer

  Exchange Types:
  ┌─────────────────────────────────────────────────┐
  │ DIRECT:  routing_key exact match                 │
  │   Exchange ──order.created──▶ OrderQueue         │
  │                                                  │
  │ TOPIC:   routing_key pattern match               │
  │   Exchange ──order.*──▶ OrderQueue               │
  │   Exchange ──*.created──▶ AuditQueue             │
  │   * = 1 word, # = 0+ words                      │
  │                                                  │
  │ FANOUT:  broadcast to ALL bound queues           │
  │   Exchange ──▶ Queue1, Queue2, Queue3            │
  │   (ignore routing key)                           │
  │                                                  │
  │ HEADERS: match by message headers                │
  │   (rarely used)                                  │
  └─────────────────────────────────────────────────┘
```

```typescript
// RabbitMQ với amqplib
const conn = await amqp.connect("amqp://localhost");
const channel = await conn.createChannel();

// Producer
await channel.assertExchange("orders", "topic", { durable: true });
channel.publish("orders", "order.created", Buffer.from(JSON.stringify(order)), {
  persistent: true, // survive broker restart
});

// Consumer
await channel.assertQueue("order-processor", { durable: true });
await channel.bindQueue("order-processor", "orders", "order.*");
channel.prefetch(10); // process max 10 messages concurrently
channel.consume("order-processor", async (msg) => {
  try {
    const order = JSON.parse(msg.content.toString());
    await processOrder(order);
    channel.ack(msg); // ✅ processed successfully
  } catch (err) {
    channel.nack(msg, false, false); // reject, don't requeue → DLX
  }
});

// Dead Letter Exchange (DLX): failed messages go here
await channel.assertQueue("order-processor", {
  durable: true,
  deadLetterExchange: "orders-dlx",
  deadLetterRoutingKey: "order.failed",
});
```

## 2. Apache Kafka

```
Kafka Architecture:

  ┌──────────────────────────────────────────┐
  │              KAFKA CLUSTER                │
  │                                          │
  │  Topic: "orders"                         │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐│
  │  │Partition 0│ │Partition 1│ │Partition 2││
  │  │[0][1][2] │ │[0][1][2] │ │[0][1]    ││
  │  │[3][4]    │ │[3]       │ │          ││
  │  └──────────┘ └──────────┘ └──────────┘│
  │                                          │
  │  Consumer Group "order-service":         │
  │  Consumer A → Partition 0               │
  │  Consumer B → Partition 1               │
  │  Consumer C → Partition 2               │
  └──────────────────────────────────────────┘

Key concepts:
  Topic       = category of messages (like a table)
  Partition   = ordered, immutable sequence of messages
  Offset      = position of message within partition
  Consumer Group = group of consumers, each partition → 1 consumer

  Message ordering: CHỈX guaranteed TRONG 1 partition!
  → Dùng key: messages cùng key → cùng partition → ordered
  → user_id=42 luôn vào 1 partition → orders của user 42 ordered
```

```typescript
// Kafka Producer (kafkajs)
const producer = kafka.producer();
await producer.send({
  topic: "orders",
  messages: [
    {
      key: order.userId.toString(), // same user → same partition!
      value: JSON.stringify(order),
      headers: { source: "order-service" },
    },
  ],
});

// Kafka Consumer
const consumer = kafka.consumer({ groupId: "notification-service" });
await consumer.subscribe({ topic: "orders", fromBeginning: false });
await consumer.run({
  eachMessage: async ({ topic, partition, message }) => {
    const order = JSON.parse(message.value.toString());
    await sendNotification(order);
    // Offset auto-committed (hoặc manual commit)
  },
});
```

```
Kafka Guarantees:
  Retention: messages kept N days (default 7) hoặc forever
  Log Compaction: giữ latest value per key (like snapshot)
  ISR (In-Sync Replicas): replicas up-to-date → failover safe
  acks=0:  fire-and-forget (fastest, may lose)
  acks=1:  leader acknowledged (balanced)
  acks=all: all ISR acknowledged (safest, slowest)
```

## 3. Delivery Guarantees

```
AT-MOST-ONCE:
  Send → don't wait/retry
  ✅ Fast, simple
  ❌ May LOSE messages
  Use: metrics, logging (acceptable loss)

AT-LEAST-ONCE:
  Send → wait ACK → retry if no ACK
  ✅ No message loss
  ❌ May DUPLICATE (retry = process twice!)
  Use: most systems + idempotent consumers
  Fix duplication: idempotency key, dedup table

EXACTLY-ONCE (HARDEST!):
  Each message processed EXACTLY once
  ⚠️ True exactly-once = distributed transactions = complex!
  Kafka: idempotent producer + transactional consumer
  Practical: at-least-once + idempotent processing = "effectively once"
```

## 4. Event-Driven Patterns

```
1. EVENT NOTIFICATION:
   Service A: "Order Created" (chỉ event, ít data)
   Service B: nhận event → query Service A cho details
   ✅ Loose coupling
   ❌ Extra call cho details

2. EVENT-CARRIED STATE TRANSFER:
   Service A: "Order Created" (bao gồm FULL order data)
   Service B: nhận event → đã có đủ data, không cần query
   ✅ No extra calls, B có local copy
   ❌ Event lớn hơn, data duplication

3. EVENT SOURCING:
   Store events, derive state (xem §3.5)

4. CQRS:
   Separate read/write models (xem §3.5)
```

## 5. Saga Pattern

```
Distributed transaction qua multiple services:
  Order Service → Payment Service → Inventory Service → Shipping Service
  Mỗi step có thể FAIL → cần COMPENSATE (undo)!

CHOREOGRAPHY (event-based):
  Order → emit "OrderCreated"
  Payment → hear event → process → emit "PaymentCompleted"
  Inventory → hear event → reserve → emit "InventoryReserved"
  Shipping → hear event → ship

  If Payment fails → emit "PaymentFailed"
  Order → hear event → cancel order (compensate)

  ✅ No central coordinator, loose coupling
  ❌ Hard to track flow, debug, handle complex compensation

ORCHESTRATION (central coordinator):
  Saga Orchestrator:
    1. Tell Payment: "charge $100"     → success
    2. Tell Inventory: "reserve items" → success
    3. Tell Shipping: "create label"   → FAIL!
    4. Compensate: Inventory.release(), Payment.refund()

  ✅ Clear flow, easy to debug, complex logic OK
  ❌ Single point of coordination, tighter coupling
```

## 6. RabbitMQ vs Kafka

| Feature     | RabbitMQ             | Kafka                          |
| ----------- | -------------------- | ------------------------------ |
| Model       | Message Queue (push) | Event Log (pull)               |
| Ordering    | Per-queue            | **Per-partition**              |
| Persistence | Until consumed       | **Retained (days/forever)**    |
| Throughput  | ~50K msg/s           | **1M+ msg/s**                  |
| Replay      | ❌                   | ✅ (re-read from offset)       |
| Use case    | Task queues, RPC     | **Event streaming, analytics** |
| Routing     | Complex (exchanges)  | Simple (topics/partitions)     |

```
Dùng RabbitMQ khi:
  ✅ Task distribution (workers)
  ✅ Complex routing logic
  ✅ Message priority
  ✅ Request-reply pattern

Dùng Kafka khi:
  ✅ Event sourcing, event streaming
  ✅ High throughput (millions msg/s)
  ✅ Message replay needed
  ✅ Real-time analytics, CDC
```

## 7. Câu Hỏi Phỏng Vấn

**Q: Kafka đảm bảo ordering thế nào?**

> A: Ordering chỉ trong **1 partition**. Messages cùng **key** → cùng partition (consistent hashing). Ví dụ: key = user_id → tất cả events của user 42 ordered. Cross-partition ordering = KHÔNG guarantee → cần design key cẩn thận.

**Q: At-least-once vs exactly-once?**

> A: At-least-once: retry nếu no ACK → may duplicate. Fix: idempotent consumer (check dedup table before processing). Exactly-once: Kafka supports with idempotent producer (PID + sequence number) + transactional consumer (read-process-commit atomically). Thực tế: at-least-once + idempotency = practical "exactly-once".

**Q: Saga Choreography vs Orchestration?**

> A: **Choreography**: services communicate qua events, no coordinator. Simple flows, loose coupling. **Orchestration**: central Saga orchestrator tells each service what to do. Complex flows, easy compensation logic. Trade-off: coupling vs complexity. Practical: dùng orchestration cho payment flows (complex compensation), choreography cho simple notification flows.
