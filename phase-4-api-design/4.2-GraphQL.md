## 1. Core Concepts

```graphql
# Schema Definition
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]! # relationship
}

type Post {
  id: ID!
  title: String!
  author: User!
  comments: [Comment!]!
}

type Query {
  user(id: ID!): User
  users(limit: Int, offset: Int): [User!]!
  post(id: ID!): Post
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
  deleteUser(id: ID!): Boolean!
}

type Subscription {
  postCreated: Post! # real-time via WebSocket
}

input CreateUserInput {
  name: String!
  email: String!
}
```

```graphql
# Client query — chỉ lấy fields CẦN!
query {
  user(id: "42") {
    name
    posts {
      title
      comments {
        body
      }
    }
  }
}
# Response chỉ chứa name, posts[].title, comments[].body
# → Không over-fetching (REST trả TẤT CẢ fields!)
# → Không under-fetching (1 request lấy user + posts + comments!)
```

## 2. Resolvers & N+1 Problem

```typescript
// N+1 Problem:
// Query: users { posts { title } }
// 1 query: SELECT * FROM users (N users)
// N queries: SELECT * FROM posts WHERE user_id = ? (mỗi user!)
// → 10 users = 11 queries!

// FIX: DataLoader (batching + caching)
const postLoader = new DataLoader(async (userIds) => {
  // 1 query thay vì N queries!
  const posts = await db.query(
    `SELECT * FROM posts WHERE user_id IN (${userIds.join(",")})`,
  );
  // Map results back to each userId
  return userIds.map((id) => posts.filter((p) => p.user_id === id));
});

// Resolver
const resolvers = {
  User: {
    posts: (user) => postLoader.load(user.id), // batched!
  },
};
// 10 users → DataLoader batches vào 1 query:
// SELECT * FROM posts WHERE user_id IN (1,2,3,...,10)
// → 2 queries total thay vì 11!
```

## 3. Performance & Security

```
⚠️ GraphQL cho client TỰ DO query → security risk!

1. Query Depth Limiting:
   { user { posts { comments { author { posts { ... } } } } } }
   → Limit depth = 5 (reject deeper queries)

2. Query Complexity Analysis:
   Assign cost to each field:
   user: 1, posts: 10, comments: 20
   Max complexity: 1000 per query

3. Persisted Queries:
   Client gửi query HASH thay vì full query
   → Server chỉ accept registered queries
   → Ngăn arbitrary queries, giảm bandwidth

4. Field-level Caching:
   @cacheControl(maxAge: 300) on type definitions
   → CDN/Apollo Server cache by field
```

## 4. Federation

```
Apollo Federation: chia schema cho multiple services

  ┌──────────────┐
  │   Gateway    │  ← 1 endpoint cho client
  │   (Apollo)   │
  └──┬────┬────┬─┘
     │    │    │
  ┌──▼─┐ ┌▼──┐ ┌▼──────┐
  │User│ │Post│ │Comment │  ← mỗi service own schema riêng
  │Svc │ │Svc │ │Service │
  └────┘ └────┘ └────────┘

  // User service
  type User @key(fields: "id") {
    id: ID!
    name: String!
  }

  // Post service (extend User)
  extend type User @key(fields: "id") {
    id: ID! @external
    posts: [Post!]!         ← Post service resolve posts cho User
  }
```

## 5. REST vs GraphQL

| Feature        | REST                    | GraphQL                       |
| -------------- | ----------------------- | ----------------------------- |
| Data fetching  | Fixed endpoints         | Flexible queries              |
| Over-fetching  | ✅ Common               | ❌ Chỉ lấy fields cần         |
| Under-fetching | ✅ Multiple requests    | ❌ 1 request                  |
| Caching        | HTTP caching (simple)   | Complex (Apollo cache)        |
| File upload    | Native                  | Workaround needed             |
| Learning curve | Thấp                    | Trung bình                    |
| Error handling | HTTP status codes       | 200 + errors array            |
| Best for       | Simple CRUD, public API | Complex relationships, mobile |

```
Khi nào GraphQL?
  ✅ Mobile apps (bandwidth sensitive)
  ✅ Complex relationships (user → posts → comments)
  ✅ Multiple client types (web, mobile, TV)
  ✅ Rapid frontend iteration

Khi nào REST?
  ✅ Simple CRUD operations
  ✅ File upload/download
  ✅ Public APIs (caching friendly)
  ✅ Microservices internal communication
```

## 6. Câu Hỏi Phỏng Vấn

**Q: N+1 problem trong GraphQL — giải thích và fix?**

> A: Nested resolvers gọi DB mỗi parent row → N parents = N+1 queries. Fix: **DataLoader** — batch multiple `.load(id)` calls vào 1 `WHERE id IN (...)` query. DataLoader cũng cache per-request. Key: DataLoader phải tạo mới mỗi request (tránh stale data).

**Q: GraphQL security concerns?**

> A: Client query tùy ý → (1) Deep queries → stack overflow. Fix: depth limiting. (2) Expensive queries → DoS. Fix: complexity analysis + cost limit. (3) Introspection → expose schema. Fix: disable in production. (4) Injection → vẫn cần sanitize inputs.
