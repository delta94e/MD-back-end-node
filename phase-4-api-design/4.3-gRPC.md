## 1. Protocol Buffers

```protobuf
// user.proto — schema definition
syntax = "proto3";

package user;

message User {
  int32 id = 1;          // field number (NOT value!)
  string name = 2;
  string email = 3;
  repeated Order orders = 4;  // array
  optional string bio = 5;   // optional field
}

message Order {
  int32 id = 1;
  double total = 2;
  OrderStatus status = 3;
}

enum OrderStatus {
  PENDING = 0;
  COMPLETED = 1;
  CANCELLED = 2;
}

// Service definition
service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc ListUsers(ListUsersRequest) returns (stream User);      // server streaming
  rpc CreateUsers(stream User) returns (CreateUsersResponse);  // client streaming
  rpc Chat(stream Message) returns (stream Message);           // bidirectional
}

message GetUserRequest { int32 id = 1; }
message ListUsersRequest { int32 limit = 1; }
message CreateUsersResponse { int32 count = 1; }
message Message { string text = 1; }
```

```
Protobuf vs JSON:
  - Binary format → 3-10x nhỏ hơn JSON
  - Schema-based → type safety, code generation
  - Backward/Forward compatible (field numbers!)
    - Add field = compatible (old code ignores new fields)
    - Remove field = compatible (new code ignores old fields)
    - Change field NUMBER = BREAKING! (number = wire format ID)
```

## 2. 4 Communication Patterns

```
┌─────────────────────────────────────────────────┐
│ 1. UNARY (Request-Response)                      │
│    Client ──request──▶ Server ──response──▶      │
│    Giống REST call. Phổ biến nhất.               │
├─────────────────────────────────────────────────┤
│ 2. SERVER STREAMING                              │
│    Client ──request──▶ Server ══stream══▶▶▶      │
│    1 request → nhiều responses                   │
│    Use: real-time updates, large result sets     │
├─────────────────────────────────────────────────┤
│ 3. CLIENT STREAMING                              │
│    Client ══stream══▶▶▶ Server ──response──▶     │
│    Nhiều requests → 1 response                   │
│    Use: file upload, batch data                  │
├─────────────────────────────────────────────────┤
│ 4. BIDIRECTIONAL STREAMING                       │
│    Client ══stream══▶▶▶                          │
│    Server ══stream══▶▶▶                          │
│    Cả hai gửi stream đồng thời                  │
│    Use: chat, real-time collaboration            │
└─────────────────────────────────────────────────┘
```

```typescript
// Node.js gRPC server (Unary)
const server = new grpc.Server();
server.addService(UserServiceService, {
  getUser: (call, callback) => {
    const user = db.findById(call.request.id);
    if (!user) {
      callback({
        code: grpc.status.NOT_FOUND,
        message: `User ${call.request.id} not found`,
      });
    } else {
      callback(null, user);
    }
  },

  // Server Streaming
  listUsers: async (call) => {
    const cursor = db.users.find({}).cursor();
    for await (const user of cursor) {
      call.write(user);
    }
    call.end();
  },
});
```

## 3. Error Handling & Deadlines

```
gRPC Status Codes (giống nhưng KHÁC HTTP!):
  OK (0)              — success
  CANCELLED (1)       — client cancelled
  INVALID_ARGUMENT (3)— bad request (= HTTP 400)
  NOT_FOUND (5)       — resource not found (= HTTP 404)
  ALREADY_EXISTS (6)  — conflict (= HTTP 409)
  PERMISSION_DENIED (7)— forbidden (= HTTP 403)
  UNAUTHENTICATED (16)— no auth (= HTTP 401)
  DEADLINE_EXCEEDED (4)— timeout (= HTTP 504)
  INTERNAL (13)       — server error (= HTTP 500)
  UNAVAILABLE (14)    — service unavailable (= HTTP 503)

Deadline Propagation:
  Service A (deadline: 5s) → Service B → Service C
  B receives remaining deadline (e.g., 4.5s) → passes to C
  C receives remaining (e.g., 4s)
  → Nếu deadline expired ở bất kỳ đâu → DEADLINE_EXCEEDED
  → Tránh cascade: request chậm → tất cả services chờ!
```

## 4. Load Balancing

```
Client-side:
  Client biết TẤT CẢ server addresses
  Client tự route (round-robin, least-connections)
  ✅ No single point of failure
  ❌ Client phức tạp hơn

Proxy-based (Envoy, Nginx):
  Client → Load Balancer → Servers
  ✅ Client đơn giản
  ❌ Extra hop, single point of failure

⚠️ gRPC dùng HTTP/2 = long-lived connections
  → L4 load balancer (TCP) → KHÔNG balance per-request!
  → Cần L7 load balancer (Envoy) hoặc client-side balancing
```

## 5. gRPC vs REST vs GraphQL

| Feature     | REST           | GraphQL          | gRPC                  |
| ----------- | -------------- | ---------------- | --------------------- |
| Protocol    | HTTP/1.1       | HTTP/1.1         | **HTTP/2**            |
| Format      | JSON (text)    | JSON (text)      | **Protobuf (binary)** |
| Performance | Tốt            | Tốt              | **Tốt nhất**          |
| Streaming   | ❌ (WebSocket) | Subscription     | **4 types**           |
| Browser     | ✅ Native      | ✅ Native        | ❌ (cần grpc-web)     |
| Best for    | Public APIs    | Flexible queries | **Microservices**     |

## 6. Câu Hỏi Phỏng Vấn

**Q: Tại sao gRPC nhanh hơn REST?**

> A: (1) **Protobuf**: binary, 3-10x nhỏ hơn JSON. (2) **HTTP/2**: multiplexing (nhiều requests trên 1 connection), header compression. (3) **Code generation**: no parsing overhead, type-safe. (4) **Streaming**: persistent connection, no TCP handshake per request.
