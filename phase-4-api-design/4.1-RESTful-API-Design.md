## 1. REST Principles

```
REST = Representational State Transfer (Roy Fielding, 2000)

6 Constraints:
  1. Client-Server     — tách biệt UI và data
  2. Stateless         — mỗi request chứa ĐỦ info, server KHÔNG giữ state
  3. Cacheable         — responses phải nói rõ cacheable hay không
  4. Uniform Interface — 4 sub-constraints:
     a. Resource identification (URLs)
     b. Resource manipulation through representations (JSON/XML)
     c. Self-descriptive messages (Content-Type, etc.)
     d. HATEOAS (links trong response cho next actions)
  5. Layered System    — client không biết nói chuyện trực tiếp với server hay proxy
  6. Code-on-Demand    — (optional) server gửi executable code

HATEOAS example (thực tế ít dùng):
{
  "id": 1,
  "name": "Alice",
  "_links": {
    "self": { "href": "/api/users/1" },
    "orders": { "href": "/api/users/1/orders" },
    "update": { "href": "/api/users/1", "method": "PUT" }
  }
}
```

## 2. URL Design

```
Resources = DANH TỪ, không phải động từ!

✅ ĐÚNG:
  GET    /api/users              ← list users
  GET    /api/users/42           ← get user 42
  POST   /api/users              ← create user
  PUT    /api/users/42           ← replace user 42
  PATCH  /api/users/42           ← partial update user 42
  DELETE /api/users/42           ← delete user 42

❌ SAI:
  GET    /api/getUsers           ← verb trong URL!
  POST   /api/createUser         ← verb!
  GET    /api/deleteUser/42      ← GET không nên thay đổi data!

Nested Resources (relationship):
  GET    /api/users/42/orders           ← orders của user 42
  POST   /api/users/42/orders           ← tạo order cho user 42
  GET    /api/users/42/orders/7         ← order #7 của user 42

  ⚠️ Nest tối đa 2 levels! Quá sâu → dùng query params:
  GET    /api/orders?user_id=42&status=pending

Query Parameters:
  GET /api/products?category=electronics&sort=-price&page=2&limit=20
  - Filter: category, status, date_from, date_to
  - Sort: sort=name (asc), sort=-price (desc)
  - Pagination: page + limit HOẶC cursor
  - Fields: fields=id,name,price (sparse fieldsets)
```

## 3. HTTP Status Codes

```
2xx SUCCESS:
  200 OK              — GET/PUT/PATCH thành công
  201 Created         — POST tạo resource mới (Location header!)
  204 No Content      — DELETE thành công, no body

3xx REDIRECT:
  301 Moved Permanently  — URL changed forever (SEO redirect)
  302 Found              — temporary redirect
  304 Not Modified       — cache still valid (ETag/If-None-Match)

4xx CLIENT ERROR:
  400 Bad Request     — invalid JSON, missing required fields
  401 Unauthorized    — NOT authenticated (token missing/expired)
  403 Forbidden       — authenticated nhưng KHÔNG có quyền
  404 Not Found       — resource không tồn tại
  409 Conflict        — duplicate (unique constraint), state conflict
  422 Unprocessable   — data hợp lệ syntax nhưng sai logic (validation)
  429 Too Many Reqs   — rate limit exceeded, Retry-After header

5xx SERVER ERROR:
  500 Internal Error  — bug, unhandled exception
  502 Bad Gateway     — upstream server down
  503 Service Unavail — server overload, maintenance
  504 Gateway Timeout — upstream server timeout
```

## 4. Pagination

### Offset-based vs Cursor-based

```
OFFSET-BASED (truyền thống):
  GET /api/posts?page=3&limit=20
  → SQL: SELECT * FROM posts ORDER BY id LIMIT 20 OFFSET 40

  ✅ Dễ implement, jump to page N
  ❌ page=10000 → CHẬM (DB vẫn scan 200,000 rows rồi skip!)
  ❌ Nếu data thêm/xóa giữa page requests → miss/duplicate items

CURSOR-BASED (preferred cho large datasets):
  GET /api/posts?after=eyJpZCI6MTAwfQ&limit=20
  → SQL: SELECT * FROM posts WHERE id > 100 ORDER BY id LIMIT 20

  Response:
  {
    "data": [...],
    "pagination": {
      "next_cursor": "eyJpZCI6MTIwfQ",  // base64({id: 120})
      "has_more": true
    }
  }

  ✅ Consistent results (không miss/dup khi data thay đổi)
  ✅ O(1) performance với index (WHERE id > cursor)
  ❌ Không jump to page N
  ❌ Phức tạp hơn implement

KEYSET PAGINATION (cho multi-column sort):
  WHERE (created_at, id) > ('2024-01-15', 500)
  ORDER BY created_at DESC, id DESC LIMIT 20
  → Cursor encode cả created_at + id
```

## 5. Versioning

```
3 strategies:

1. URL Path (phổ biến nhất):
   /api/v1/users, /api/v2/users
   ✅ Dễ hiểu, dễ route
   ❌ URL thay đổi, break bookmarks

2. Header:
   Accept: application/vnd.myapp.v2+json
   ✅ Clean URLs
   ❌ Khó test (cần set header), ít visible

3. Query Parameter:
   /api/users?version=2
   ✅ Dễ test
   ❌ Cluttered URLs, caching issues

Best Practice:
  - Dùng URL path cho major versions
  - Minor changes → backward compatible (add fields, not remove)
  - Sunset policy: v1 supported 12 months after v2 release
```

## 6. Error Handling — RFC 7807

```json
// RFC 7807 Problem Details — standard error format
{
  "type": "https://api.example.com/errors/validation",
  "title": "Validation Error",
  "status": 422,
  "detail": "Email field is required and must be valid",
  "instance": "/api/users",
  "errors": [
    { "field": "email", "message": "Email is required" },
    { "field": "age", "message": "Must be a positive number" }
  ],
  "traceId": "abc-123-def"
}
```

```typescript
// NestJS implementation
@Catch()
export class GlobalExceptionFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const res = ctx.getResponse();
    const req = ctx.getRequest();

    const status =
      exception instanceof HttpException ? exception.getStatus() : 500;

    res.status(status).json({
      type: `https://api.example.com/errors/${status}`,
      title: this.getTitle(status),
      status,
      detail: exception instanceof Error ? exception.message : "Unknown error",
      instance: req.url,
      traceId: req.headers["x-trace-id"],
      timestamp: new Date().toISOString(),
    });
  }
}
```

## 7. Rate Limiting

```
4 Algorithms:

┌─────────────────────────────────────────┐
│ 1. FIXED WINDOW                          │
│   Window: 1 min → max 100 requests       │
│   Problem: burst at window boundary      │
│   (100 at :59, 100 at :00 = 200 in 2s!) │
├─────────────────────────────────────────┤
│ 2. SLIDING WINDOW LOG                    │
│   Track timestamp of every request       │
│   Count in last 60s                      │
│   ✅ Accurate  ❌ Memory-heavy           │
├─────────────────────────────────────────┤
│ 3. SLIDING WINDOW COUNTER                │
│   Weighted between current + previous    │
│   window. Balance accuracy + memory      │
├─────────────────────────────────────────┤
│ 4. TOKEN BUCKET (phổ biến nhất)          │
│   Bucket holds tokens (capacity)         │
│   Tokens added at fixed rate             │
│   Request consumes 1 token               │
│   No tokens → 429!                       │
│   ✅ Allows bursts  ✅ Simple            │
└─────────────────────────────────────────┘
```

```typescript
// Redis-based Token Bucket
async function rateLimit(userId: string, limit: number, windowSec: number) {
  const key = `rate:${userId}`;
  const current = await redis.incr(key);
  if (current === 1) await redis.expire(key, windowSec);
  if (current > limit) {
    const ttl = await redis.ttl(key);
    throw new TooManyRequestsException({
      retryAfter: ttl,
      limit,
      remaining: 0,
    });
  }
  return { remaining: limit - current };
}

// Response headers:
// X-RateLimit-Limit: 100
// X-RateLimit-Remaining: 42
// X-RateLimit-Reset: 1709300060 (unix timestamp)
// Retry-After: 30 (seconds, on 429)
```

## 8. Idempotency

```
Idempotent = gọi N lần cho KẾT QUẢ GIỐNG NHAU

  GET    ✅ Idempotent + Safe (không thay đổi state)
  HEAD   ✅ Idempotent + Safe
  PUT    ✅ Idempotent (replace toàn bộ)
  DELETE ✅ Idempotent (delete lần 2 = 404, nhưng state giống)
  PATCH  ❌ Không nhất thiết (increment counter → mỗi lần khác)
  POST   ❌ Không idempotent (create → mỗi lần tạo mới)

  POST idempotent với Idempotency Key:
  POST /api/payments
  Idempotency-Key: "unique-uuid-123"

  Server: key chưa thấy → process + store result
          key đã thấy → return stored result (không process lại!)
  → Safe retry: network timeout → retry cùng key → không charge 2 lần!
```

```typescript
// Idempotency middleware
async function idempotencyMiddleware(req, res, next) {
  const key = req.headers["idempotency-key"];
  if (!key) return next();

  const cached = await redis.get(`idem:${key}`);
  if (cached) {
    const { status, body } = JSON.parse(cached);
    return res.status(status).json(body);
  }

  // Intercept response to cache it
  const originalJson = res.json.bind(res);
  res.json = (body) => {
    redis.set(
      `idem:${key}`,
      JSON.stringify({ status: res.statusCode, body }),
      "EX",
      86400,
    );
    return originalJson(body);
  };
  next();
}
```

## 9. Câu Hỏi Phỏng Vấn

**Q: Cursor-based vs Offset-based pagination — trade-offs?**

> A: **Offset**: dễ implement, jump to page N. Nhưng: O(N) performance (OFFSET 10000 = scan 10000 rows), inconsistent khi data thay đổi. **Cursor**: O(1) với index (WHERE id > cursor), consistent results. Nhưng: không jump to page, implement phức tạp hơn. Dùng cursor cho: real-time feeds, large datasets, mobile infinite scroll. Dùng offset cho: admin dashboards, small datasets.

**Q: Rate limiting: Token Bucket vs Sliding Window?**

> A: **Token Bucket**: cho phép bursts (tốt cho API), simple (Redis INCR + EXPIRE). **Sliding Window Counter**: chính xác hơn, smooth traffic. Token Bucket phổ biến nhất vì: natural burst handling, easy to implement, low memory.

**Q: Idempotency key — tại sao cần?**

> A: Network không reliable. Client timeout → không biết server đã process hay chưa → retry. Không có idempotency key → charge 2 lần! Idempotency key = unique ID per request → server check: đã process? return cached result. Stripe, PayPal đều dùng.
