## 1. Compute

```
EC2 (Elastic Compute Cloud):
  Virtual machines — full control
  Instance types:
    t3.micro  → burstable, dev/test
    c6i.xlarge→ compute-optimized (CPU-heavy)
    r6i.xlarge→ memory-optimized (Redis, in-memory DB)
    m6i.xlarge→ general purpose (most workloads)

  Auto Scaling Group:
    Min: 2, Max: 10, Target CPU: 70%
    → Scale out khi CPU > 70%, scale in khi < 30%

Lambda (Serverless):
  Event → Function → Result → Shutdown
  ✅ Pay per invocation, auto-scale to 0
  ❌ Cold start (100-500ms Node.js), 15 min timeout
  ❌ No persistent connections
  Concurrency: reserved (guaranteed) vs provisioned (warm)

ECS/Fargate:
  ECS: container orchestration (AWS-managed)
  Fargate: serverless containers (no EC2 management)
  → Between Lambda (FaaS) and EC2 (IaaS)
  ECS + Fargate ← phổ biến nhất cho microservices trên AWS

EKS:
  Managed Kubernetes on AWS
  → Khi đã invest vào K8s ecosystem
```

## 2. Storage

```
S3 (Simple Storage Service):
  Object storage: files, images, backups, logs

  Storage Classes:
    Standard:            frequent access        ($0.023/GB)
    Standard-IA:         infrequent access       ($0.0125/GB)
    Glacier Instant:     archive, instant access  ($0.004/GB)
    Glacier Deep Archive: rare access (12h retrieval) ($0.00099/GB)

  Lifecycle policies: Standard → IA (30 days) → Glacier (90 days)

  Pre-signed URLs:
    const url = s3.getSignedUrl("putObject", {
      Bucket: "uploads", Key: "photo.jpg", Expires: 300
    });
    // Client uploads DIRECTLY to S3 (bypass server!)
    // ✅ Giảm load server, ✅ faster upload

EBS (Elastic Block Store):
  Disk volumes cho EC2 (like SSD attached to VM)
  gp3: general purpose SSD ($0.08/GB)
  io2: high IOPS (databases)

EFS (Elastic File System):
  Shared file system (multiple EC2/containers mount cùng lúc)
  NFS protocol, auto-scaling
```

## 3. Database

```
RDS (Relational Database Service):
  Managed PostgreSQL, MySQL, MariaDB
  Multi-AZ: automatic failover (standby in another AZ)
  Read Replicas: up to 15 replicas for read scaling
  Auto backups: point-in-time recovery

Aurora:
  AWS-optimized MySQL/PostgreSQL
  5x throughput MySQL, 3x PostgreSQL
  Storage auto-scales (10GB → 128TB)
  Aurora Serverless: auto-scale compute to 0

DynamoDB:
  Managed NoSQL (key-value + document)

  Partition Key design (QUAN TRỌNG!):
    ✅ High cardinality: userId, orderId
    ❌ Low cardinality: status="active" → hot partition!

  GSI (Global Secondary Index):
    Query by non-primary-key attributes
    Eventually consistent

  LSI (Local Secondary Index):
    Same partition key, different sort key
    Strongly consistent

  On-demand vs Provisioned capacity:
    On-demand: auto-scale, pay per request
    Provisioned: cheaper if predictable traffic

ElastiCache:
  Managed Redis or Memcached
  → Session store, caching, rate limiting
```

```
  Security Group: stateful firewall (instance-level)
    Allow inbound: port 3000 from ALB SG
    Allow outbound: all (default)

  NACL: stateless firewall (subnet-level)
    Allow/Deny rules, evaluated by number order

  NAT Gateway: private subnet → internet (outbound only)
  Internet Gateway: public subnet ↔ internet

Route 53: managed DNS
  Record types: A, CNAME, ALIAS
  Routing: simple, weighted, latency-based, geolocation, failover

CloudFront: CDN (edge caching worldwide)
```

## 5. Messaging

```
SQS (Simple Queue Service):
  Standard: unlimited throughput, at-least-once, best-effort ordering
  FIFO: exactly-once, strict ordering, 300 msg/s (3000 with batching)
  DLQ (Dead Letter Queue): failed messages sent here
  Visibility Timeout: message invisible during processing

SNS (Simple Notification Service):
  Pub/Sub: 1 message → multiple subscribers
  Fan-out pattern: SNS → SQS queue 1 + SQS queue 2 + Lambda

EventBridge:
  Event bus: event routing rules
  → More powerful than SNS for event-driven architectures

Kinesis:
  Real-time data streaming (like Kafka-managed)
  Kinesis Data Streams: ingest real-time data
  Kinesis Firehose: load to S3/Redshift/Elasticsearch
```

## 6. Monitoring & IAM

```
CloudWatch:
  Metrics: CPU, memory, request count, custom metrics
  Logs: centralized logging from all services
  Alarms: CPU > 80% → SNS notification → PagerDuty

X-Ray:
  Distributed tracing (like Jaeger/Zipkin)
  Trace requests across Lambda, API Gateway, ECS

CloudTrail:
  Audit log: WHO did WHAT and WHEN on AWS
  → Security compliance, forensics

IAM (Identity & Access Management):
  Least Privilege Principle:
    ✅ Lambda role: only S3:GetObject on specific bucket
    ❌ Lambda role: S3:* on * (too broad!)

  Policy structure:
  {
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }

  Cross-account: assume role in another account
  Service roles: EC2/Lambda assume role to access other services
```

## 7. Câu Hỏi Phỏng Vấn

**Q: SQS Standard vs FIFO?**

> A: **Standard**: unlimited throughput, at-least-once delivery, best-effort ordering. Dùng cho: background jobs, notifications. **FIFO**: exactly-once, strict order, 300 msg/s. Dùng cho: payment processing, order sequencing. Trade-off: throughput vs ordering guarantee.

**Q: DynamoDB partition key design?**

> A: High cardinality (userId, orderId) → even distribution. Low cardinality (status, country) → hot partition → throttling! Composite key: partition key (userId) + sort key (timestamp) cho range queries. GSI cho query patterns khác primary key.
