## 1. Docker Deep Dive

### Dockerfile Best Practices

```dockerfile
# ❌ BAD: large image, root user, no caching
FROM node:20
COPY . .
RUN npm install
CMD ["node", "index.js"]

# ✅ GOOD: multi-stage, non-root, layer caching
# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production    # deterministic install
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:20-alpine AS production
# Security: non-root user
RUN addgroup -g 1001 appgroup && adduser -u 1001 -G appgroup -s /bin/sh -D appuser
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER appuser                    # NEVER run as root!
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/index.js"]
```

```
Layer Caching:
  Mỗi instruction = 1 layer → cached nếu KHÔNG thay đổi

  COPY package*.json ./ ← thay đổi ít
  RUN npm ci             ← cached nếu package.json không đổi!
  COPY . .               ← source code thay đổi thường xuyên

  → Đặt instructions ÍT thay đổi LÊN TRƯỚC!

Image Size:
  node:20       → ~1GB
  node:20-slim  → ~200MB
  node:20-alpine→ ~130MB
  distroless    → ~80MB (no shell, no package manager!)
```

### Docker Networking

```
Bridge (default):
  Container ↔ Container (cùng host, cùng network)
  docker network create my-net
  docker run --network my-net app1
  docker run --network my-net app2
  → app1 gọi app2 bằng container name: http://app2:3000

Host:
  Container share network với host
  → Không port mapping, performance tốt hơn
  → ❌ Port conflicts

Overlay:
  Container ↔ Container (khác hosts, Swarm/K8s)
  → Multi-host networking
```

### Docker Compose

```yaml
# docker-compose.yml
version: "3.8"
services:
  api:
    build: .
    ports: ["3000:3000"]
    environment:
      DATABASE_URL: postgres://db:5432/app
      REDIS_URL: redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: app
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s

  cache:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

volumes:
  pgdata:
```

### Core Resources

```yaml
# Deployment — manage ReplicaSets → manage Pods
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myapp:1.0.0
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "250m" # 0.25 CPU cores
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
```

### Services

```yaml
# ClusterIP — internal only (default)
apiVersion: v1
kind: Service
metadata:
  name: api-service
spec:
  type: ClusterIP # chỉ access TRONG cluster
  selector:
    app: api
  ports:
    - port: 80
      targetPort: 3000

# NodePort — expose on each node's IP
# LoadBalancer — cloud LB (AWS ELB, GCP LB)
# Ingress — HTTP routing (like reverse proxy)
```

```yaml
# Ingress — HTTP routing rules
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /users
            pathType: Prefix
            backend:
              service:
                name: user-service
                port:
                  number: 80
          - path: /orders
            pathType: Prefix
            backend:
              service:
                name: order-service
                port:
                  number: 80
```

### Auto-Scaling & Config

```yaml
# HPA — Horizontal Pod Autoscaler
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70 # scale khi CPU > 70%


# ConfigMap — non-sensitive config
# Secret — sensitive data (base64 encoded, encrypt at rest)
# PVC — persistent storage cho StatefulSets
```

```
Key Resources Summary:
  Pod:          smallest unit, 1+ containers
  Deployment:   manage Pods (rolling update, rollback)
  StatefulSet:  ordered Pods, stable network IDs (databases)
  DaemonSet:    1 Pod per node (monitoring agents, log collectors)
  Job:          run-to-completion (batch processing)
  CronJob:      scheduled Jobs
  Namespace:    virtual clusters (dev, staging, production)
  RBAC:         Role-Based Access Control
```

## 3. Helm

```yaml
# Helm = package manager cho Kubernetes
# Chart = template + values = Kubernetes manifests

# values.yaml
replicaCount: 3
image:
  repository: myapp
  tag: "1.0.0"
resources:
  limits:
    cpu: 500m
    memory: 512Mi

# template/deployment.yaml
spec:
  replicas: {{ .Values.replicaCount }}
  containers:
    - image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
      resources: {{ toYaml .Values.resources | nindent 8 }}

# Commands:
# helm install my-release ./my-chart
# helm upgrade my-release ./my-chart --set image.tag=2.0.0
# helm rollback my-release 1
```

## 4. Service Mesh

```
Problem: microservices communication = complex!
  mTLS, retry, circuit breaker, tracing cho MỖI service?

Service Mesh (Istio, Linkerd):
  ┌──────────────────────────────┐
  │         POD                  │
  │  ┌─────────┐  ┌──────────┐  │
  │  │  App    │←→│  Sidecar │  │  ← Envoy proxy
  │  │Container│  │  Proxy   │  │
  │  └─────────┘  └──────────┘  │
  └──────────────────────────────┘

  ALL traffic goes through sidecar proxy:
    ✅ mTLS (mutual TLS): encrypted service-to-service
    ✅ Traffic management: canary, A/B, retry, timeout
    ✅ Circuit breaker: per-service
    ✅ Observability: tracing, metrics (automatic!)
    ❌ Complexity: extra resource usage, latency (~1ms/hop)
```

## 5. Câu Hỏi Phỏng Vấn

**Q: Docker multi-stage build — tại sao?**

> A: Stage 1: install dev deps + build. Stage 2: copy ONLY production artifacts. Result: image nhỏ hơn 5-10x (không chứa devDependencies, source code, build tools). Security: smaller attack surface.

**Q: K8s Deployment vs StatefulSet?**

> A: **Deployment**: stateless apps (API servers). Pods interchangeable, scale freely. **StatefulSet**: stateful apps (databases). Stable network identity (pod-0, pod-1), ordered scaling, persistent storage (PVC). Database replicas cần biết primary/secondary → StatefulSet.
