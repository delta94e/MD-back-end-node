## 1. Tại Sao IaC?

```
Manual (Console):
  ❌ Không reproducible (click-ops)
  ❌ Không version control
  ❌ Không review process
  ❌ Drift: actual ≠ expected state

Infrastructure as Code:
  ✅ Reproducible: chạy lại = kết quả giống
  ✅ Version controlled: Git history, PR review
  ✅ Testable: terraform plan, cdk diff
  ✅ Self-documenting: code IS documentation
  ✅ Consistent: dev = staging = production
```

## 2. Terraform

```hcl
# main.tf — Declare infrastructure
terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }

  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "ap-southeast-1"
    dynamodb_table = "terraform-locks"     # state locking!
    encrypt        = true
  }
}

provider "aws" { region = "ap-southeast-1" }

# VPC Module
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
  name    = "production"
  cidr    = "10.0.0.0/16"
  azs     = ["ap-southeast-1a", "ap-southeast-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.3.0/24", "10.0.4.0/24"]
}

# RDS
resource "aws_db_instance" "postgres" {
  identifier     = "prod-db"
  engine         = "postgres"
  engine_version = "16"
  instance_class = "db.t3.medium"
  allocated_storage = 100
  multi_az       = true               # failover!

  db_subnet_group_name   = module.vpc.database_subnet_group
  vpc_security_group_ids = [aws_security_group.db.id]

  backup_retention_period = 7
  deletion_protection     = true
}

# Output
output "db_endpoint" {
  value = aws_db_instance.postgres.endpoint
}
```

```
Terraform Workflow:
  terraform init     → download providers, init backend
  terraform plan     → preview changes (ALWAYS review!)
  terraform apply    → create/update resources
  terraform destroy  → delete all resources

State Management:
  terraform.tfstate = source of truth for current infra
  ⚠️ NEVER store state locally in production!
  → Remote state: S3 + DynamoDB locking (prevent concurrent apply)

Modules:
  Reusable components (like functions for infrastructure)
  module "vpc" { source = "terraform-aws-modules/vpc/aws" }

Workspaces:
  terraform workspace new staging
  terraform workspace select production
  → Same code, different state files per environment
```

## 3. AWS CDK

```typescript
// CDK = Infrastructure as TypeScript (imperative IaC)
import * as cdk from "aws-cdk-lib";
import * as ec2 from "aws-cdk-lib/aws-ec2";
import * as rds from "aws-cdk-lib/aws-rds";
import * as ecs from "aws-cdk-lib/aws-ecs";

export class AppStack extends cdk.Stack {
  constructor(scope: cdk.App, id: string) {
    super(scope, id);

    // L2 Constructs (high-level, opinionated)
    const vpc = new ec2.Vpc(this, "VPC", {
      maxAzs: 2,
      natGateways: 1,
    });

    const db = new rds.DatabaseInstance(this, "Database", {
      engine: rds.DatabaseInstanceEngine.postgres({
        version: rds.PostgresEngineVersion.VER_16,
      }),
      instanceType: ec2.InstanceType.of(
        ec2.InstanceClass.T3,
        ec2.InstanceSize.MEDIUM,
      ),
      vpc,
      multiAz: true,
      deletionProtection: true,
    });

    // ECS Fargate Service
    const cluster = new ecs.Cluster(this, "Cluster", { vpc });
    new ecs.FargateService(this, "API", {
      cluster,
      taskDefinition: taskDef,
      desiredCount: 2,
    });
  }
}

// Construct Levels:
// L1: CloudFormation resources 1:1 (CfnBucket)
// L2: Opinionated defaults (Bucket → versioning, encryption)
// L3: Patterns (ApplicationLoadBalancedFargateService)
```

```bash
# CDK Workflow:
cdk synth   # generate CloudFormation template
cdk diff    # preview changes
cdk deploy  # deploy stack
cdk destroy # delete stack
```

## 4. Terraform vs CDK

| Feature     | Terraform                         | AWS CDK                             |
| ----------- | --------------------------------- | ----------------------------------- |
| Language    | HCL (declarative)                 | TypeScript/Python (imperative)      |
| Cloud       | **Multi-cloud** (AWS, GCP, Azure) | AWS only                            |
| State       | Self-managed (S3)                 | CloudFormation                      |
| Learning    | Lower (HCL simple)                | Higher (programming knowledge)      |
| Flexibility | Moderate                          | **High** (loops, conditions, types) |
| Community   | **Largest**                       | Growing                             |

```
Dùng Terraform khi:
  ✅ Multi-cloud strategy
  ✅ Team familiar with HCL
  ✅ Large ecosystem of modules

Dùng CDK khi:
  ✅ AWS-only
  ✅ Team = TypeScript developers
  ✅ Complex logic (conditions, loops)
  ✅ Type safety + IDE support
```

## 5. Câu Hỏi Phỏng Vấn

**Q: Terraform state — tại sao quan trọng?**

> A: State file = mapping giữa code và real infrastructure. Terraform compare desired (code) vs current (state) → plan changes. Remote state (S3 + DynamoDB lock) cho team: prevent concurrent apply, shared source of truth. State drift: actual infra ≠ state → `terraform refresh` hoặc import.
