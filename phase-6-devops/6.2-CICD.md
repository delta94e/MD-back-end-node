## 1. Pipeline Design

```
CI/CD Pipeline:

  ┌──────┐  ┌──────┐  ┌────────┐  ┌──────┐  ┌──────────┐  ┌──────────┐
  │ Push │→│Build │→│  Test  │→│ Scan │→│ Deploy   │→│ Deploy   │
  │      │  │      │  │        │  │      │  │ Staging  │→│Production│
  └──────┘  └──────┘  └────────┘  └──────┘  └──────────┘  └──────────┘
                                     │
                               Security Scan:
                               - SAST (static code)
                               - Dependency audit
                               - Container scan
                               - Secret detection

CI (Continuous Integration):
  Code push → build → unit tests → integration tests → artifacts
  ✅ Mỗi commit được test → phát hiện bugs sớm

CD (Continuous Delivery):
  Artifacts → deploy to staging → manual approval → production

CD (Continuous Deployment):
  Artifacts → deploy to staging → auto tests → AUTO deploy production
  → Không cần manual approval (mature teams only!)
```

## 2. GitHub Actions

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_DB: test
          POSTGRES_PASSWORD: test
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"
      - run: npm ci
      - run: npm run lint
      - run: npm run test:unit
      - run: npm run test:integration
        env:
          DATABASE_URL: postgres://postgres:test@localhost:5432/test

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --production
      - uses: aquasecurity/trivy-action@master # container scan
        with:
          image-ref: myapp:${{ github.sha }}

  deploy:
    needs: [test, security]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Production
        run: |
          # kubectl, helm, or AWS deploy commands
          helm upgrade --install myapp ./chart \
            --set image.tag=${{ github.sha }}
```

## 3. Deployment Strategies

```
ROLLING UPDATE (K8s default):
  Old: [v1] [v1] [v1] [v1]
  →    [v2] [v1] [v1] [v1]    ← thay từng pod
  →    [v2] [v2] [v1] [v1]
  →    [v2] [v2] [v2] [v2]
  ✅ Zero downtime, gradual
  ❌ Mixed versions during rollout

BLUE/GREEN:
  Blue (v1): ████████ ← live traffic
  Green (v2): ████████ ← idle, deploy + test

  Ready? Switch traffic:
  Blue (v1): ████████ ← idle (rollback ready!)
  Green (v2): ████████ ← live traffic

  ✅ Instant rollback (switch back)
  ❌ 2x infrastructure cost

CANARY:
  v1: ████████████████████ (95% traffic)
  v2: ██ (5% traffic) ← monitor errors, latency

  OK? Gradually increase:
  v1: ██████████████ (70%)
  v2: ██████████ (30%)
  ...
  v2: ████████████████████ (100%)

  ✅ Low risk, real traffic testing
  ❌ Complex traffic splitting, longer rollout
```

## 4. GitOps

```
GitOps = Git is the SINGLE SOURCE OF TRUTH for infrastructure

  Developer → Push manifest changes to Git
  ArgoCD/Flux → Watch Git → Auto-sync to cluster

  ┌──────┐    ┌──────┐    ┌────────┐    ┌─────────┐
  │ Dev  │───▶│ Git  │◀───│ ArgoCD │───▶│  K8s    │
  │      │    │ Repo │    │ (watch)│    │ Cluster │
  └──────┘    └──────┘    └────────┘    └─────────┘

  ✅ Audit trail (Git history)
  ✅ Declarative (desired state in Git)
  ✅ Auto-sync (drift detection → fix)
  ✅ Easy rollback (git revert)
```

## 5. Feature Flags

```
Feature Flag = toggle feature ON/OFF without deploy

  if (featureFlags.isEnabled("new-checkout")) {
    showNewCheckout();
  } else {
    showOldCheckout();
  }

  Use cases:
    - Trunk-based development (merge to main, hide behind flag)
    - Gradual rollout (10% → 50% → 100%)
    - A/B testing (variant A vs B)
    - Kill switch (disable broken feature instantly)
    - Beta access (specific users/groups)

  Tools: LaunchDarkly, Unleash, Flagsmith
  ⚠️ Tech debt: remove flags after rollout complete!
```

## 6. Câu Hỏi Phỏng Vấn

**Q: Blue/Green vs Canary deployment?**

> A: **Blue/Green**: instant switch, instant rollback. 2x cost. Dùng khi: need instant rollback, database migrations. **Canary**: gradual (5% → 100%), real traffic testing. Dùng khi: need validation with real users, minimize blast radius. Can combine: canary first → blue/green switch.
