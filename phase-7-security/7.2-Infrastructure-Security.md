## 2. Secrets Management

```
Secret Lifecycle:
  Create вҶ’ Store вҶ’ Access вҶ’ Rotate вҶ’ Revoke

вқҢ BAD PRACTICES:
  - Hardcode in source code: const API_KEY = "sk_live_abc123"
  - Commit .env to Git (even private repos!)
  - Share secrets via Slack/email
  - Same secrets across environments

вң… GOOD PRACTICES:
  - .env in .gitignore
  - Different secrets per environment (dev вү  staging вү  prod)
  - Rotate regularly (every 90 days)
  - Audit access logs
```

## 3. Container Security

```dockerfile
# Security Best Practices:

# 1. Minimal base image
FROM node:20-alpine     # NOT node:20 (Debian full)
# OR distroless (no shell, no package manager!)
FROM gcr.io/distroless/nodejs20-debian12

# 2. Non-root user
RUN addgroup -g 1001 app && adduser -u 1001 -G app -s /bin/sh -D app
USER app

# 3. Read-only filesystem
# docker run --read-only --tmpfs /tmp myapp
# K8s: securityContext.readOnlyRootFilesystem: true

# 4. No new privileges
# K8s: securityContext.allowPrivilegeEscalation: false

# 5. Drop all capabilities
# K8s: securityContext.capabilities.drop: ["ALL"]
```

```yaml
# Kubernetes Pod Security Context
apiVersion: v1
kind: Pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1001
    fsGroup: 1001
  containers:
    - name: app
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
      volumeMounts:
        - name: tmp
          mountPath: /tmp
  volumes:
    - name: tmp
      emptyDir: {}
```

```
Image Scanning:
  Build time: Trivy, Snyk Container, Docker Scout
  Runtime: Falco (detect anomalous behavior)

  trivy image myapp:latest
  вҶ’ CVE-2024-xxxx (HIGH): upgrade libssl
```

## 4. Security Architecture

```
Zero Trust Architecture:
  "Never trust, always verify"

  Traditional: trust everything INSIDE network perimeter
  Zero Trust:  trust NOTHING, verify EVERY request

  Principles:
    1. Verify identity: every request authenticated
    2. Least privilege: minimum access needed
    3. Assume breach: design as if already compromised
    4. Micro-segmentation: isolate services
    5. Continuous monitoring: detect anomalies

  Implementation:
    вң… mTLS between services (service mesh)
    вң… Short-lived credentials (auto-expire)
    вң… Identity-based access (not network-based)
    вң… Audit all access
```

## 5. CГўu Hб»Ҹi Phб»Ҹng VбәҘn

**Q: Secrets management вҖ” best practices?**

> A: (1) NEVER in source code or Git. (2) Use managed service (AWS Secrets Manager, Vault). (3) Automatic rotation every 90 days. (4) Different secrets per environment. (5) Least privilege IAM policies. (6) Audit access logs. (7) K8s: External Secrets Operator, not plain K8S Secrets.

**Q: Container security вҖ” how to harden?**

> A: (1) Minimal base image (alpine/distroless). (2) Non-root user. (3) Read-only filesystem. (4) Drop ALL capabilities. (5) No privilege escalation. (6) Scan images (Trivy). (7) Use image digest (sha256), not tags. (8) Runtime monitoring (Falco).

**Q: Zero Trust вҖ” giбәЈi thГӯch?**

> A: KhГҙng trust bбәҘt kб»і request nГ o, kб»ғ cбәЈ tб»« internal network. Mб»Қi request phбәЈi: authenticated (mTLS, JWT), authorized (RBAC, check permissions), encrypted (TLS), logged (audit trail). Thay vГ¬ perimeter security (firewall = trusted inside), verify Tб»ӘNG request. Implementation: service mesh (mTLS), short-lived tokens, micro-segmentation.
