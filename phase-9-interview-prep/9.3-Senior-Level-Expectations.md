## 1. Junior vs Mid vs Senior

```
JUNIOR (0-2 years):
  ✅ Can: complete tasks with guidance
  ✅ Focus: learn, ask questions, write working code
  ❌ Need: mentoring, code review, clear requirements
  Scope: single features, small tasks

MID (2-5 years):
  ✅ Can: work independently on features
  ✅ Focus: quality, testing, best practices
  ❌ Need: architectural guidance, prioritization help
  Scope: features, small projects

SENIOR (5+ years):
  ✅ Can: design systems, mentor team, drive technical decisions
  ✅ Focus: architecture, scalability, team productivity
  ✅ Own: end-to-end delivery, monitoring, operational excellence
  Scope: projects, cross-team initiatives, org-level impact

STAFF (special):
  ✅ Can: define technical direction, solve org-wide problems
  ✅ Focus: company-wide architecture, technical strategy
  Scope: multi-team, company-level impact
```

## 2. 7 Kỹ Năng Senior

### Technical Depth

```
Không chỉ biết DÙNG, mà hiểu TẠI SAO và THẾ NÀO:

Junior: "Tôi dùng Redis để cache"
Senior: "Tôi dùng Redis cache-aside pattern với LRU eviction,
  TTL 5 phút cho product catalog vì:
  - Read/write ratio 100:1 → cache-aside phù hợp
  - Product data thay đổi ít → 5 min TTL acceptable
  - LRU vì working set nhỏ hơn memory
  - Dùng Redis Cluster 3 shards cho HA
  - Thundering herd: mutex lock + stale-while-revalidate"

→ Giải thích từ HIGH-LEVEL xuống BYTE-LEVEL
→ Nêu TẠI SAO chọn approach này
→ Nêu ALTERNATIVES đã consider
```

### Technical Breadth

```
Biết NHIỀU options + trade-offs cho mỗi bài toán:

"Chọn database?" → Không phải "PostgreSQL tốt nhất"
→ "PostgreSQL cho strong consistency + complex queries.
   DynamoDB cho high scale + simple access patterns.
   MongoDB cho flexible schema + rapid prototyping.
   Depends on: consistency needs, query patterns, scale, team expertise"

Senior biết KHI NÀO dùng cái gì
Junior chỉ biết CÁI MÌNH BIẾT
```

### System Thinking

```
Nhìn BỨC TRANH TOÀN CẢNH:

Junior: "Tôi thêm field X vào API"
Senior: "Thêm field X ảnh hưởng:
  - API: backward compatibility? versioning cần?
  - Database: migration cần? index cần?
  - Cache: invalidation cần?
  - Consumers: frontend/mobile cần update?
  - Testing: integration tests update?
  - Monitoring: new metrics cần?
  - Documentation: API docs update?"

→ Hiểu IMPACT của mỗi thay đổi qua ENTIRE system
```

```
  Frontend  Backend  DB  DevOps  AI
     │         │      │    │      │
     │    ┌────┴────┐ │    │      │

     │    │ Node.js │ │    │      │
     │    │ System  │ │    │      │

     │    │   DB    │ │    │      │
     │    └─────────┘ │    │      │
                      │
              DEEP in your area
              AWARE of adjacent areas

    DEEP: Node.js, PostgreSQL, System Design, API Design
    AWARE: Frontend basics, Docker/K8s, AWS, Security
    FAMILIAR: ML basics, mobile, data engineering
```

## 4. Trade-off Analysis

```
Senior KHÔNG NÓI "X tốt hơn Y"
Senior NÓI "X tốt hơn Y TRONG CONTEXT Z VÌ..."

Framework:
  1. State the context: traffic, team size, timeline
  2. List options: A, B, C
  3. Analyze each: pros, cons, risks
  4. Recommend: "Given our context, I recommend B because..."
  5. Acknowledge risks: "The risk is..., mitigated by..."

Ví dụ:
  "Monolith vs Microservices cho startup?"

  Context: 3 developers, MVP, 6 month deadline

  Monolith:
    ✅ Ship faster, simpler ops, shared DB
    ❌ Tech debt accumulates, hard to scale specific parts

  Microservices:
    ✅ Independent scaling, tech flexibility
    ❌ Distributed complexity, 3 devs = overhead

  Recommend: "Modular Monolith — ship fast now,
  organized for future extraction. Conway's Law:
  3 devs = 3 services MAX, but we don't need that yet."
```

## 5. Ownership & Impact

```
Senior Ownership = END-TO-END:

  ❌ Junior: "Tôi code xong, QA sẽ test"
  ✅ Senior: "Tôi own từ design → implement → test →
    deploy → monitor → on-call → iterate"

  Code:        write, review, test
  Deploy:      CI/CD, rollout strategy
  Monitor:     alerts, dashboards, SLO/SLA
  On-call:     incident response, postmortem
  Documentation: design docs, runbooks
  Roadmap:     propose improvements, reduce tech debt

Measure Impact:
  ✅ "Giảm p99 latency 80% → improved user retention 5%"
  ✅ "Automated deployment → 5x faster release cycle"
  ✅ "Mentored 3 junior → 2 promoted to mid in 6 months"
  ✅ "Reduced on-call incidents 60% with better testing"
```

## 6. Communication & Documentation

```
RFC (Request for Comments) / Design Doc:

  Structure:
  1. Problem Statement: what problem are we solving?
  2. Context: why now? constraints?
  3. Proposed Solution: architecture, data model, API
  4. Alternatives Considered: why NOT other approaches
  5. Risks & Mitigations: what could go wrong
  6. Timeline & Milestones: phased approach
  7. Open Questions: cần input từ team

  ⚠️ Senior viết RFC TRƯỚC code
  → Team review, feedback, alignment
  → Better design, fewer rewrites

Presenting to Stakeholders:
  Technical audience: architecture deep dive
  Product/Business: impact, timeline, trade-offs (no jargon!)
  Executive: 3 bullets: problem, solution, impact

  "We're migrating to microservices.
   This reduces deploy time from 2 hours to 15 minutes,
   enabling 5x faster feature delivery to customers."
```
