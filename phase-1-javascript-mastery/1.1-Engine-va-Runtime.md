## 1. V8 Engine Architecture

V8 là JavaScript engine của Chrome & Node.js, viết bằng C++.

```
Source Code (JavaScript)
        │
        ▼
   ┌──────────┐
   │  PARSER  │    Lexical Analysis → Tokens → Syntax Analysis
   └────┬─────┘
        │
        ▼
   ┌──────────┐
   │   AST    │    Abstract Syntax Tree — cây biểu diễn code
   └────┬─────┘
        │
        ▼
   ┌──────────────┐
   │   IGNITION   │    Interpreter — chạy ngay, generate bytecode
   │ (Interpreter)│    → Nhanh khởi động, không cần compile trước
   └──────┬───────┘
          │
          │ Hot function detected (gọi nhiều lần)
          │ Collect type feedback
          ▼
   ┌──────────────┐
   │   TURBOFAN   │    JIT Compiler — compile sang Machine Code
   │(JIT Compiler)│    → Tối ưu dựa trên type feedback
   └──────┬───────┘
          │
          │ Type thay đổi? (bail out)
          │ De-optimize → quay lại Ignition
          ▼
   Machine Code (tối ưu, chạy trực tiếp trên CPU)
```

### Hidden Classes & Inline Caches

```javascript
// V8 tạo "Hidden Class" cho mỗi object shape

// ✅ TỐT — cùng shape → cùng Hidden Class → inline cache HIT
function Point(x, y) {
  this.x = x; // Hidden Class C0 → C1 (add x)
  this.y = y; // Hidden Class C1 → C2 (add x, y)
}
const p1 = new Point(1, 2); // shape: {x, y} → Hidden Class C2
const p2 = new Point(3, 4); // CÙNG shape → REUSE C2 → NHANH!

// ❌ XẤU — khác shape → khác Hidden Class → inline cache MISS
const obj1 = { x: 1, y: 2 }; // HC_1: {x, y}
const obj2 = { y: 2, x: 1 }; // HC_2: {y, x} ← KHÁC THỨ TỰ!
const obj3 = { x: 1, y: 2, z: 3 }; // HC_3: {x, y, z} ← THÊM property

// ❌ Delete property → V8 fallback sang dictionary mode (CHẬM!)
delete obj1.x; // Phá hủy Hidden Class optimization
```

## 2. Memory Management & Garbage Collection

### 2.1 Stack vs Heap

```
STACK (tự động, nhanh, cố định):
  - Primitive values: number, string, boolean, null, undefined, symbol, bigint
  - Function call frames (execution context)
  - References (pointers) đến objects trong Heap
  - LIFO: push khi gọi function, pop khi return

HEAP (dynamic, chậm hơn, linh hoạt):
  - Objects, Arrays, Functions, Maps, Sets
  - Closures (captured variables)
  - Quản lý bởi Garbage Collector

  ┌─── Stack ───┐     ┌──────── Heap ────────────┐
  │ a = 42      │     │                          │
  │ b = "hello" │     │  ┌─────────────────┐     │
  │ ref ────────┼─────┼─▶│ { name: "Alice" }│     │
  │             │     │  └─────────────────┘     │
  │ arr ────────┼─────┼─▶┌───────────────┐       │
  │             │     │  │ [1, 2, 3]     │       │
  └─────────────┘     │  └───────────────┘       │
                      └──────────────────────────┘
```

### 2.2 V8 Generational Garbage Collection

```
V8 chia Heap thành 2 vùng:

┌────────────────────────────────────────────────────┐
│                    V8 HEAP                          │
│                                                    │
│  ┌────────────────┐    ┌──────────────────────┐   │
│  │   YOUNG GEN    │    │      OLD GEN         │   │
│  │   (1-8 MB)     │    │    (hàng trăm MB)    │   │
│  │                │    │                      │   │
│  │  ┌─────┬─────┐ │    │  Objects sống lâu    │   │
│  │  │From │ To  │ │    │  (survived 2+ GC)   │   │
│  │  │Space│Space│ │    │                      │   │
│  │  └─────┴─────┘ │    │  Mark-Sweep-Compact  │   │
│  │                │    │  (chậm, ít chạy)     │   │
│  │  Scavenger     │    │                      │   │
│  │  (nhanh, chạy  │    │                      │   │
│  │   thường xuyên)│    │                      │   │
│  └────────────────┘    └──────────────────────┘   │
└────────────────────────────────────────────────────┘

Scavenger (Young Gen — Minor GC):
  1. Objects mới tạo → From-Space
  2. GC: copy objects SỐNG từ From-Space → To-Space
  3. Swap From ↔ To
  4. Objects survive 2 lần GC → promote sang Old Gen
  → NHANH vì hầu hết objects chết sớm ("infant mortality")

Mark-Sweep-Compact (Old Gen — Major GC):
  1. MARK: đánh dấu tất cả reachable objects từ root
  2. SWEEP: thu hồi memory của unmarked objects
  3. COMPACT: di chuyển objects liền kề, giảm fragmentation
  → CHẬM nhưng ít chạy

Orinoco (V8's GC optimizations):
  - Concurrent marking: mark trong background thread
  - Incremental marking: chia nhỏ marking thành nhiều bước
  - Parallel scavenging: nhiều threads cùng GC
  → Giảm "stop-the-world" pauses!
```

### 2.3 Memory Leaks phổ biến

```javascript
// 1. Global variables (quên "use strict")
function leak1() {
  name = "oops"; // quên let/const → global! → KHÔNG BAO GIỜ bị GC
}

// 2. Forgotten timers
const data = loadHugeData();
setInterval(() => {
  processData(data); // data KHÔNG BAO GIỜ bị GC vì timer giữ reference
}, 1000);
// Fix: clearInterval khi không cần

// 3. Closures giữ reference unintentionally
function leak3() {
  const hugeArray = new Array(1000000).fill("x");
  return function () {
    console.log(hugeArray.length); // closure giữ TOÀN BỘ hugeArray!
  };
}

// 4. DOM references (browser)
const elements = {};
elements.button = document.getElementById("btn");
document.body.removeChild(document.getElementById("btn"));
// elements.button vẫn giữ reference → button vẫn trong memory!

// Debug: --inspect --expose-gc, Chrome DevTools Heap Snapshot
```

## 3. Execution Context & Scope

### 3.1 Execution Context là gì

```
Mỗi khi call function → tạo Execution Context (EC) mới

EC gồm:
  1. Variable Environment (VE): var, function declarations
  2. Lexical Environment (LE): let, const, block scope
  3. this binding

Global EC (khi script bắt đầu):
  ┌─────────────────────────────┐
  │ Global Execution Context    │
  │                             │
  │ Variable Environment:       │
  │   var a = 10                │
  │   function foo() { ... }    │
  │                             │
  │ Lexical Environment:        │
  │   let b = 20                │
  │   const c = 30              │
  │                             │
  │ this = window (browser)     │
  │      = global (Node.js)     │
  └─────────────────────────────┘
```

### 3.2 Creation Phase vs Execution Phase

```javascript
console.log(a); // undefined  (var → hoisted, initialized to undefined)
console.log(b); // ReferenceError! (let → TDZ)
console.log(foo); // [Function: foo] (function declaration → hoisted hoàn chỉnh)

var a = 10;
let b = 20;
function foo() {
  return 42;
}

// CREATION PHASE (trước khi chạy code):
// 1. Scan var → allocate memory, set = undefined
// 2. Scan function declarations → allocate memory, set = function
// 3. Scan let/const → allocate memory, BUT NOT initialized (TDZ!)

// EXECUTION PHASE (chạy từng line):
// 1. a = 10 (gán giá trị)
// 2. b = 20 (gán giá trị, ra khỏi TDZ)
```

## 4. Call Stack

```
function multiply(a, b) { return a * b; }
function square(n) { return multiply(n, n); }
function printSquare(n) {
  const result = square(n);
  console.log(result);
}
printSquare(5);

Call Stack evolution:
  ┌───────────┐   ┌───────────┐   ┌───────────┐   ┌───────────┐
  │           │   │           │   │multiply   │   │           │
  │           │   │square(5)  │   │square(5)  │   │square(5)  │
  │printSq(5) │   │printSq(5) │   │printSq(5) │   │printSq(5) │
  │  main()   │   │  main()   │   │  main()   │   │  main()   │
  └───────────┘   └───────────┘   └───────────┘   └───────────┘
   call printSq    call square    call multiply    multiply returns
                                                   (pop)

Stack Overflow:
  function infinite() { infinite(); } // Maximum call stack size exceeded
  // Node.js default: ~10,000-15,000 frames
```

## 5. Event Loop — Chi Tiết Nhất

### 5.1 Tổng quan

```
JavaScript = SINGLE-THREADED. Nhưng non-blocking nhờ EVENT LOOP.

  ┌─────────────────────────────────────────────────┐
  │                 V8 ENGINE                        │
  │  ┌───────────┐    ┌───────────────────────┐     │
  │  │ Call Stack │    │    Memory Heap         │     │
  │  │           │    │                       │     │
  │  │ foo()    │    │ Objects, Closures      │     │
  │  │ main()   │    │                       │     │
  │  └───────────┘    └───────────────────────┘     │
  └──────────┬──────────────────────────────────────┘
             │
   ┌─────────▼──────────┐
   │    EVENT LOOP       │     Libuv (Node.js) / Browser APIs
   │                     │
   │  Microtask Queue    │◀── Promise.then, queueMicrotask
   │  [task1, task2]     │    process.nextTick (Node.js only)
   │                     │
   │  Macrotask Queue    │◀── setTimeout, setInterval, I/O
   │  [task3, task4]     │    setImmediate (Node.js)
   │                     │
   └─────────────────────┘

Event Loop algorithm:
  1. Chạy hết code trong Call Stack
  2. Drain TOÀN BỘ Microtask Queue (bao gồm microtasks mới)
  3. Lấy 1 task từ Macrotask Queue → chạy
  4. Quay lại bước 2
```

### 5.2 Node.js Event Loop Phases (libuv)

```
   ┌───────────────────────────┐
┌─▶│         timers            │  setTimeout, setInterval callbacks
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │     pending callbacks     │  I/O callbacks deferred from previous loop
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │       idle, prepare       │  Internal use only
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │          poll             │  Retrieve new I/O events
│  │                           │  Executes I/O callbacks
│  │  (nếu queue rỗng:         │  ← Phần lớn thời gian ĐỢI ở đây
│  │   check for setImmediate? │
│  │   check for timers?)      │
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │          check            │  setImmediate callbacks
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │      close callbacks      │  socket.on('close'), etc.
│  └───────────┬───────────────┘
└──────────────┘

⚠️ process.nextTick: chạy BETWEEN EVERY PHASE (trước khi chuyển phase)
⚠️ Promise.then (microtask): chạy SAU mỗi macrotask VÀ sau nextTick
```

### 5.3 Phân tích output

```javascript
console.log("1 - start");

setTimeout(() => console.log("2 - setTimeout"), 0);

Promise.resolve().then(() => console.log("3 - promise"));

process.nextTick(() => console.log("4 - nextTick"));

setImmediate(() => console.log("5 - setImmediate"));

console.log("6 - end");

// Output:
// 1 - start     ← synchronous
// 6 - end       ← synchronous
// 4 - nextTick  ← nextTick queue (drain before any phase)
// 3 - promise   ← microtask queue (after nextTick)
// 2 - setTimeout← timers phase
// 5 - setImmediate ← check phase

// Thứ tự ưu tiên:
// 1. Synchronous code (Call Stack)
// 2. process.nextTick (nextTick queue)
// 3. Microtasks (Promise.then, queueMicrotask)
// 4. Macrotasks (setTimeout, setInterval, I/O, setImmediate)
```

## 6. Hoisting

```javascript
// ===== var hoisting =====
console.log(x); // undefined (NOT ReferenceError!)
var x = 5;
// V8 sees: var x = undefined; console.log(x); x = 5;

// ===== let/const — TDZ (Temporal Dead Zone) =====
console.log(y); // ReferenceError: Cannot access 'y' before initialization
let y = 10;
// y EXISTS in memory nhưng CHƯA initialized → TDZ

// ===== function declaration — hoisted HOÀN CHỈNH =====
greet(); // "Hello!" ← works!
function greet() {
  console.log("Hello!");
}

// ===== function expression — chỉ variable hoisted =====
sayHi(); // TypeError: sayHi is not a function
var sayHi = function () {
  console.log("Hi!");
};
// V8 sees: var sayHi = undefined; sayHi(); ← calling undefined!

// ===== TDZ edge case =====
let a = f(); // ReferenceError!
function f() {
  return b;
}
let b = 42;
// f() try to access b → b trong TDZ!
```

## 7. Câu Hỏi Phỏng Vấn

**Q: V8 JIT compilation hoạt động thế nào? Ignition vs TurboFan?**

> A: V8 dùng 2-tier compilation. **Ignition** (interpreter): parse JS → AST → bytecode, chạy ngay. Thu thập type feedback (profiling). **TurboFan** (JIT compiler): function được gọi nhiều lần ("hot") → compile sang optimized machine code dựa trên type feedback. Nếu type thay đổi (ví dụ: function nhận number lần đầu, string lần sau) → **deoptimize** (bail out) quay lại Ignition. Best practice: giữ type consistent.

**Q: Event Loop: console.log output cho đoạn code phức tạp?**

> A: Phân tích: (1) Chạy hết sync code. (2) Drain nextTick queue. (3) Drain microtask queue (Promise.then). (4) 1 macrotask (setTimeout/setImmediate). (5) Lại drain nextTick + microtask. Repeat.

**Q: Memory leak trong Node.js — cách detect và fix?**

> A: Common causes: global variables, forgotten timers/intervals, closures, event listeners không unbind. Detect: `--inspect` + Chrome DevTools Heap Snapshot, `process.memoryUsage()`, `node --expose-gc`. Fix: clear timers, remove listeners, use WeakMap/WeakRef, avoid global state.
