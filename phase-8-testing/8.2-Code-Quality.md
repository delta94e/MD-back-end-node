## 1. SOLID Principles

```typescript
// S — Single Responsibility
// ❌ God class: làm mọi thứ
class UserService {
  createUser() {
    /* ... */
  }
  sendEmail() {
    /* ... */
  } // ← không phải responsibility của UserService!
  generateReport() {
    /* ... */
  } // ← cũng không phải!
}

// ✅ Mỗi class 1 responsibility
class UserService {
  createUser() {}
}
class EmailService {
  sendEmail() {}
}
class ReportService {
  generateReport() {}
}

// O — Open/Closed
// Open for extension, Closed for modification
// ❌ Modify existing code mỗi khi thêm payment method
function processPayment(method: string) {
  if (method === "credit") {
    /* ... */
  } else if (method === "paypal") {
    /* ... */
  } else if (method === "crypto") {
    /* new! must modify! */
  }
}

// ✅ Extend without modifying
interface PaymentProcessor {
  process(amount: number): Promise<void>;
}
class CreditCardProcessor implements PaymentProcessor {
  /* ... */
}
class PayPalProcessor implements PaymentProcessor {
  /* ... */
}
class CryptoProcessor implements PaymentProcessor {
  /* ... */
} // just ADD!

// L — Liskov Substitution
// Subtype phải thay thế được parent type mà không break
// ❌ Square extends Rectangle → setWidth changes height = surprise!
// ✅ Design by contract: subclass honors parent's behavior

// I — Interface Segregation
// ❌ Fat interface
interface Worker {
  work(): void;
  eat(): void; // Robot doesn't eat!
  sleep(): void; // Robot doesn't sleep!
}

// ✅ Split interfaces
interface Workable {
  work(): void;
}
interface Eatable {
  eat(): void;
}
class Robot implements Workable {
  work() {}
}
class Human implements Workable, Eatable {
  work() {}
  eat() {}
}

// D — Dependency Inversion
// High-level modules should NOT depend on low-level modules
// ❌ Direct dependency
class OrderService {
  private db = new PostgresDatabase(); // tightly coupled!
}

// ✅ Depend on abstraction
class OrderService {
  constructor(private db: Database) {} // interface, not implementation!
}
// Inject: new OrderService(new PostgresDatabase())
// Test: new OrderService(new MockDatabase())
```

## 2. Design Patterns

### Creational

```typescript
// Factory Pattern
// Tạo objects mà không expose creation logic
class NotificationFactory {
  static create(type: "email" | "sms" | "push"): Notification {
    switch (type) {
      case "email":
        return new EmailNotification();
      case "sms":
        return new SMSNotification();
      case "push":
        return new PushNotification();
    }
  }
}
const notification = NotificationFactory.create("email");

// Singleton (⚠️ Controversial — impedes testing, hidden dependencies)
class Logger {
  private static instance: Logger;
  private constructor() {}
  static getInstance(): Logger {
    if (!Logger.instance) Logger.instance = new Logger();
    return Logger.instance;
  }
}
// ⚠️ Prefer DI over Singleton in most cases!
```

### Structural

```typescript
// Adapter Pattern
// Đánh interface khác nhau thành 1 interface chung
interface PaymentGateway {
  charge(amount: number): Promise<{ id: string }>;
}

class StripeAdapter implements PaymentGateway {
  constructor(private stripe: Stripe) {}
  async charge(amount: number) {
    const intent = await this.stripe.paymentIntents.create({
      amount: amount * 100, // Stripe uses cents!
      currency: "usd",
    });
    return { id: intent.id };
  }
}

class PayPalAdapter implements PaymentGateway {
  async charge(amount: number) {
    const order = await paypal.createOrder({ amount });
    return { id: order.id };
  }
}
// Swap payment gateways without changing business logic!

// Decorator Pattern
// Thêm behavior mà không modify existing code
function LogExecutionTime(
  target: any,
  key: string,
  descriptor: PropertyDescriptor,
) {
  const original = descriptor.value;
  descriptor.value = async function (...args: any[]) {
    const start = Date.now();
    const result = await original.apply(this, args);
    console.log(`${key} took ${Date.now() - start}ms`);
    return result;
  };
}

class UserService {
  @LogExecutionTime // decorator!
  async findUser(id: number) {
    /* ... */
  }
}
```

### Behavioral

```typescript
// Strategy Pattern
// Chọn algorithm tại runtime
interface CompressionStrategy {
  compress(data: Buffer): Buffer;
}
class GzipStrategy implements CompressionStrategy {
  /* ... */
}
class BrotliStrategy implements CompressionStrategy {
  /* ... */
}

class FileProcessor {
  constructor(private strategy: CompressionStrategy) {}
  process(data: Buffer) {
    return this.strategy.compress(data);
  }
}
// Runtime: new FileProcessor(new GzipStrategy())

// Observer Pattern (Event Emitter)
class OrderEventEmitter extends EventEmitter {}
const orderEvents = new OrderEventEmitter();

orderEvents.on("orderCreated", (order) => emailService.send(order));
orderEvents.on("orderCreated", (order) => analyticsService.track(order));
orderEvents.on("orderCreated", (order) => inventoryService.reserve(order));

// Emit → all listeners execute (loose coupling!)
orderEvents.emit("orderCreated", newOrder);

// Repository Pattern
interface UserRepository {
  findById(id: number): Promise<User | null>;
  save(user: User): Promise<User>;
  delete(id: number): Promise<void>;
}

class PostgresUserRepository implements UserRepository {
  async findById(id: number) {
    return this.db.query("SELECT * FROM users WHERE id = $1", [id]);
  }
}
// Swap DB: just implement another repository!
```

## 3. TypeScript Advanced

```typescript
// Utility Types
type Partial<T> = { [K in keyof T]?: T[K] }; // all optional
type Required<T> = { [K in keyof T]-?: T[K] }; // all required
type Pick<T, K> = { [P in K]: T[P] }; // subset of fields
type Omit<T, K> = Pick<T, Exclude<keyof T, K>>; // exclude fields
type Record<K, V> = { [P in K]: V }; // key-value map
type Readonly<T> = { readonly [K in keyof T]: T[K] };

// Generics
function findById<T extends { id: number }>(
  items: T[],
  id: number,
): T | undefined {
  return items.find((item) => item.id === id);
}

// Conditional Types
type IsString<T> = T extends string ? "yes" : "no";
type A = IsString<string>; // "yes"
type B = IsString<number>; // "no"

// Mapped Types
type CreateDTO<T> = Omit<T, "id" | "createdAt" | "updatedAt">;
type UpdateDTO<T> = Partial<CreateDTO<T>>;

interface User {
  id: number;
  name: string;
  email: string;
  createdAt: Date;
}
type CreateUserDTO = CreateDTO<User>; // { name: string; email: string }
type UpdateUserDTO = UpdateDTO<User>; // { name?: string; email?: string }

// Discriminated Unions
type Result<T> = { success: true; data: T } | { success: false; error: string };

function handleResult(result: Result<User>) {
  if (result.success) {
    console.log(result.data.name); // TypeScript knows data exists!
  } else {
    console.log(result.error); // TypeScript knows error exists!
  }
}

// strict mode: tsconfig.json
// "strict": true enables:
//   strictNullChecks, noImplicitAny, strictFunctionTypes, etc.
// → ALWAYS enable in production projects!
```

## 4. Clean Code

```
Meaningful Names:
  ❌ const d = 86400;
  ✅ const SECONDS_IN_A_DAY = 86400;
  ❌ function calc(a, b) {}
  ✅ function calculateDiscount(price, discountPercent) {}

Small Functions:
  ❌ 1 function 200 lines doing everything
  ✅ Each function does ONE thing, <20 lines ideally
  ✅ If you need a comment to explain a block → extract to function

DRY vs WET:
  DRY (Don't Repeat Yourself): extract common logic
  WET (Write Everything Twice): acceptable for 2 occurrences
  ⚠️ Wrong abstraction WORSE than duplication!
  → Duplicate until pattern is CLEAR, then abstract

YAGNI (You Ain't Gonna Need It):
  Don't build features "just in case"
  Build what you need NOW

KISS (Keep It Simple, Stupid):
  ❌ over-engineered AbstractFactoryStrategyBuilderProxy
  ✅ simple function that solves the problem
```

## 6. Code Review

```
Reviewer Checklist:
  ✅ Does it solve the problem? (PR description)
  ✅ Edge cases handled?
  ✅ Error handling adequate?
  ✅ Tests cover critical paths?
  ✅ No security issues? (injection, auth bypass)
  ✅ Performance: N+1 queries, unnecessary loops?
  ✅ Naming clear? Code readable without comments?

Giving Feedback:
  ❌ "This code is terrible"
  ✅ "Consider using Map here instead of array.find() for O(1) lookup"
  ❌ "Why did you do it this way?"
  ✅ "What was the thinking behind this approach? I'm wondering if X would be simpler"

  Nit: minor style preference (prefix with "nit:")
  Suggestion: improvement idea (non-blocking)
  Request: must fix before merge (blocking)

Receiving Feedback:
  ✅ Don't take it personally — code ≠ you
  ✅ Ask for clarification if unclear
  ✅ Explain your reasoning, then accept if reviewer's point is valid
  ✅ "I see your point, let me refactor this"
```

## 7. Câu Hỏi Phỏng Vấn

**Q: SOLID — giải thích Dependency Inversion?**

> A: High-level modules (business logic) KHÔNG depend on low-level modules (database, HTTP). Cả hai depend on **abstractions** (interfaces). OrderService depends on `Database` interface, NOT `PostgresDatabase` class. Benefits: swap implementations (test with mock, production with real), loose coupling, testability.

**Q: Singleton — tại sao controversial?**

> A: (1) Hidden dependencies — not in constructor, hard to trace. (2) Global state — hard to test (shared across tests). (3) Tight coupling — hard to swap. Prefer: DI container (NestJS manages instance lifecycle). Use Singleton only for: logger, config (stateless utilities).

**Q: DRY — khi nào KHÔNG nên DRY?**

> A: **Wrong abstraction WORSE than duplication.** Khi 2 pieces code GIỐNG nhưng sẽ DIVERGE → abstract sớm = coupling chặt, change 1 breaks the other. Rule: duplicate until pattern clear (3+ occurrences), then abstract. "Duplication is far cheaper than the wrong abstraction" — Sandi Metz.
