## 1. Testing Pyramid

```
                    ┌──────┐
                    │ E2E  │  ← Ít nhất, chậm nhất, đắt nhất
                    │ Tests│
                   ┌┴──────┴┐
                   │Integra-│
                   │ tion   │  ← Vừa phải
                  ┌┴────────┴┐
                  │  Unit    │  ← NHIỀU nhất, nhanh nhất, rẻ nhất
                  │  Tests   │
                  └──────────┘

  Unit:        70% — test từng function/class isolated
  Integration: 20% — test multiple components together (API + DB)
  E2E:         10% — test full user flow

  ⚠️ Anti-pattern: Ice Cream Cone (quá nhiều E2E, ít unit)
  → Chậm, flaky, khó debug
```

## 2. Unit Testing

### AAA Pattern & Jest

```typescript
// AAA = Arrange - Act - Assert

describe("OrderService", () => {
  describe("calculateTotal", () => {
    it("should apply discount for orders over $100", () => {
      // Arrange
      const items = [
        { name: "Laptop", price: 999, quantity: 1 },
        { name: "Mouse", price: 29, quantity: 2 },
      ];
      const service = new OrderService();

      // Act
      const total = service.calculateTotal(items, "SAVE10");

      // Assert
      expect(total).toBe(951.3); // (999 + 58) * 0.9
    });

    it("should throw for negative quantity", () => {
      const items = [{ name: "X", price: 10, quantity: -1 }];
      const service = new OrderService();

      expect(() => service.calculateTotal(items)).toThrow("Invalid quantity");
    });
  });
});
```

### Mocking

```typescript
// 1. jest.mock — mock entire module
jest.mock("../services/email.service");

// 2. Manual mock — fine-grained control
const mockEmailService = {
  sendEmail: jest.fn().mockResolvedValue({ success: true }),
};

// 3. Dependency Injection — BEST for testability
class OrderService {
  constructor(
    private emailService: EmailService, // inject dependency
    private paymentService: PaymentService,
  ) {}

  async createOrder(order: Order) {
    const payment = await this.paymentService.charge(order.total);
    await this.emailService.send(order.email, "Order confirmed");
    return { id: order.id, paymentId: payment.id };
  }
}

// Test: inject mocks
it("should create order and send email", async () => {
  const mockEmail = { send: jest.fn().mockResolvedValue(true) };
  const mockPayment = { charge: jest.fn().mockResolvedValue({ id: "pay_1" }) };
  const service = new OrderService(mockEmail, mockPayment);

  const result = await service.createOrder({
    id: 1,
    total: 100,
    email: "a@b.com",
  });

  expect(result.paymentId).toBe("pay_1");
  expect(mockEmail.send).toHaveBeenCalledWith("a@b.com", "Order confirmed");
  expect(mockPayment.charge).toHaveBeenCalledWith(100);
});
```

### Code Coverage

```
Coverage types:
  Line:      mỗi dòng code được execute?
  Branch:    cả if + else branches?
  Function:  mỗi function được gọi?
  Statement: mỗi statement?

  jest --coverage
  ┌──────────────────────────────────┐
  │ File          | Lines | Branch   │
  │ order.ts      | 95%   | 87%     │
  │ payment.ts    | 88%   | 72%     │  ← branch thấp = thiếu edge cases
  └──────────────────────────────────┘

  ⚠️ Coverage ≠ Quality!
  100% coverage nhưng không assert đúng thứ = useless
  80% coverage với meaningful assertions > 100% coverage meaningless
  → Target: 80% line coverage, focus on CRITICAL paths
```

## 3. Integration Testing

```typescript
// Supertest — test HTTP endpoints
import request from "supertest";
import { app } from "../app";

describe("POST /api/users", () => {
  it("should create user and return 201", async () => {
    const response = await request(app)
      .post("/api/users")
      .send({ name: "Alice", email: "alice@example.com" })
      .expect(201);

    expect(response.body).toMatchObject({
      id: expect.any(Number),
      name: "Alice",
      email: "alice@example.com",
    });
  });

  it("should return 422 for invalid email", async () => {
    await request(app)
      .post("/api/users")
      .send({ name: "Alice", email: "not-an-email" })
      .expect(422);
  });
});
```

```typescript
// Testcontainers — real DB in Docker for tests
import { PostgreSqlContainer } from "@testcontainers/postgresql";

let container;
let db;

beforeAll(async () => {
  container = await new PostgreSqlContainer().withDatabase("test").start();

  db = await createConnection(container.getConnectionUri());
  await db.runMigrations();
}, 30000); // 30s timeout for container start

afterAll(async () => {
  await db.close();
  await container.stop();
});

beforeEach(async () => {
  await db.query("DELETE FROM users"); // clean state per test
});

it("should persist user to database", async () => {
  await request(app).post("/api/users").send({ name: "Bob" }).expect(201);

  const users = await db.query("SELECT * FROM users");
  expect(users).toHaveLength(1);
  expect(users[0].name).toBe("Bob");
});
```

## 4. E2E Testing

```typescript
// Playwright — browser & API E2E testing
import { test, expect } from "@playwright/test";

test("user can login and see dashboard", async ({ page }) => {
  await page.goto("http://localhost:3000/login");
  await page.fill("#email", "admin@example.com");
  await page.fill("#password", "password123");
  await page.click("#login-button");

  await expect(page).toHaveURL("/dashboard");
  await expect(page.locator("h1")).toContainText("Welcome");
});

// API E2E: full flow testing
test("complete order flow", async ({ request }) => {
  // 1. Create user
  const user = await request.post("/api/users", {
    data: { name: "Alice", email: "alice@test.com" },
  });
  const userId = (await user.json()).id;

  // 2. Create order
  const order = await request.post("/api/orders", {
    data: { userId, items: [{ productId: 1, quantity: 2 }] },
  });
  expect(order.status()).toBe(201);

  // 3. Verify order status
  const orderId = (await order.json()).id;
  const status = await request.get(`/api/orders/${orderId}`);
  expect((await status.json()).status).toBe("pending");
});
```

## 5. Load Testing

```javascript
// k6 — load testing
import http from "k6/http";
import { check, sleep } from "k6";

export const options = {
  stages: [
    { duration: "1m", target: 50 }, // ramp up to 50 users
    { duration: "3m", target: 50 }, // hold at 50
    { duration: "1m", target: 200 }, // spike to 200
    { duration: "3m", target: 200 }, // hold at 200
    { duration: "1m", target: 0 }, // ramp down
  ],
  thresholds: {
    http_req_duration: ["p(95)<200"], // 95% requests < 200ms
    http_req_failed: ["rate<0.01"], // <1% error rate
  },
};

export default function () {
  const res = http.get("http://localhost:3000/api/products");
  check(res, {
    "status is 200": (r) => r.status === 200,
    "response time < 200ms": (r) => r.timings.duration < 200,
  });
  sleep(1);
}

// SLA definitions:
// p50 < 50ms    (median)
// p95 < 200ms   (tail)
// p99 < 500ms   (worst case)
// Error rate < 0.1%
// Throughput > 1000 RPS
```

## 6. Contract Testing

```
Problem: microservices test independently → integration breaks!
  User Service ← API → Order Service
  User Service changes response format → Order Service breaks!

Pact (Consumer-Driven Contract Testing):

  1. Consumer (Order) defines EXPECTED interactions:
     "When I call GET /users/42, I expect { id: 42, name: string }"

  2. Contract (Pact file) generated from consumer tests

  3. Provider (User) verifies it CAN fulfill the contract:
     "Can I respond to GET /users/42 with { id, name }? ✅"

  4. Both sides can deploy INDEPENDENTLY if contracts pass!

  ┌──────────┐     Contract     ┌──────────┐
  │ Consumer │─── (Pact file) ──│ Provider │
  │ (Order)  │    defines what  │ (User)   │
  │ Tests    │    is expected    │ Verifies │
  └──────────┘                  └──────────┘
```

## 7. Chaos Engineering

```
"Things WILL break. Better to break them intentionally and LEARN."

Principles (Netflix):
  1. Define steady state (baseline metrics)
  2. Hypothesize: "system should handle X failure"
  3. Introduce failure in PRODUCTION (controlled!)
  4. Observe: did steady state hold?
  5. Fix weaknesses found

Experiments:
  ✅ Kill random pods/instances (Chaos Monkey)
  ✅ Network latency injection (50ms → 500ms)
  ✅ Network partition (service A can't reach B)
  ✅ CPU/Memory exhaustion
  ✅ Database failover (kill primary)
  ✅ DNS failure

Game Days:
  Planned chaos exercises for the team
  → Everyone learns how system fails + how to recover
  → Run quarterly, document findings

Tools: Gremlin, Chaos Monkey, Litmus (K8s), AWS FIS
```

## 8. Câu Hỏi Phỏng Vấn

**Q: Unit test vs Integration test — khi nào dùng?**

> A: **Unit**: test business logic isolated, mock dependencies. Nhanh (1000 tests < 5s). Dùng cho: calculations, validations, transformations. **Integration**: test real interactions (API + DB + cache). Chậm hơn (testcontainers). Dùng cho: API endpoints, DB queries, external service calls. Cả hai CẦN — unit cho logic, integration cho glue.

**Q: Code coverage 100% — có nên target?**

> A: KHÔNG. Coverage đo lines executed, KHÔNG đo correctness. 100% coverage nhưng assert wrong things = false confidence. Target: 80% lines, nhưng focus **critical paths**: payment, authentication, business rules. Untested code ở utility/logging acceptable.
