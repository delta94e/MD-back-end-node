# §0.3.3 — UDP & So Sánh TCP vs UDP

> **Mục tiêu**: Hiểu UDP hoạt động ra sao, tại sao "không đáng tin cậy" nhưng vẫn cực kỳ quan trọng, và khi nào chọn UDP vs TCP.

---

## 1. UDP Là Gì

UDP (User Datagram Protocol) — giao thức **connectionless**, **unreliable**, **no ordering** — nhưng **nhanh** và **đơn giản**.

```
UDP Header (chỉ 8 bytes! so với TCP 20+ bytes):
┌──────────────────────────────────────────┐
│  Source Port (16-bit)  │ Dest Port (16)  │
├──────────────────────────────────────────┤
│  Length (16-bit)       │ Checksum (16)   │
└──────────────────────────────────────────┘
│                                          │
│              Data (Payload)              │
│                                          │
└──────────────────────────────────────────┘

Không có: sequence number, ACK, window, handshake, retransmission
→ Gửi xong = quên ngay, không quan tâm đến đã được nhận chưa
```

---

## 2. TCP vs UDP — So Sánh Chi Tiết

| Tiêu chí               | TCP                                   | UDP                      |
| ---------------------- | ------------------------------------- | ------------------------ |
| **Connection**         | Connection-oriented (3-way handshake) | Connectionless           |
| **Reliability**        | Guaranteed delivery (retransmit)      | Best-effort (có thể mất) |
| **Ordering**           | Ordered (sequence numbers)            | Không đảm bảo thứ tự     |
| **Flow Control**       | Sliding window                        | Không có                 |
| **Congestion Control** | Slow Start, AIMD                      | Không có                 |
| **Header size**        | 20-60 bytes                           | 8 bytes                  |
| **Latency**            | Cao (handshake + ACK)                 | **Thấp**                 |
| **Overhead**           | Nhiều                                 | **Ít**                   |
| **Use cases**          | HTTP, HTTPS, SSH, Email, DB           | DNS, Video, Gaming, VoIP |

---

## 3. Khi Nào Dùng UDP?

```
✅ Dùng UDP khi:
  1. LATENCY quan trọng hơn RELIABILITY
     - Gaming: mất 1 frame → chấp nhận, delay 200ms → KHÔNG chấp nhận
     - Video call: mất vài pixel → OK, lag → không thể dùng

  2. Data có TIME-SENSITIVE nature
     - DNS: query nhanh, nếu mất → gửi lại (application layer)
     - DHCP: broadcast discover

  3. MULTICAST / BROADCAST cần thiết
     - UDP hỗ trợ multicast, TCP thì KHÔNG

  4. Application tự xử lý reliability
     - QUIC (HTTP/3): UDP + reliability ở application layer
     - Online games: custom reliability cho critical data

✅ Dùng TCP khi:
  1. Data PHẢI đến đầy đủ, đúng thứ tự
     - File transfer, email, HTTP API, database queries
  2. Không chấp nhận mất data
     - Financial transactions, chat messages
```

---

## 4. Use Cases Thực Tế

| Protocol           | Transport                                | Tại sao?                                               |
| ------------------ | ---------------------------------------- | ------------------------------------------------------ |
| DNS                | **UDP** (default), TCP (large responses) | Query nhỏ, cần nhanh, mất → retry                      |
| HTTP/1.1, 2        | TCP                                      | Data phải reliable + ordered                           |
| **HTTP/3**         | **UDP** (QUIC)                           | QUIC thêm reliability trên UDP, tránh TCP HOL blocking |
| Video streaming    | UDP (RTP)                                | Mất frame OK, retransmit gây delay tệ hơn              |
| Online gaming      | UDP                                      | Latency-critical, old state không cần retransmit       |
| VoIP (Zoom, Teams) | UDP (RTP/SRTP)                           | Real-time audio, mất sample → chấp nhận                |
| IoT sensors        | UDP (CoAP)                               | Resource-constrained devices, simple protocol          |

## 5. QUIC — Tại Sao HTTP/3 Chọn UDP?

```
TCP Head-of-Line Blocking:
  TCP stream gồm packets: [1] [2] [3] [4] [5]
  Packet [2] mất → TCP PHẢI chờ retransmit [2]
  → Packets [3][4][5] dù đã nhận cũng phải CHỜ!
  → Tất cả HTTP/2 streams trên cùng TCP connection bị block!

QUIC giải quyết:
  - Chạy trên UDP (không bị TCP HOL blocking)
  - Tự implement reliability PER STREAM
  - Stream A mất packet → CHỈ stream A chờ, stream B/C tiếp tục!
  - 0-RTT connection (đã kết nối trước đó → gửi data ngay)
  - Tích hợp TLS 1.3 → ít handshake hơn
```

---

## 6. Câu Hỏi Phỏng Vấn

**Q: TCP vs UDP — khi nào dùng cái nào?**

> A: **TCP** khi cần reliability + ordering (HTTP API, database, file transfer). **UDP** khi cần low latency + chấp nhận mất data (real-time video/gaming, DNS queries). Trend hiện tại: QUIC (HTTP/3) dùng UDP + implement reliability ở application layer — lấy tốt nhất của cả hai.

**Q: HTTP/3 dùng UDP mà vẫn reliable — tại sao không dùng TCP luôn?**

> A: TCP Head-of-Line blocking: 1 packet mất → toàn bộ TCP connection bị block. QUIC trên UDP tự implement reliability **per stream** → 1 stream mất packet không ảnh hưởng streams khác. Thêm nữa: QUIC tích hợp TLS 1.3 nên giảm handshake rounds, và hỗ trợ connection migration (chuyển network mà không cần reconnect).
