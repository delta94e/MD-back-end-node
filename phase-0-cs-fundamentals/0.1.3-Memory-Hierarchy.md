## 1. Tổng Quan Phân Cấp Bộ Nhớ

```
            Tốc độ ▲        Chi phí/GB ▲        Dung lượng
                   │                   │
  Registers ──────►│  ~0.3ns          │  $$$$$$$     │  ~KB
                   │                   │              │
  L1 Cache ───────►│  ~1ns            │  $$$$$$      │  32-64KB
                   │                   │              │
  L2 Cache ───────►│  ~4ns            │  $$$$$       │  256KB-1MB
                   │                   │              │
  L3 Cache ───────►│  ~12ns           │  $$$$        │  8-64MB
                   │                   │              │
  RAM ────────────►│  ~100ns          │  $$$         │  8-128GB
                   │                   │              │
  SSD ────────────►│  ~100μs          │  $$          │  256GB-4TB
                   │                   │              │
  HDD ────────────►│  ~10ms           │  $           │  1-20TB
                   │                   │              ▼
                   │                   │         Lớn nhất
              Nhanh nhất          Đắt nhất
```

**Nguyên lý cốt lõi**: Không thể có bộ nhớ vừa nhanh, vừa lớn, vừa rẻ. Nên hệ thống phân tầng: **data "hot" nằm ở tầng nhanh, data "cold" nằm ở tầng chậm**.

### So sánh con số thực tế

Nếu 1 CPU cycle = 1 giây (quy đổi thời gian cho dễ hình dung):

| Thao tác                  | Thời gian thực | Quy đổi tương đương |
| ------------------------- | -------------- | ------------------- |
| L1 Cache hit              | ~1 ns          | **1 giây**          |
| L2 Cache hit              | ~4 ns          | 4 giây              |
| L3 Cache hit              | ~12 ns         | 12 giây             |
| RAM access                | ~100 ns        | **1.5 phút**        |
| SSD random read           | ~100 μs        | **1.5 ngày**        |
| HDD seek                  | ~10 ms         | **5 tháng**         |
| Network (same DC)         | ~500 μs        | **1 tuần**          |
| Network (cross-continent) | ~150 ms        | **4.5 năm**         |

> Khi bạn đọc 1 byte từ disk thay vì cache, tương đương với việc chờ **5 tháng** thay vì **1 giây**. Đây là lý do caching quan trọng đến vậy.

## 2. RAM — Random Access Memory

### 2.1 RAM là gì?

RAM là bộ nhớ chính (main memory) — nơi chương trình đang chạy và data đang xử lý được lưu trữ.

**Đặc điểm**:

- **Random Access**: Truy cập bất kỳ vị trí nào đều mất cùng thời gian (~100ns)
- **Volatile**: Mất dữ liệu khi tắt nguồn (khác SSD/HDD)
- **Byte-addressable**: Mỗi byte có một địa chỉ duy nhất

### 2.2 RAM vs ROM

| Đặc điểm | RAM                             | ROM                            |
| -------- | ------------------------------- | ------------------------------ |
| Đọc/Ghi  | Cả hai                          | Chỉ đọc (hoặc ghi rất chậm)    |
| Volatile | Có (mất khi tắt nguồn)          | Không                          |
| Tốc độ   | Nhanh                           | Chậm hơn                       |
| Vai trò  | Lưu data chương trình đang chạy | Lưu firmware, BIOS, bootloader |

### 2.3 DRAM vs SRAM

```
SRAM (Static RAM):
  - Dùng cho CPU Cache (L1, L2, L3)
  - 6 transistors / bit → đắt, nhưng cực nhanh
  - Không cần refresh

DRAM (Dynamic RAM):
  - Dùng cho Main Memory (RAM stick)
  - 1 transistor + 1 capacitor / bit → rẻ, lớn, nhưng chậm hơn
  - Cần refresh mỗi ~64ms (capacitor leak charge)
```

## 3. Virtual Memory — Bộ Nhớ Ảo

### 3.1 Vấn đề: Tại sao cần Virtual Memory?

**Không có Virtual Memory** (thời kỳ đầu):

- Mỗi chương trình truy cập **physical memory** trực tiếp
- Chương trình A có thể vô tình ghi đè memory của chương trình B → **crash**
- Nếu RAM 4GB nhưng chương trình cần 8GB → **không chạy được**
- Mỗi chương trình phải biết chính xác nó được load ở địa chỉ nào → **phức tạp**

**Với Virtual Memory** (giải pháp):

- Mỗi process có **không gian địa chỉ ảo** riêng (thường 4GB trên 32-bit, 128TB+ trên 64-bit)
- Process nghĩ nó "sở hữu" toàn bộ memory → đơn giản hóa lập trình
- OS ánh xạ (map) virtual addresses → physical addresses
- Hai process CÓ THỂ dùng cùng virtual address nhưng trỏ đến physical address khác nhau → **isolation**

### 3.2 Cách Virtual Memory hoạt động

```
              Process A                    Physical RAM            Process B
          (Virtual Memory)                                    (Virtual Memory)
       ┌──────────────────┐           ┌──────────────────┐  ┌──────────────────┐
       │ 0x0000: Code     │───────┐   │ Frame 0: A code  │  │ 0x0000: Code     │
       │ 0x1000: Data     │──┐    └──▶│ Frame 1: B data  │◄─│ 0x1000: Data     │
       │ 0x2000: Heap     │  │    ┌──▶│ Frame 2: A data  │  │ 0x2000: Heap     │
       │ 0x3000: (unused) │  └────┘   │ Frame 3: A heap  │  │ 0x3000: Stack    │
       │ ...              │       ┌──▶│ Frame 4: B code  │◄─│                  │
       │ 0xFFFF: Stack    │───┐   │   │ Frame 5: B heap  │  │ 0xFFFF: Stack    │
       └──────────────────┘   │   │   │ Frame 6: (free)  │  └──────────────────┘
                              │   │   │ Frame 7: A stack │
                              └───┘   │ Frame 8: B stack │
                                      └──────────────────┘

    Cùng virtual address 0x1000   → khác physical address!
    A's 0x1000 → Frame 2          B's 0x1000 → Frame 1
```

**3 lợi ích chính**:

1. **Isolation**: Process A không thể đọc/ghi memory của Process B
2. **Abstraction**: Developer không cần quan tâm physical memory layout
3. **Overcommit**: Tổng virtual memory > physical RAM (dùng disk swap)

## 4. Paging — Phân Trang

### 4.1 Page là gì?

Virtual memory được chia thành **pages** (trang) cố định, thường **4KB** mỗi page. Physical memory được chia thành **frames** cùng kích thước. OS duy trì **Page Table** để ánh xạ page → frame.

```
                    Page Table (mỗi process có riêng)
Virtual Memory      ┌────────┬────────┬───────┐     Physical Memory
 ┌──────────┐       │ Page # │Frame # │ Valid │      ┌──────────┐
 │ Page 0   │──────▶│   0    │   5    │   1   │─────▶│ Frame 5  │
 │ Page 1   │──────▶│   1    │   2    │   1   │─────▶│ Frame 2  │
 │ Page 2   │──────▶│   2    │   -    │   0   │ (on disk!)
 │ Page 3   │──────▶│   3    │   8    │   1   │─────▶│ Frame 8  │
 │ Page 4   │──────▶│   4    │   1    │   1   │─────▶│ Frame 1  │
 └──────────┘       └────────┴────────┴───────┘      └──────────┘

 Valid = 1: page đang ở RAM
 Valid = 0: page ở disk (cần load vào RAM khi truy cập)
```

### 4.2 Address Translation (Chuyển đổi địa chỉ)

```
Virtual Address = Page Number + Offset

Ví dụ (page size = 4KB = 4096 bytes = 2^12):
  Virtual Address: 0x00003A7F

  Page Number = 0x00003A7F >> 12 = 0x3   (3 bits cao)
  Offset      = 0x00003A7F & 0xFFF = 0xA7F  (12 bits thấp)

  Tra page table: Page 3 → Frame 8
  Physical Address = Frame 8 * 4096 + 0xA7F = 0x8A7F
```

### 4.3 Multi-level Page Tables

Process 64-bit có 2⁶⁴ virtual addresses → single-level page table quá lớn. Giải pháp: **multi-level page table** (x86-64 dùng 4 levels).

```
Virtual Address (48-bit dùng trên x86-64):
┌──────┬──────┬──────┬──────┬────────────┐
│ PML4 │ PDP  │  PD  │  PT  │   Offset   │
│ 9bit │ 9bit │ 9bit │ 9bit │   12bit    │
└──┬───┴──┬───┴──┬───┴──┬───┴─────┬──────┘
   │      │      │      │         │
   ▼      ▼      ▼      ▼         ▼
  Level4→Level3→Level2→Level1→ Physical Frame + Offset
```

Lợi ích: Chỉ cần allocate page table entries cho vùng memory **thực sự dùng**, tiết kiệm rất nhiều RAM.

## 5. Page Faults — Lỗi Trang

### 5.1 Khi nào xảy ra?

Khi CPU truy cập virtual address mà page tương ứng **không nằm trong RAM** (Valid bit = 0).

### 5.2 Quy trình xử lý Page Fault

```
  CPU truy cập address
        │
        ▼
  Tra Page Table     ──── Valid=1 ────▶  Truy cập bình thường
        │
     Valid=0
        │
        ▼
  ┌─ PAGE FAULT! (trap to OS) ─┐
  │                              │
  │  1. OS tìm page trên disk   │
  │  2. Tìm free frame ở RAM    │
  │     (nếu hết → chọn victim  │
  │      page để evict/swap out) │
  │  3. Load page từ disk → RAM  │
  │  4. Update page table       │
  │     (Valid=1, Frame=#)       │
  │  5. Restart instruction      │
  └──────────────────────────────┘

  Chi phí: ~1-10ms (disk I/O) = CHẬM CỰC KỲ so với cache hit (1ns)
```

### 5.3 Page Replacement Algorithms

Khi RAM đầy, OS phải chọn page nào để "đuổi ra" (evict) nhường chỗ:

| Algorithm   | Cách hoạt động                                  | Nhận xét                                         |
| ----------- | ----------------------------------------------- | ------------------------------------------------ |
| **FIFO**    | Đuổi page vào sớm nhất                          | Đơn giản nhưng không hiệu quả (Bélády's anomaly) |
| **LRU**     | Đuổi page ít dùng gần đây nhất                  | Tốt nhưng tốn chi phí tracking                   |
| **Clock**   | Xấp xỉ LRU bằng reference bit + circular buffer | Linux dùng biến thể này                          |
| **Optimal** | Đuổi page sẽ không dùng lâu nhất                | Lý thuyết — không thể implement thực tế          |

> **Liên hệ**: LRU cũng được dùng trong **Redis eviction** và **cache invalidation** — cùng nguyên lý!

### 5.4 Thrashing

Khi quá nhiều processes cạnh tranh RAM → page fault liên tục → CPU **phần lớn thời gian chờ disk I/O** thay vì tính toán. Giải pháp: thêm RAM, giảm số processes, hoặc dùng working set model.

## 6. TLB — Translation Lookaside Buffer

### 6.1 Vấn đề

Mỗi memory access cần tra page table (nằm trong RAM) → thêm 1 lần RAM access → **chậm gấp đôi**.

### 6.2 TLB là gì?

TLB là **cache chuyên biệt** cho page table entries, nằm trong CPU. Thường chỉ 64-1024 entries nhưng hit rate >99%.

```
CPU cần address translation:

  Virtual Address
        │
        ▼
  ┌───────────┐
  │    TLB    │──── TLB HIT ────▶ Physical Address (nhanh, ~1ns)
  │  (cache)  │                    Không cần truy cập page table!
  └─────┬─────┘
        │
     TLB MISS
        │
        ▼
  ┌───────────┐
  │Page Table │──── Tìm thấy ──▶ Physical Address + Update TLB
  │ (in RAM)  │                   Lần sau sẽ TLB hit
  └─────┬─────┘
        │
     Page Fault
        │
        ▼
  Load from Disk (rất chậm)
```

### 6.3 TLB và Context Switch

Khi OS chuyển từ Process A sang Process B → **TLB phải FLUSH** (xóa sạch) vì page table khác nhau. Đây là một lý do context switch tốn kém.

> Một số CPU hiện đại có **ASID (Address Space ID)** trong TLB entry, cho phép giữ entries của nhiều processes → giảm chi phí context switch.

## 7. Stack vs Heap Memory

```
Process Memory Layout (Virtual Address Space):

  High Address  ┌──────────────────────┐
                │       Stack          │  ▼ grows downward
                │ (function vars,      │  Tự động quản lý (LIFO)
                │  return addresses)   │  Giới hạn (~1-8MB)
                ├──────────────────────┤
                │         ↓            │
                │    (free space)      │
                │         ↑            │
                ├──────────────────────┤
                │       Heap           │  ▲ grows upward
                │ (dynamic allocation) │  Manual/GC quản lý
                │ (malloc, new, etc.)  │  Lớn (~GB)
                ├──────────────────────┤
                │       BSS            │  Uninitialized globals
                ├──────────────────────┤
                │       Data           │  Initialized globals
                ├──────────────────────┤
                │       Text (Code)    │  Machine instructions
  Low Address   └──────────────────────┘
```

| Đặc điểm       | Stack                               | Heap                              |
| -------------- | ----------------------------------- | --------------------------------- |
| **Quản lý**    | Tự động (compiler)                  | Manual (`malloc/free`) hoặc GC    |
| **Tốc độ**     | Rất nhanh (chỉ di chuyển SP)        | Chậm hơn (fragmentation, GC)      |
| **Kích thước** | Nhỏ (1-8MB)                         | Lớn (GB)                          |
| **Lifetime**   | Theo function scope                 | Tùy ý (đến khi free/GC)           |
| **Lưu gì**     | Biến local, tham số, return address | Objects, dynamic arrays, closures |
| **Thread**     | Mỗi thread có stack riêng           | Shared giữa tất cả threads        |

## 8. Liên Hệ Với Node.js

### 8.1 V8 Memory Structure

```javascript
// V8 chia Heap thành các vùng:
//
// ┌─────────────────────────────────────────────┐
// │  New Space (Young Generation)  ~1-8MB       │  ← new objects
// │  ├── Semi-space From                        │
// │  └── Semi-space To                          │  Scavenger GC (fast, frequent)
// ├─────────────────────────────────────────────┤
// │  Old Space (Old Generation)    ~hundreds MB │  ← survived objects
// │                                             │  Mark-Sweep-Compact GC
// ├─────────────────────────────────────────────┤
// │  Large Object Space                         │  ← objects > 256KB
// ├─────────────────────────────────────────────┤
// │  Code Space                                 │  ← JIT compiled code
// └─────────────────────────────────────────────┘
```

### 8.2 Memory Limits & Monitoring

```javascript
// V8 default heap limit:
//   64-bit: ~1.5GB (có thể tăng bằng --max-old-space-size)
//   node --max-old-space-size=4096 app.js  (set 4GB)

// Kiểm tra memory usage:
const mem = process.memoryUsage();
console.log({
  rss: `${(mem.rss / 1024 / 1024).toFixed(1)} MB`, // Resident Set Size
  heapTotal: `${(mem.heapTotal / 1024 / 1024).toFixed(1)} MB`, // Heap allocated
  heapUsed: `${(mem.heapUsed / 1024 / 1024).toFixed(1)} MB`, // Heap actually used
  external: `${(mem.external / 1024 / 1024).toFixed(1)} MB`, // C++ objects (Buffers)
});
// rss: tổng physical memory process dùng (bao gồm code, stack, heap, shared libs)
// heapUsed < heapTotal (free space trong heap)
// external: Buffer, TypedArray (allocated bởi C++, ngoài V8 heap)
```

### 8.3 Memory Leak phổ biến trong Node.js

```javascript
// 1. Global array tích lũy
const cache = [];
app.get("/data", (req, res) => {
  const data = heavyComputation();
  cache.push(data); // ← LEAK: cache lớn dần, không bao giờ xóa
  res.json(data);
});

// 2. Closures giữ reference
function createHandler() {
  const hugeData = Buffer.alloc(100 * 1024 * 1024); // 100MB
  return (req, res) => {
    // hugeData bị closure "giữ" → GC không thể thu hồi
    res.json({ size: hugeData.length });
  };
}

// 3. EventEmitter listeners không remove
const emitter = new EventEmitter();
setInterval(() => {
  emitter.on("data", (d) => console.log(d)); // thêm listener mỗi interval!
}, 1000);
// → MaxListenersExceededWarning → memory leak
```

## 9. Câu Hỏi Phỏng Vấn

**Q: Giải thích Virtual Memory hoạt động ra sao.**

> A: Virtual Memory cho mỗi process một không gian địa chỉ ảo riêng biệt. OS dùng Page Table để ánh xạ virtual addresses → physical addresses. Khi process truy cập virtual address, MMU (Memory Management Unit) tra page table (thường qua TLB cache) để tìm physical frame tương ứng. Điều này cung cấp: (1) isolation giữa processes, (2) abstraction cho developer, (3) khả năng dùng nhiều memory hơn physical RAM thông qua swapping.

**Q: Page fault xảy ra khi nào? Ảnh hưởng performance thế nào?**

> A: Page fault xảy ra khi CPU truy cập virtual address mà page tương ứng không có trong RAM (valid bit = 0 trong page table). OS phải: (1) tìm page trên disk, (2) tìm free frame (hoặc evict page khác), (3) load page từ disk → RAM (~1-10ms), (4) update page table, (5) restart instruction. Chi phí: ~10⁷ ns so với normal access ~100ns → chậm hơn **100,000 lần**. Quá nhiều page faults → thrashing.

**Q: TLB miss ảnh hưởng performance thế nào?**

> A: TLB hit: address translation trong ~1ns. TLB miss: phải walk page table trong RAM → thêm 1-4 lần RAM access (multi-level page table) → ~100-400ns thay vì ~1ns. Với 4-level page table (x86-64), mỗi TLB miss có thể cần 4 RAM accesses. Context switch flush TLB → "cold start" cho process mới với nhiều TLB misses cho đến khi working set được cache lại.
