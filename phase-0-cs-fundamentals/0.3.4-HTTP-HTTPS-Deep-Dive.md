## 1. HTTP Request/Response Cycle

```
Client (Browser)                                    Server (Node.js)
     │                                                    │
     │  ── HTTP Request ──────────────────────────────▶   │
     │     GET /api/users HTTP/1.1                        │
     │     Host: api.example.com                          │
     │     Accept: application/json                       │
     │     Authorization: Bearer eyJhbGc...              │
     │                                                    │
     │                                                    │  Process request
     │                                                    │  Query database
     │                                                    │  Build response
     │                                                    │
     │  ◀──────────────────────────────── HTTP Response ──│
     │     HTTP/1.1 200 OK                                │
     │     Content-Type: application/json                 │
     │     Content-Length: 245                             │
     │     Cache-Control: max-age=3600                    │
     │                                                    │
     │     {"users": [...]}                               │
     │                                                    │
```

## 2. HTTP Methods

| Method      | Mục đích                       | Idempotent? | Safe? | Cacheable? |
| ----------- | ------------------------------ | :---------: | :---: | :--------: |
| **GET**     | Lấy resource                   |     ✅      |  ✅   |     ✅     |
| **POST**    | Tạo resource mới               |     ❌      |  ❌   |     ❌     |
| **PUT**     | Thay thế toàn bộ               |     ✅      |  ❌   |     ❌     |
| **PATCH**   | Cập nhật một phần              |     ❌      |  ❌   |     ❌     |
| **DELETE**  | Xóa resource                   |     ✅      |  ❌   |     ❌     |
| **HEAD**    | Như GET nhưng không body       |     ✅      |  ✅   |     ✅     |
| **OPTIONS** | Kiểm tra methods hỗ trợ (CORS) |     ✅      |  ✅   |     ❌     |

> **Idempotent**: Gọi 1 lần hay N lần → cùng kết quả. PUT /users/1 {name:"A"} gọi 10 lần → vẫn name="A". POST /users gọi 10 lần → tạo 10 users!

## 3. Status Codes

```
1xx: Informational    (hiếm gặp)
  100 Continue        — "Gửi body tiếp đi"
  101 Switching Protocols — WebSocket upgrade

2xx: Success
  200 OK              — Thành công
  201 Created         — Tạo resource mới thành công (POST)
  204 No Content      — Thành công, không có body (DELETE)

3xx: Redirection
  301 Moved Permanently — URL cũ → URL mới (vĩnh viễn, cached)
  302 Found           — Redirect tạm thời
  304 Not Modified    — Cache vẫn valid, dùng cached version

4xx: Client Error
  400 Bad Request     — Request body/params sai format
  401 Unauthorized    — Chưa authenticate (cần login)
  403 Forbidden       — Đã authenticate nhưng không có quyền
  404 Not Found       — Resource không tồn tại
  409 Conflict        — Conflict (ví dụ: duplicate email)
  422 Unprocessable   — Validation failed
  429 Too Many Requests — Rate limit exceeded

5xx: Server Error
  500 Internal Error  — Server bug
  502 Bad Gateway     — Reverse proxy không nhận được response
  503 Service Unavailable — Server quá tải / maintenance
  504 Gateway Timeout — Upstream server không phản hồi kịp
```

## 4. Headers Quan Trọng

```
REQUEST Headers:
  Host: api.example.com          ← Bắt buộc (HTTP/1.1)
  Accept: application/json       ← Client muốn nhận format gì
  Content-Type: application/json ← Body gửi format gì
  Authorization: Bearer token    ← Authentication
  User-Agent: Mozilla/5.0...     ← Client info
  Accept-Encoding: gzip, br      ← Compression client hỗ trợ
  If-None-Match: "etag123"       ← Conditional request (caching)
  Cookie: session=abc123         ← Cookies

RESPONSE Headers:
  Content-Type: application/json ← Body format
  Content-Length: 1234           ← Body size (bytes)
  Cache-Control: max-age=3600   ← Cache 1 giờ
  ETag: "etag456"               ← Version identifier
  Set-Cookie: session=xyz       ← Set cookie
  Access-Control-Allow-Origin: * ← CORS
  X-RateLimit-Remaining: 95     ← Custom: rate limit info
```

### Caching Flow

```
Request 1: GET /api/data
Response: 200 OK, ETag: "abc123", Cache-Control: max-age=60

Trong 60 giây: Browser dùng cached version (KHÔNG gọi server)

Sau 60 giây:
Request 2: GET /api/data, If-None-Match: "abc123"
  Nếu data không đổi: 304 Not Modified (NO body → tiết kiệm bandwidth!)
  Nếu data đã đổi:    200 OK, ETag: "def456", body mới
```

## 5. Cookies & Sessions

```
Cookie Flow:
  1. Client login: POST /login {email, password}
  2. Server verify → Set-Cookie: sid=abc123; HttpOnly; Secure; SameSite=Strict; Path=/
  3. Browser tự động gửi Cookie header trong MỌI request tiếp theo đến domain đó
  4. Server đọc cookie → lookup session → xác định user

Cookie Attributes:
  HttpOnly — JS không đọc được (chống XSS)
  Secure — chỉ gửi qua HTTPS
  SameSite=Strict — chống CSRF (không gửi cross-site)
  Max-Age=3600 — cookie hết hạn sau 1 giờ
  Path=/ — áp dụng cho mọi path
  Domain=.example.com — áp dụng cho subdomains
```

## 6. HTTP/1.1 vs HTTP/2 vs HTTP/3

### HTTP/1.1 — Vấn đề

```
HTTP/1.1:
  1 connection = 1 request at a time (sequential)

  Client ──▶ [GET /style.css]  ──▶ Server
  Client ◀── [Response]        ◀── Server
  Client ──▶ [GET /script.js]  ──▶ Server    ← PHẢI CHỜ response trước!
  Client ◀── [Response]        ◀── Server
  Client ──▶ [GET /image.png]  ──▶ Server
  Client ◀── [Response]        ◀── Server

  Total: 3 round trips! (HEAD-OF-LINE BLOCKING)

  Workaround: mở 6 TCP connections song song → 6× handshake cost
```

### HTTP/2 — Multiplexing

```
HTTP/2:
  1 TCP connection = nhiều streams song song!

  Client ══╡ Stream 1: GET /style.css  ╞══▶
           │ Stream 2: GET /script.js  │     Server
  Client ◀═╡ Stream 1: Response CSS   ╞═══
           │ Stream 3: GET /image.png  │
  Client ◀═╡ Stream 2: Response JS    ╞═══
  Client ◀═╡ Stream 3: Response Image ╞═══

  Tất cả trên 1 TCP connection!

Tính năng HTTP/2:
  ✅ Multiplexing — nhiều request/response song song
  ✅ Header Compression (HPACK) — giảm header overhead
  ✅ Server Push — server gửi resource trước khi client request
  ✅ Stream Priority — ưu tiên resources quan trọng
  ✅ Binary Protocol — nhanh hơn text (HTTP/1.1)

Vấn đề còn lại:
  ❌ TCP Head-of-Line Blocking — 1 TCP packet mất → TẤT CẢ streams chờ!
     (Vấn đề ở transport layer, không phải application layer)
```

### HTTP/3 (QUIC) — Giải quyết TCP HOL

```
HTTP/3:
  Dùng QUIC (trên UDP) thay vì TCP

  Ưu điểm:
  ✅ Per-stream reliability — stream A mất packet → CHỈ A chờ
  ✅ 0-RTT handshake (reconnection) — gửi data ngay!
  ✅ Connection migration — chuyển Wi-Fi → 4G không cần reconnect
  ✅ Integrated TLS 1.3 — ít round trips hơn
```

### So sánh

| Feature            | HTTP/1.1                      | HTTP/2                       | HTTP/3         |
| ------------------ | ----------------------------- | ---------------------------- | -------------- |
| Transport          | TCP                           | TCP                          | **QUIC (UDP)** |
| Multiplexing       | ❌ (6 connections workaround) | ✅                           | ✅             |
| HOL Blocking       | Application + TCP             | ✅ Fixed Application, ❌ TCP | ✅ Fixed both  |
| Header Compression | ❌                            | ✅ HPACK                     | ✅ QPACK       |
| Server Push        | ❌                            | ✅                           | ✅             |
| Handshake          | TCP (1.5 RTT) + TLS (1-2 RTT) | TCP + TLS                    | **0-1 RTT**    |

## 7. Liên Hệ Node.js

```javascript
// Node.js HTTP/2 server
const http2 = require("http2");
const fs = require("fs");

const server = http2.createSecureServer({
  key: fs.readFileSync("server.key"),
  cert: fs.readFileSync("server.crt"),
});

server.on("stream", (stream, headers) => {
  const path = headers[":path"];
  stream.respond({ ":status": 200, "content-type": "application/json" });
  stream.end(JSON.stringify({ message: "HTTP/2 response!" }));

  // Server Push example
  if (path === "/index.html") {
    const push = stream.pushStream(
      { ":path": "/style.css" },
      (err, pushStream) => {
        pushStream.respond({ ":status": 200, "content-type": "text/css" });
        pushStream.end("body { color: red; }");
      },
    );
  }
});

server.listen(8443);
```

## 8. Câu Hỏi Phỏng Vấn

**Q: HTTP/2 multiplexing giải quyết vấn đề gì của HTTP/1.1?**

> A: HTTP/1.1 chỉ cho 1 request/response tại 1 thời điểm trên 1 TCP connection (Head-of-Line blocking ở **Application layer**). Browser workaround bằng 6 parallel connections → 6× TCP handshake overhead. HTTP/2 multiplexing cho phép nhiều streams trên 1 TCP connection → giảm handshake, tăng throughput. Tuy nhiên, HOL blocking ở **TCP layer** vẫn còn — 1 TCP packet mất → tất cả streams chờ.

**Q: PUT vs PATCH khác gì?**

> A: **PUT**: thay thế **toàn bộ** resource. Nếu gửi PUT /users/1 {name: "A"} → tất cả fields khác bị xóa/reset. **PATCH**: cập nhật **một phần**. PATCH /users/1 {name: "A"} → chỉ name đổi. PUT là idempotent (gọi N lần = 1 lần). PATCH lý thuyết không idempotent nhưng thực tế thường implement idempotent.

**Q: Giải thích 401 vs 403?**

> A: **401 Unauthorized** (sai tên — nên là Unauthenticated): client chưa authenticate (chưa login, token hết hạn). Client nên thử lại với credentials. **403 Forbidden**: client ĐÃ authenticate nhưng KHÔNG có quyền truy cập resource. Thử lại cũng không giúp gì.
