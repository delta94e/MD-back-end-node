## 1. Vấn Đề: HTTP Không "Real-Time"

```
HTTP = Request → Response (Client luôn phải HỎI trước)

  Client: "Có tin nhắn mới không?"  →  Server: "Không"
  Client: "Có tin nhắn mới không?"  →  Server: "Không"
  Client: "Có tin nhắn mới không?"  →  Server: "Có! Đây là message"
  Client: "Có tin nhắn mới không?"  →  Server: "Không"

  Polling mỗi 1 giây = 3600 requests/giờ × 10,000 users = 36 TRIỆU requests/giờ!
  Hầu hết responses là "Không" → LÃ PHÍ bandwidth + server resources
```

## 2. Long Polling

```
Client: "Có tin nhắn mới không?" ──▶ Server
                                      │ Giữ connection MỞ...
                                      │ Chờ đến khi có data...
                                      │ (timeout 30 giây nếu không có)
                                      │
                                      │ Data đến!
Client: ◀── "Đây là message mới!" ── │
Client: "Có tin nhắn mới không?" ──▶ │  (ngay lập tức gửi request mới)

Ưu điểm: Đơn giản, compatible mọi browser
Nhược điểm: Vẫn overhead HTTP headers mỗi lần
            Connection churn (đóng mở liên tục)
            Server giữ nhiều pending connections
```

## 3. Server-Sent Events (SSE)

```
Client mở 1 HTTP connection (GET), Server liên tục PUSH data:

  Client: GET /events HTTP/1.1
  Accept: text/event-stream         ──▶ Server

  Server: HTTP/1.1 200 OK
  Content-Type: text/event-stream   ──▶ Connection giữ MỞ

  Server push: data: {"user": "A", "msg": "Hello"}\n\n
  Server push: data: {"user": "B", "msg": "Hi"}\n\n
  Server push: data: {"user": "A", "msg": "Bye"}\n\n
  ... (liên tục cho đến khi đóng)

Ưu điểm: Đơn giản, auto-reconnect, event IDs (resume)
Nhược điểm: UNIDIRECTIONAL (chỉ server → client)
            Text-only (không binary)
            Giới hạn connections (browser: 6/domain HTTP/1.1)
```

## 4. WebSocket

```
WebSocket = BIDIRECTIONAL, FULL-DUPLEX communication trên 1 TCP connection.

Bắt đầu bằng HTTP Upgrade handshake:

  Client: GET /chat HTTP/1.1
          Upgrade: websocket
          Connection: Upgrade
          Sec-WebSocket-Key: dGhlIHNhbX...    ──▶ Server

  Server: HTTP/1.1 101 Switching Protocols
          Upgrade: websocket
          Connection: Upgrade
          Sec-WebSocket-Accept: s3pPLM...     ──▶ Client

  ═════ WebSocket Connection ESTABLISHED ═══════

  Client → Server: {"type":"message","text":"Hello"}
  Server → Client: {"type":"message","text":"Hi back!"}
  Client → Server: {"type":"typing","user":"Alice"}
  Server → Client: {"type":"typing","user":"Alice"}
  Server → Client: {"type":"notification","count":5}

  Cả hai bên gửi BẤT KỲ lúc nào, không cần chờ!
```

### WebSocket Features

```
✅ Full-duplex: gửi/nhận cùng lúc
✅ Low latency: chỉ 2-byte header (vs HTTP ~100+ bytes)
✅ Binary support: text + binary data
✅ No HTTP overhead per message
✅ Sub-protocols: cho phép define protocol riêng

Production concerns:
  - Heartbeat/Ping-Pong: detect dead connections
  - Reconnection logic: auto-reconnect khi mất kết nối
  - Scaling: sticky sessions hoặc Redis pub/sub
  - Load balancing: cần L7 load balancer hiểu WebSocket
```

## 5. So Sánh 3 Phương Pháp

|                      | Long Polling         | SSE                    | WebSocket                |
| -------------------- | -------------------- | ---------------------- | ------------------------ |
| **Direction**        | Client → Server      | Server → Client        | **Bidirectional**        |
| **Protocol**         | HTTP                 | HTTP                   | WS (after HTTP upgrade)  |
| **Latency**          | Medium               | Low                    | **Lowest**               |
| **Overhead per msg** | Full HTTP headers    | ~10 bytes              | **~2 bytes**             |
| **Binary**           | ✅                   | ❌                     | ✅                       |
| **Auto-reconnect**   | Manual               | ✅ Built-in            | Manual                   |
| **Browser support**  | ✅ Mọi browser       | ✅ Tốt (no IE)         | ✅ Tốt                   |
| **Complexity**       | Thấp                 | Thấp                   | **Trung bình**           |
| **Best for**         | Simple notifications | Live feeds, dashboards | **Chat, gaming, collab** |

### Khi nào dùng cái nào?

```
SSE: Server push only, text data, notifications, live scores
     → Đơn giản nhất, auto-reconnect

WebSocket: Cần bidirectional, binary, low latency
     → Chat, multiplayer games, collaborative editing, trading

Long Polling: Fallback khi SSE/WebSocket không available
     → Simple compatibility requirement
```

## 6. Node.js Implementation

```javascript
// ===== WebSocket Server (ws library) =====
const WebSocket = require("ws");
const wss = new WebSocket.Server({ port: 8080 });

wss.on("connection", (ws) => {
  console.log("Client connected");

  ws.on("message", (data) => {
    const msg = JSON.parse(data);
    // Broadcast to all clients
    wss.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(JSON.stringify(msg));
      }
    });
  });

  // Heartbeat
  ws.isAlive = true;
  ws.on("pong", () => {
    ws.isAlive = true;
  });
});

// Heartbeat interval — detect dead connections
setInterval(() => {
  wss.clients.forEach((ws) => {
    if (!ws.isAlive) return ws.terminate();
    ws.isAlive = false;
    ws.ping();
  });
}, 30000);

// ===== SSE Server =====
const http = require("http");
http
  .createServer((req, res) => {
    if (req.url === "/events") {
      res.writeHead(200, {
        "Content-Type": "text/event-stream",
        "Cache-Control": "no-cache",
        Connection: "keep-alive",
      });

      const interval = setInterval(() => {
        res.write(`data: ${JSON.stringify({ time: Date.now() })}\n\n`);
      }, 1000);

      req.on("close", () => clearInterval(interval));
    }
  })
  .listen(3000);
```

## 7. Câu Hỏi Phỏng Vấn

**Q: WebSocket vs SSE vs Long Polling — trade-offs?**

> A: **WebSocket**: bidirectional, lowest latency, binary support → chat, gaming, real-time collaboration. Nhưng phức tạp hơn (heartbeat, reconnection, scaling cần sticky sessions). **SSE**: unidirectional (server → client), auto-reconnect, đơn giản → live feeds, notifications, dashboards. **Long Polling**: compatibility tốt nhất, nhưng overhead lớn per message. Chọn: SSE nếu chỉ cần server push, WebSocket nếu cần bidirectional.

**Q: WebSocket scale ra sao? 1 server handle bao nhiêu connections?**

> A: Mỗi WebSocket = 1 TCP connection ≈ 1 file descriptor + ~10KB memory. 1 server (tuned): 50,000-100,000+ connections. Scale: (1) Multiple servers + L7 load balancer (sticky sessions). (2) Redis pub/sub hoặc NATS để broadcast cross-server. (3) Socket.IO rooms/namespaces cho logical grouping.
