## 1. Tại Sao Cần Mô Hình Phân Tầng?

Tưởng tượng bạn gửi thư quốc tế:

- Bạn viết thư (nội dung) → **Application**
- Bỏ vào phong bì, dán tem → **Presentation/Session**
- Đưa cho bưu điện, ghi địa chỉ → **Transport/Network**
- Bưu điện chuyển qua xe tải, máy bay → **Data Link/Physical**

Mỗi tầng chỉ lo **nhiệm vụ của mình** và giao tiếp qua **giao diện chuẩn** với tầng kế tiếp. Thay đổi 1 tầng không ảnh hưởng tầng khác.

## 2. OSI Model — 7 Layers

```
Layer 7: APPLICATION      "Người dùng tương tác"
  │  HTTP, HTTPS, FTP, SMTP, DNS, WebSocket, gRPC
  │  → Cung cấp giao diện cho ứng dụng
  ▼
Layer 6: PRESENTATION     "Dịch thuật & mã hóa"
  │  SSL/TLS encryption, JPEG, JSON, XML, compression
  │  → Mã hóa, nén, chuyển đổi format
  ▼
Layer 5: SESSION           "Quản lý phiên"
  │  Session establishment, authentication, checkpointing
  │  → Mở/đóng/duy trì kết nối logic
  ▼
Layer 4: TRANSPORT         "Giao hàng tin cậy"
  │  TCP (reliable), UDP (fast), port numbers
  │  → End-to-end delivery, flow control, error recovery
  ▼
Layer 3: NETWORK           "Tìm đường đi"
  │  IP (IPv4/IPv6), ICMP, routing (OSPF, BGP)
  │  → Logical addressing (IP), routing giữa networks
  ▼
Layer 2: DATA LINK         "Giao tiếp cùng mạng"
  │  Ethernet, Wi-Fi (802.11), MAC address, ARP
  │  → Frame data, error detection (CRC), MAC addressing
  ▼
Layer 1: PHYSICAL          "Tín hiệu vật lý"
     Cables, fiber optics, radio waves, hubs
     → Bit transmission: 0 và 1 qua điện/quang/sóng
```

### Cách nhớ 7 layers

```
Từ dưới lên:
  Please    (Physical)
  Do        (Data Link)
  Not       (Network)
  Throw     (Transport)
  Sausage   (Session)
  Pizza     (Presentation)
  Away      (Application)
```

## 3. TCP/IP Model — 4 Layers

TCP/IP là mô hình **thực tế** mà Internet dùng. OSI là mô hình **lý thuyết** tham khảo.

```
┌──────────────────┬───────────────────────────────┐
│    OSI Model     │       TCP/IP Model            │
├──────────────────┼───────────────────────────────┤
│ 7. Application   │                               │
│ 6. Presentation  │   Application Layer           │
│ 5. Session       │   (HTTP, DNS, FTP, SMTP)      │
├──────────────────┼───────────────────────────────┤
│ 4. Transport     │   Transport Layer             │
│                  │   (TCP, UDP)                  │
├──────────────────┼───────────────────────────────┤
│ 3. Network       │   Internet Layer              │
│                  │   (IP, ICMP, ARP)             │
├──────────────────┼───────────────────────────────┤
│ 2. Data Link     │   Network Access Layer        │
│ 1. Physical      │   (Ethernet, Wi-Fi)           │
└──────────────────┴───────────────────────────────┘
```

> **Trong phỏng vấn**: Dùng OSI khi nói **lý thuyết** (giải thích layer nào làm gì). Dùng TCP/IP khi nói **thực tế** (debug network issues, explain packet flow).

## 4. Encapsulation & Decapsulation

Khi gửi data, mỗi layer **thêm header** (đóng gói). Khi nhận, mỗi layer **bóc header** (tháo gói).

```
GỬI (Sender) — Encapsulation:

  Application:    [HTTP Data.........................]
                           │ thêm TCP header
  Transport:      [TCP Hdr][HTTP Data.................]
                           │ thêm IP header
  Network:        [IP Hdr][TCP Hdr][HTTP Data.........]
                           │ thêm Ethernet header + trailer
  Data Link:      [Eth Hdr][IP Hdr][TCP Hdr][Data][Eth Trl]
                           │ chuyển thành bits
  Physical:       10110011010101001010101011101010...

NHẬN (Receiver) — Decapsulation: ngược lại
  Physical → Data Link (bóc Eth) → Network (bóc IP) → Transport (bóc TCP) → Application
```

## 5. PDU Ở Mỗi Layer

| Layer       | PDU Name                               | Chứa gì                           |
| ----------- | -------------------------------------- | --------------------------------- |
| Application | **Data / Message**                     | HTTP request/response, email      |
| Transport   | **Segment** (TCP) / **Datagram** (UDP) | Source/dest port, sequence number |
| Network     | **Packet**                             | Source/dest IP address, TTL       |
| Data Link   | **Frame**                              | Source/dest MAC address, CRC      |
| Physical    | **Bits**                               | 0 và 1                            |

```
Ví dụ cụ thể: GET /api/users HTTP/1.1

┌─────────────────────────────────────────────────────────────┐
│ Ethernet Frame                                              │
│ ┌──────────────────────────────────────────────────┐        │
│ │ IP Packet                                        │        │
│ │ ┌─────────────────────────────────────────────┐  │        │
│ │ │ TCP Segment                                 │  │        │
│ │ │ ┌───────────────────────────────────────┐   │  │        │
│ │ │ │ HTTP Data                             │   │  │        │
│ │ │ │ GET /api/users HTTP/1.1               │   │  │        │
│ │ │ │ Host: api.example.com                 │   │  │        │
│ │ │ └───────────────────────────────────────┘   │  │        │
│ │ │ Src Port: 52341  Dst Port: 443             │  │        │
│ │ │ Seq: 1000  Ack: 0  Flags: [PSH,ACK]       │  │        │
│ │ └─────────────────────────────────────────────┘  │        │
│ │ Src IP: 192.168.1.100    Dst IP: 93.184.216.34   │        │
│ │ TTL: 64   Protocol: TCP                          │        │
│ └──────────────────────────────────────────────────┘        │
│ Src MAC: AA:BB:CC:DD:EE:FF  Dst MAC: 11:22:33:44:55:66     │
│ CRC: 0xA1B2C3D4                                            │
└─────────────────────────────────────────────────────────────┘
```

## 6. Liên Hệ Với Backend Development

| Layer       | Backend Developer cần biết                                 |
| ----------- | ---------------------------------------------------------- |
| Application | HTTP methods, status codes, headers, WebSocket, gRPC       |
| Transport   | TCP vs UDP trade-offs, port numbers, connection pooling    |
| Network     | IP addressing, DNS resolution, load balancing, CDN routing |
| Data Link   | VLAN, ARP (debug network issues)                           |

```javascript
// Node.js — bạn thường làm việc ở Application layer
const http = require("http");
const server = http.createServer((req, res) => {
  // Application Layer: HTTP request/response
  // Transport Layer: TCP connection (handled by OS kernel)
  // Network Layer: IP routing (handled by OS + routers)
  res.writeHead(200, { "Content-Type": "application/json" });
  res.end(JSON.stringify({ users: [] }));
});
server.listen(3000); // Port 3000 = Transport Layer concept
```

## 7. Câu Hỏi Phỏng Vấn

**Q: OSI vs TCP/IP — khác gì? Dùng cái nào trong thực tế?**

> A: OSI = 7 layers, mô hình **tham khảo lý thuyết** (ISO standard). TCP/IP = 4 layers, mô hình **thực tế** Internet dùng. TCP/IP merge Application/Presentation/Session thành 1. Trong thực tế dùng TCP/IP, nhưng dùng thuật ngữ OSI khi nói về "layer nào" (ví dụ: "L4 load balancer" = Transport layer, "L7 proxy" = Application layer).

**Q: Khi bạn gõ URL vào browser, data đi qua các layers thế nào?**

> A: Browser tạo HTTP request (L7) → TCP segment hóa + thêm port (L4) → IP thêm địa chỉ nguồn/đích (L3) → Ethernet frame hóa + MAC (L2) → bits truyền qua cable/wifi (L1). Router nhận: bóc L2, đọc L3 (IP) quyết định đường đi, đóng gói L2 mới, gửi tiếp. Server nhận: bóc từ L1→L7, xử lý request, gửi response theo chiều ngược lại.
