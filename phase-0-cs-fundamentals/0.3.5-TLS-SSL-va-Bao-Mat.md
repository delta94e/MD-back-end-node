# §0.3.5 — TLS/SSL & Bảo Mật Giao Tiếp

> **Mục tiêu**: Hiểu TLS handshake, symmetric vs asymmetric encryption, certificate chain, và HSTS. Backend developer phải hiểu HTTPS hoạt động thế nào để cấu hình và debug đúng.

---

## 1. TLS Là Gì

TLS (Transport Layer Security) = lớp mã hóa giữa Application và Transport layer. SSL là phiên bản cũ (deprecated). "HTTPS" = HTTP + TLS.

```
Không có TLS:
  Client ─── "password123" ──────▶ Server
  Attacker (sniff): "password123" ← THẤY HẾT!

Có TLS:
  Client ─── "x8f2k9..." (encrypted) ──▶ Server
  Attacker: "x8f2k9..." ← KHÔNG ĐỌC ĐƯỢC!
```

---

## 2. Symmetric vs Asymmetric Encryption

```
SYMMETRIC (cùng key):
  Key: "my-secret-key"
  Encrypt: AES("hello", key) = "x8f2k9"
  Decrypt: AES("x8f2k9", key) = "hello"
  ✅ NHANH (100-1000x faster)
  ❌ Làm sao chia sẻ key an toàn?

ASYMMETRIC (2 keys: public + private):
  Public Key (ai cũng biết): encrypt data
  Private Key (chỉ server biết): decrypt data

  Client: RSA_Encrypt("hello", server_public_key) = "abc123"
  Server: RSA_Decrypt("abc123", server_private_key) = "hello"
  ✅ AN TOÀN chia sẻ key (public key là public)
  ❌ CHẬM (không dùng cho bulk data)

TLS kết hợp cả hai:
  1. Asymmetric: trao đổi "session key" an toàn
  2. Symmetric: dùng session key để encrypt data (nhanh!)
```

## 3. TLS 1.3 Handshake

```
  Client                                         Server
    │                                               │
    │  ── Client Hello ──────────────────────────▶  │
    │     Supported cipher suites                   │
    │     Client random                             │
    │     Key share (ECDHE public key)              │
    │                                               │
    │  ◀────────────────────────── Server Hello ──  │
    │     Chosen cipher suite                       │
    │     Server random                             │
    │     Key share (server ECDHE public key)       │
    │     Certificate + Signature                   │
    │     Finished (encrypted)                      │
    │                                               │
    │  ── Client Finished (encrypted) ───────────▶  │
    │                                               │
    │  ════ ENCRYPTED DATA TRANSFER ═══════════   │

TLS 1.3: chỉ 1 RTT! (TLS 1.2 cần 2 RTTs)
0-RTT resumption: nếu đã kết nối trước → gửi data ngay!
```

## 4. Certificate Chain

```
Root CA (Certificate Authority)
  │  Được trust bởi browser/OS (pre-installed)
  │  Ví dụ: DigiCert, Let's Encrypt, Comodo
  │
  ├── Intermediate CA
  │     │  Signed bởi Root CA
  │     │
  │     └── Server Certificate (api.example.com)
  │           Signed bởi Intermediate CA
  │           Chứa: domain name, public key, expiry date

Verification:
  1. Server gửi certificate
  2. Browser check: ai sign certificate này? → Intermediate CA
  3. Browser check: ai sign Intermediate? → Root CA
  4. Root CA có trong trust store? → YES → TRUST!
```

## 5. HSTS (HTTP Strict Transport Security)

```
Vấn đề: User gõ "example.com" → browser gửi HTTP trước
  → Attacker có thể intercept request HTTP đầu tiên!

HSTS header trong response:
  Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Nghĩa là:
  - Trong 1 năm (31536000s), LUÔN dùng HTTPS cho domain này
  - Áp dụng cho cả subdomains
  - preload: browser sẽ hardcode domain vào HSTS preload list
```

## 6. Câu Hỏi Phỏng Vấn

**Q: TLS handshake hoạt động thế nào (TLS 1.3)?**

> A: (1) Client Hello: gửi cipher suites + ECDHE public key. (2) Server Hello: chọn cipher, gửi ECDHE public key + certificate + signature. (3) Cả hai tính shared secret từ ECDHE → derive session keys. (4) Client Finished. Chỉ 1 RTT (TLS 1.2 cần 2). Data được encrypt bằng symmetric encryption (AES-GCM) với session key — nhanh hơn asymmetric.

**Q: Symmetric vs Asymmetric — tại sao TLS dùng cả hai?**

> A: Asymmetric (RSA/ECDHE) an toàn cho key exchange nhưng **100-1000x chậm hơn**. Symmetric (AES) nhanh nhưng cần shared secret. TLS dùng asymmetric để **trao đổi session key** an toàn, sau đó dùng symmetric cho **bulk data encryption**. Best of both worlds.
