## 1. Logarithms — Tại Sao log n Ở Khắp Nơi

### 1.1 Định nghĩa

Logarithm trả lời: **"cần nhân bao nhiêu lần?"**

```
log₂(8) = 3    vì    2³ = 8     → "chia đôi 8 bao nhiêu lần? 3 lần"
log₂(16) = 4   vì    2⁴ = 16
log₂(1024) = 10 vì   2¹⁰ = 1024
log₂(1,000,000) ≈ 20
log₂(1,000,000,000) ≈ 30

Trong CS: log luôn mặc định là log₂ (base 2)
→ log₂(n) = "số lần chia đôi n cho đến khi còn 1"
```

### 1.2 Tại sao Binary Search là O(log n)

```
Sorted array: [1, 3, 5, 7, 9, 11, 13, 15]   n = 8

Search(9):
  Step 1: [1, 3, 5, 7, | 9, 11, 13, 15]  mid=7, 9>7 → phải
  Step 2: [9, 11, | 13, 15]              mid=11, 9<11 → trái
  Step 3: [9]                             mid=9, FOUND!

  3 steps = log₂(8) = 3 ✓

Tổng quát: mỗi bước loại 1/2 → cần log₂(n) bước
  n = 10^6 → chỉ cần ~20 bước!
  n = 10^9 → chỉ cần ~30 bước!

Các thuật toán O(log n):
  Binary Search, BST search (balanced), Binary Heap insert
  → Tất cả chia đôi search space mỗi bước
```

### 1.3 Tại sao Tree height là O(log n)

```
Perfect Binary Tree:
  Level 0:  1 node
  Level 1:  2 nodes
  Level 2:  4 nodes
  Level 3:  8 nodes
  ...
  Level h:  2^h nodes

  Total nodes: 1 + 2 + 4 + ... + 2^h = 2^(h+1) - 1

  Nếu total = n:
    n ≈ 2^h → h = log₂(n)

  BST balanced → height = log₂(n) → search = O(log n) ✓
```

### 1.4 Log Properties (cần nhớ)

```
log(a × b) = log(a) + log(b)
log(a / b) = log(a) - log(b)
log(a^k)   = k × log(a)

Change of base:
  log_a(n) = log_b(n) / log_b(a)
  → log₂(n) = ln(n) / ln(2) ≈ 1.443 × ln(n)
  → Vì khác nhau chỉ hằng số → Big-O: O(log₂ n) = O(log₁₀ n) = O(log n)
```

## 2. Combinatorics — Tổ Hợp & Hoán Vị

### 2.1 Hoán Vị (Permutation) — THỨ TỰ QUAN TRỌNG

**Sắp xếp** n phần tử, chọn r phần tử, **có thứ tự**.

```
P(n, r) = n! / (n - r)!

Ví dụ: Xếp 3 người vào 3 ghế
  P(3, 3) = 3! = 6 cách: ABC, ACB, BAC, BCA, CAB, CBA

Ví dụ: Chọn 2 từ {A, B, C} có thứ tự
  P(3, 2) = 3! / 1! = 6: AB, AC, BA, BC, CA, CB

Trong coding: LC #46 Permutations → O(n!) solutions
```

### 2.2 Tổ Hợp (Combination) — THỨ TỰ KHÔNG QUAN TRỌNG

**Chọn** r phần tử từ n phần tử, **không có thứ tự**.

```
C(n, r) = n! / (r! × (n - r)!)

Ví dụ: Chọn 2 từ {A, B, C} không thứ tự
  C(3, 2) = 3! / (2! × 1!) = 3: AB, AC, BC
  (AB = BA, chỉ đếm 1 lần)

Trong coding:
  Subsets of size k → C(n, k) subsets
  LC #77 Combinations → backtracking
  Pascal's Triangle: C(n, r) = C(n-1, r-1) + C(n-1, r)
```

### 2.3 Tại sao quan trọng cho coding?

```javascript
// Biết tổng số solutions → biết complexity

// Tất cả subsets: 2^n (mỗi element include/exclude)
function subsets(nums) {
  /* ... */
}
// 2^20 = 1,048,576 → OK
// 2^30 ≈ 10^9 → quá chậm!

// Tất cả permutations: n!
function permutations(nums) {
  /* ... */
}
// 10! = 3,628,800 → OK
// 15! = 1.3 × 10^12 → quá chậm!

// Rule of thumb cho constraints:
//   n ≤ 10  → O(n!) OK (permutations)
//   n ≤ 20  → O(2^n) OK (subsets)
//   n ≤ 25  → O(2^n) vẫn khả thi
//   n ≤ 10^3 → O(n²) cần thiết
//   n ≤ 10^6 → O(n log n) hoặc O(n)
```

## 3. Probability — Xác Suất Cơ Bản

### 3.1 Công thức cơ bản

```
P(A) = số kết quả thuận lợi / tổng số kết quả
P(A) ∈ [0, 1]

P(A ∪ B) = P(A) + P(B) - P(A ∩ B)   (union)
P(A ∩ B) = P(A) × P(B)                (nếu A, B độc lập)
P(Ā)     = 1 - P(A)                   (complement)
```

### 3.2 Ứng dụng trong Backend

**Load Balancing — Random assignment:**

```
n servers, request random → P(server_i overloaded) phụ thuộc distribution
Random không đều → dùng Consistent Hashing!
```

**Bloom Filter — False Positive Rate:**

```
m bits, k hash functions, n elements inserted

P(false positive) ≈ (1 - e^(-kn/m))^k

Ví dụ: m=1000 bits, k=7
  n=100: FP ≈ 0.8%
  n=200: FP ≈ 5.5%
  n=500: FP ≈ 42%  → quá cao!

Trade-off: nhiều bits (m) → ít false positive nhưng tốn memory
         nhiều hash functions (k) → optimal khi k = (m/n) × ln(2)
```

**Birthday Paradox — Hash Collision:**

```
n items, m buckets
P(ít nhất 1 collision) ≈ 1 - e^(-n²/2m)

23 người → P(2 người cùng birthday) > 50%!
→ Hash table: collision xảy ra SỚM HƠN bạn nghĩ
→ UUID (128-bit): collision ở ~2^64 entries (Birthday bound)
```

### 3.3 Expected Value

```
E[X] = Σ (giá trị × xác suất)

Ví dụ: Random algorithm — expected number of attempts
  Quick Select: E[comparisons] = O(n) average
  Randomized Quick Sort: E[comparisons] = O(n log n)

  Skip List: mỗi node có p=0.5 chance "promote" lên level cao hơn
  → Expected height = O(log n)
  → Expected search = O(log n)
```

## 4. Modular Arithmetic — Số Học Modulo

### 4.1 Cơ bản

```
a mod m = phần dư khi chia a cho m

17 mod 5 = 2    (17 = 3 × 5 + 2)
-3 mod 5 = 2    (trong toán học; JS: -3 % 5 = -3 ⚠️)

Properties:
  (a + b) mod m = ((a mod m) + (b mod m)) mod m
  (a × b) mod m = ((a mod m) × (b mod m)) mod m
  (a - b) mod m = ((a mod m) - (b mod m) + m) mod m
```

### 4.2 Ứng dụng trong CS

```javascript
// 1. HASH FUNCTION — phổ biến nhất
function hash(key, tableSize) {
  return key % tableSize;
  // Modulo đảm bảo output nằm trong [0, tableSize-1]
}

// 2. CIRCULAR BUFFER
function nextIndex(current, capacity) {
  return (current + 1) % capacity;
  // 0 → 1 → 2 → ... → capacity-1 → 0 (wrap around!)
}

// 3. ROTATE ARRAY (LC #189)
function rotate(nums, k) {
  const n = nums.length;
  k = k % n; // nếu k > n → chỉ cần rotate k % n
  // ...
}

// 4. LARGE NUMBERS — tránh overflow
// "Tính (a^b) mod m" — Modular Exponentiation
function modPow(base, exp, mod) {
  let result = 1;
  base %= mod;
  while (exp > 0) {
    if (exp % 2 === 1) result = (result * base) % mod;
    exp = Math.floor(exp / 2);
    base = (base * base) % mod;
  }
  return result;
}
// O(log exp) — dùng trong RSA, Diffie-Hellman
```

### 4.3 Modulo trong Cryptography

```
Diffie-Hellman Key Exchange (simplified):
  Public: p (prime), g (generator)
  Alice secret: a → sends g^a mod p
  Bob secret: b → sends g^b mod p
  Both compute: g^(ab) mod p = SHARED SECRET!

  Attacker biết g^a mod p và g^b mod p
  Nhưng KHÔNG THỂ tính a hoặc b (Discrete Logarithm Problem)
  → An toàn!
```

## 5. Câu Hỏi Phỏng Vấn

**Q: Tại sao Binary Search là O(log n)?**

> A: Mỗi bước so sánh middle element, loại bỏ nửa array. n → n/2 → n/4 → ... → 1. Số bước = log₂(n). Với n = 10^9 (1 tỷ phần tử), chỉ cần ~30 lần so sánh. Đây là sức mạnh của logarithm — tăng n gấp đôi chỉ thêm 1 bước!

**Q: Bloom Filter hoạt động thế nào? False positive rate?**

> A: Bloom Filter dùng m-bit array + k hash functions. Insert: set k bits. Check: nếu TẤT CẢ k bits = 1 → "có thể exists" (false positive possible). Nếu BẤT KỲ bit = 0 → "chắc chắn NOT exists" (no false negative). FP rate ≈ (1 - e^(-kn/m))^k. Dùng trong: cache (check trước khi query DB), spam filter, network routing.

**Q: Modular arithmetic dùng ở đâu trong backend?**

> A: (1) Hash functions: `key % tableSize`. (2) Circular buffers: `(index + 1) % capacity`. (3) Rate limiting: sliding window counters. (4) Consistent hashing: `hash(key) % ring_size`. (5) Cryptography: RSA, Diffie-Hellman dùng modular exponentiation.
