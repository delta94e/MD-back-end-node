## 1. Process — Tiến Trình

### 1.1 Process là gì?

Process là một **chương trình đang chạy** (program in execution). Khi bạn chạy `node app.js`, OS tạo một process mới.

Mỗi process có:

- **Không gian địa chỉ riêng** (virtual address space) — isolated từ process khác
- **Process Control Block (PCB)** — metadata OS dùng để quản lý

```
┌──────────────────────────────────────────────────────┐
│              Process Control Block (PCB)              │
├────────────────────────┬─────────────────────────────┤
│ Process ID (PID)       │ 12345                       │
│ Process State          │ Running / Ready / Waiting   │
│ Program Counter        │ 0x00401000                  │
│ CPU Registers          │ R0=5, R1=10, ...            │
│ Memory Info            │ Page table pointer           │
│ I/O Status             │ Open files: [fd0, fd1, fd3] │
│ Priority               │ 20                          │
│ Parent PID             │ 12300                       │
│ Scheduling Info        │ CPU time used: 500ms        │
└────────────────────────┴─────────────────────────────┘
```

### 1.2 Process States (Trạng thái process)

```
                    ┌───────────────┐
         Tạo mới   │               │  Kết thúc
  ───────────────▶  │     NEW       │──────────────────▶ TERMINATED
                    │               │
                    └───────┬───────┘
                            │ admitted
                            ▼
                    ┌───────────────┐     I/O hoặc      ┌──────────────┐
                    │               │    event wait      │              │
          ┌────────▶│    READY      │────────────────▶   │   WAITING    │
          │         │  (chờ CPU)    │                    │  (chờ I/O)   │
          │         └───────┬───────┘                    └──────┬───────┘
          │                 │                                   │
          │        scheduler│dispatch              I/O complete │
          │                 ▼                      event occurs │
          │         ┌───────────────┐                           │
          │         │               │                           │
          └─────────│   RUNNING     │◀──────────────────────────┘
           preempt  │  (đang chạy)  │
          (time up) │               │
                    └───────────────┘
```

### 1.3 Process tạo Process (fork)

```
Terminal chạy `node app.js`:

  bash (PID: 100)
    │
    ├──▶ fork() ──▶ node (PID: 101)     ← process con
    │                  │
    │                  ├──▶ fork() ──▶ node worker (PID: 102)  ← cluster
    │                  ├──▶ fork() ──▶ node worker (PID: 103)
    │                  └──▶ fork() ──▶ node worker (PID: 104)
    │
    └── (parent process vẫn chạy)
```

## 2. Thread — Luồng

### 2.1 Thread là gì?

Thread là **đơn vị thực thi nhỏ nhất** trong một process. Một process có thể có nhiều threads chạy **đồng thời** trong cùng không gian địa chỉ.

```
┌──────────────────────────────────────────────────────────┐
│                     Process                               │
│                                                           │
│  ┌──────────────────────── SHARED ──────────────────────┐│
│  │  Code (Text)  │  Data (Globals)  │  Heap  │  Files   ││
│  └──────────────────────────────────────────────────────┘│
│                                                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │  Thread 1   │  │  Thread 2   │  │  Thread 3   │      │
│  │ ─────────── │  │ ─────────── │  │ ─────────── │      │
│  │ Stack       │  │ Stack       │  │ Stack       │      │
│  │ Registers   │  │ Registers   │  │ Registers   │      │
│  │ PC          │  │ PC          │  │ PC          │      │
│  │ Thread ID   │  │ Thread ID   │  │ Thread ID   │      │
│  └─────────────┘  └─────────────┘  └─────────────┘      │
│     (PRIVATE)        (PRIVATE)        (PRIVATE)          │
└──────────────────────────────────────────────────────────┘
```

### 2.2 Shared vs Private

| Shared (chung)         | Private (riêng mỗi thread) |
| ---------------------- | -------------------------- |
| Code segment           | Stack                      |
| Data segment (globals) | Registers (bao gồm PC, SP) |
| Heap                   | Thread ID                  |
| Open files & sockets   | Signal mask                |
| PID                    | errno                      |

## 3. Process vs Thread — So Sánh Chi Tiết

| Tiêu chí           | Process                                        | Thread                                                     |
| ------------------ | ---------------------------------------------- | ---------------------------------------------------------- |
| **Memory**         | Isolated (riêng hoàn toàn)                     | Shared heap, code, data                                    |
| **Tạo mới**        | Nặng (~ms, copy page tables)                   | Nhẹ (~μs)                                                  |
| **Context Switch** | Chậm (flush TLB, swap page tables)             | Nhanh hơn (cùng address space)                             |
| **Communication**  | IPC (pipes, sockets, shared memory) → phức tạp | Đọc/ghi shared memory → đơn giản nhưng cần synchronization |
| **Crash impact**   | Process crash → chỉ process đó chết            | Thread crash → **toàn bộ process chết**                    |
| **Security**       | Isolated tốt hơn                               | Một thread lỗi có thể corrupt data của thread khác         |
| **Scalability**    | Multi-machine (distributed) dễ hơn             | Giới hạn trong 1 machine                                   |

### Tại sao Node.js chọn single-threaded + multi-process?

```
Lý do chọn Single-Threaded Event Loop:
  1. Không cần locks/mutexes → không deadlock, no race conditions
  2. Developer experience đơn giản — không cần lo thread safety
  3. I/O-bound workloads: single thread đủ (CPU không phải bottleneck)

Khi cần multi-core:
  → cluster module (multi-process) → safe, isolated
  → worker_threads (multi-thread) → share memory khi cần performance
```

## 4. Context Switching

### 4.1 Context Switch là gì?

OS chuyển CPU từ chạy Process/Thread A sang Process/Thread B. OS phải **lưu state A** và **load state B**.

```
     Process A running          Context Switch          Process B running
  ┌──────────────────┐    ┌──────────────────────┐    ┌──────────────────┐
  │ Đang thực thi    │    │ 1. Save A's state    │    │ Tiếp tục thực thi│
  │ instruction      │───▶│    (registers, PC,   │───▶│ từ state đã save │
  │                  │    │     page table ptr)  │    │ trước đó         │
  │                  │    │ 2. Load B's state    │    │                  │
  │                  │    │ 3. Flush TLB (nếu    │    │                  │
  │                  │    │    khác process)     │    │                  │
  └──────────────────┘    └──────────────────────┘    └──────────────────┘
                            Cost: ~1-10 μs
                            (+ TLB cold start)
```

### 4.2 Chi phí thực tế

| Loại switch                        | Direct cost                                | Indirect cost                              |
| ---------------------------------- | ------------------------------------------ | ------------------------------------------ |
| **Thread → Thread** (cùng process) | ~1-2 μs (save/load registers)              | Minimal (cùng TLB)                         |
| **Process → Process**              | ~3-5 μs (save/load registers + page table) | TLB flush → TLB misses → ~100s ns per miss |

> **Thread context switch CÓ nhẹ hơn process context switch** — chủ yếu vì không cần flush TLB (cùng address space). Nhưng chi phí direct thì gần nhau, **indirect cost** (TLB cold) mới tạo sự khác biệt lớn.

## 5. CPU Scheduling Algorithms

OS cần quyết định process/thread nào được chạy tiếp khi CPU rảnh.

### 5.1 Các thuật toán phổ biến

**FCFS (First Come First Served)**:

```
Process:    P1(24ms)    P2(3ms)    P3(3ms)
Thứ tự:    P1 ──────────────────────▶ P2 ──▶ P3 ──▶
Wait time: P1=0, P2=24, P3=27.  Average = 17ms

Vấn đề: "Convoy effect" — process ngắn phải chờ process dài
```

**SJF (Shortest Job First)**:

```
Thứ tự:    P2 ──▶ P3 ──▶ P1 ──────────────────────▶
Wait time: P2=0, P3=3, P1=6.  Average = 3ms  (OPTIMAL cho average wait!)

Vấn đề: Starvation — process dài có thể không bao giờ được chạy
         Không thể biết trước burst time
```

**Round Robin (RR)**:

```
Time quantum = 4ms

P1(24) → P2(3) → P3(3) → P1(20) → P1(16) → P1(12) → ...
  4ms      3ms     3ms     4ms       4ms       4ms

Ưu điểm: Fair, response time tốt (time-sharing systems)
Nhược điểm: Context switch overhead nếu quantum quá nhỏ
```

**Priority Scheduling**:

```
Mỗi process có priority. Priority cao chạy trước.
Vấn đề: Starvation (priority thấp bị bỏ đói)
Giải pháp: Aging — tăng priority theo thời gian chờ
```

**Multilevel Feedback Queue (Linux CFS)**: Linux dùng **Completely Fair Scheduler** — mỗi process được chia thời gian CPU tỷ lệ thuận với weight (dựa trên nice value). Dùng red-black tree để O(log n) pick next process.

## 6. Deadlock

### 6.1 Deadlock là gì?

Hai hay nhiều processes **chờ đợi lẫn nhau** giữ resource, không ai nhường → tất cả đều bị block vĩnh viễn.

```
  Thread A                    Thread B
  ──────────                  ──────────
  Lock(Resource1) ✅          Lock(Resource2) ✅
       │                           │
  Đợi Resource2... ⏳         Đợi Resource1... ⏳
       │                           │
       └───── DEADLOCK! ──────────┘
       Cả hai chờ nhau mãi mãi
```

### 6.2 Bốn điều kiện Coffman (cần và đủ)

Deadlock xảy ra **khi và chỉ khi** cả 4 điều kiện đồng thời tồn tại:

| #   | Điều kiện            | Giải thích                                           |
| --- | -------------------- | ---------------------------------------------------- |
| 1   | **Mutual Exclusion** | Resource chỉ 1 process dùng tại 1 thời điểm          |
| 2   | **Hold and Wait**    | Process giữ resource này trong khi chờ resource khác |
| 3   | **No Preemption**    | Resource không thể bị cưỡng bức lấy lại              |
| 4   | **Circular Wait**    | Tồn tại vòng tròn chờ: A→B→C→A                       |

### 6.3 Phòng tránh Deadlock

| Phá điều kiện     | Cách làm                                                   |
| ----------------- | ---------------------------------------------------------- |
| Mutual Exclusion  | Dùng read-only resources khi có thể                        |
| Hold and Wait     | Yêu cầu tất cả resources cùng lúc (atomic)                 |
| No Preemption     | Cho phép timeout → release resources                       |
| **Circular Wait** | **Lock ordering** — luôn acquire locks theo thứ tự cố định |

### 6.4 Ví dụ thực tế (Database)

```sql
-- Transaction A:                      -- Transaction B:
BEGIN;                                  BEGIN;
UPDATE accounts SET bal=bal-100         UPDATE accounts SET bal=bal-50
  WHERE id=1;  -- Lock row 1             WHERE id=2;  -- Lock row 2

UPDATE accounts SET bal=bal+100         UPDATE accounts SET bal=bal+50
  WHERE id=2;  -- Chờ row 2!             WHERE id=1;  -- Chờ row 1!

-- DEADLOCK! PostgreSQL sẽ detect và rollback 1 transaction
```

### 6.5 Deadlock trong Node.js?

Node.js single-threaded → **không có traditional deadlock** (không có locks). Nhưng có thể gặp **logical deadlock** dạng:

- Promise chờ Promise (circular dependency)
- Distributed deadlock (Service A gọi Service B, B gọi A, cả hai đợi response)

```javascript
// Logical deadlock: circular Promise dependency
const a = new Promise((resolve) => {
  b.then(() => resolve("a done")); // a chờ b
});
const b = new Promise((resolve) => {
  a.then(() => resolve("b done")); // b chờ a
});
// → Cả hai Promise sẽ pending mãi mãi
```

## 7. Inter-Process Communication (IPC)

Processes có memory riêng biệt → cần cơ chế giao tiếp:

| Phương pháp           | Tốc độ         | Mô tả                                                            |
| --------------------- | -------------- | ---------------------------------------------------------------- |
| **Pipe**              | Nhanh          | Unidirectional byte stream giữa parent-child                     |
| **Named Pipe (FIFO)** | Nhanh          | Pipe có tên trong file system, không cần parent-child            |
| **Message Queue**     | Trung bình     | Kernel quản lý queue, structured messages                        |
| **Shared Memory**     | **Nhanh nhất** | Cùng physical memory mapped vào 2 processes. Cần synchronization |
| **Socket**            | Linh hoạt      | TCP/UDP, hoạt động cả local và network                           |
| **Signal**            | Đơn giản       | Thông báo sự kiện (SIGTERM, SIGKILL, SIGUSR1)                    |
| **Semaphore**         | Đồng bộ        | Counting mechanism cho resource access                           |

### Node.js IPC

```javascript
// Parent-child IPC qua fork()
// parent.js
const { fork } = require("child_process");
const child = fork("./child.js");

child.send({ type: "COMPUTE", data: [1, 2, 3] }); // IPC channel
child.on("message", (result) => console.log("Result:", result));

// child.js
process.on("message", (msg) => {
  if (msg.type === "COMPUTE") {
    const sum = msg.data.reduce((a, b) => a + b, 0);
    process.send({ type: "RESULT", value: sum }); // gửi về parent
  }
});
```

## 8. Concurrency vs Parallelism

```
Concurrency (đồng thời):
  Nhiều tasks TIẾN TRIỂN cùng lúc, nhưng có thể không thực sự chạy song song.
  Single-core: OS chuyển đổi nhanh giữa tasks (time-slicing)

    Core 0:  ─A─B─A─C─B─A─C─B─  (interleaved)

Parallelism (song song):
  Nhiều tasks THỰC SỰ chạy cùng lúc trên nhiều cores.

    Core 0:  ─A─A─A─A─A─A─A─A─
    Core 1:  ─B─B─B─B─B─B─B─B─
    Core 2:  ─C─C─C─C─C─C─C─C─
```

> **Node.js**: Concurrent (event loop xử lý nhiều requests cùng lúc) nhưng NOT parallel trên single thread. Dùng `cluster`/`worker_threads` để đạt parallelism.

## 9. Liên Hệ Với Node.js

| OS Concept | Node.js Equivalent                                                                  |
| ---------- | ----------------------------------------------------------------------------------- |
| Process    | `node app.js` = 1 OS process                                                        |
| fork()     | `child_process.fork()`, `cluster.fork()`                                            |
| Thread     | Event loop = main thread, libuv = thread pool (4 threads), `worker_threads`         |
| IPC        | `process.send()` / `process.on('message')`                                          |
| Signal     | `process.on('SIGTERM')`, `process.kill(pid, signal)`                                |
| Scheduling | OS scheduler. Node.js event loop là **application-level scheduler** cho async tasks |
| Mutex/Lock | Không cần (single-threaded). `worker_threads` dùng `Atomics` + `SharedArrayBuffer`  |

## 10. Câu Hỏi Phỏng Vấn

**Q: Process vs Thread khác nhau thế nào ở mức OS?**

> A: Process có isolated virtual address space (riêng page table, code, data, heap, stack). Thread chia sẻ address space của process (chung code, data, heap, files) nhưng có stack và registers riêng. Tạo process cần copy/set up page tables (~ms), tạo thread chỉ cần allocate stack (~μs). Process crash chỉ ảnh hưởng process đó; thread crash làm chết cả process.

**Q: Context switch của thread có thực sự nhẹ hơn process không?**

> A: Direct cost (save/load registers) gần bằng nhau (~1-5μs). Nhưng **indirect cost** khác biệt lớn: process switch phải thay đổi page table pointer → TLB flush → sau đó mỗi memory access đều TLB miss cho đến khi TLB "ấm" lại. Thread switch giữ nguyên address space → TLB entries vẫn valid → rất ít cache miss. Nên tổng chi phí thread switch nhẹ hơn đáng kể.

**Q: Deadlock 4 điều kiện cần và đủ? Cách phòng tránh?**

> A: (1) Mutual Exclusion, (2) Hold and Wait, (3) No Preemption, (4) Circular Wait. Cách phòng tránh hiệu quả nhất: **Lock Ordering** (phá Circular Wait) — tất cả threads/processes acquire locks theo cùng một thứ tự cố định. Trong database: dùng timeout (SET lock_timeout) hoặc deadlock detection (PostgreSQL tự detect và rollback victim). Trong distributed systems: dùng distributed lock với TTL (timeout tự release).
