## 1. Bốn Mô Hình I/O

Khi process gọi `read()` trên socket/file, có hai giai đoạn:

1. **Chờ data sẵn sàng** (data từ network/disk đến kernel buffer)
2. **Copy data** từ kernel buffer → user buffer

Dựa vào hành vi ở 2 giai đoạn này, có 4 mô hình I/O:

```
                         Wait for data    Copy data
                         ──────────────   ──────────
Blocking I/O:            Block            Block        ← đơn giản nhất
Non-blocking I/O:        Return immediately (poll)  Block
I/O Multiplexing:        Block on select/epoll      Block
Async I/O (AIO):         Non-block         Non-block  ← notify khi xong
```

## 2. Blocking I/O

```
Application                        Kernel
    │                                 │
    │  read(socket, buf)              │
    │ ──────────────────────────────▶ │
    │                                 │  Chờ data đến...
    │       BLOCKED!                  │  (network packet,
    │   (thread ngủ,                  │   disk sector)
    │    không làm gì)                │
    │                                 │  Data đến!
    │                                 │  Copy kernel→user
    │ ◀────────────────────────────── │
    │  return data                    │
    │                                 │
    ▼  Tiếp tục xử lý                │
```

**Đặc điểm**:

- Thread bị **block hoàn toàn** cho đến khi I/O hoàn tất
- Mỗi client cần **1 thread riêng** → C10K problem (10,000 clients = 10,000 threads)
- Thread tốn ~1MB stack memory + context switch overhead

```javascript
// Node.js: fs.*Sync() là blocking I/O
const data = fs.readFileSync("/data/big-file.txt"); // Thread ngủ ở đây!
// Không gì khác được xử lý cho đến khi đọc xong
console.log("Done"); // Chỉ chạy sau khi file đọc xong
```

**Mô hình server**: Thread-per-connection (Java Servlet truyền thống, PHP-FPM)

```
Client 1 ──▶ Thread 1 ──▶ read(DB)... block... return... respond
Client 2 ──▶ Thread 2 ──▶ read(DB)... block... return... respond
Client 3 ──▶ Thread 3 ──▶ read(DB)... block... return... respond
...
Client 1000 ──▶ Thread 1000 (1GB memory cho stacks!)
```

## 3. Non-blocking I/O

```
Application                        Kernel
    │                                 │
    │  read(socket, buf, O_NONBLOCK)  │
    │ ──────────────────────────────▶ │
    │ ◀─── EWOULDBLOCK (not ready)   │  (chưa có data)
    │                                 │
    │  ... làm việc khác ...          │
    │                                 │
    │  read(socket, buf, O_NONBLOCK)  │  POLLING
    │ ──────────────────────────────▶ │
    │ ◀─── EWOULDBLOCK (not ready)   │  (vẫn chưa)
    │                                 │
    │  read(socket, buf, O_NONBLOCK)  │
    │ ──────────────────────────────▶ │
    │ ◀─── data!                      │  Data đến! Copy và return
    │                                 │
    ▼  Xử lý data                    │
```

**Đặc điểm**:

- Thread **không bị block** — nhận `EWOULDBLOCK` ngay nếu chưa có data
- Cần **polling liên tục** (busy-waiting) → lãng phí CPU
- Không thực dụng khi dùng đơn lẻ → kết hợp với I/O multiplexing

## 4. I/O Multiplexing — select, poll, epoll

> 💡 Ý tưởng: **Một thread** giám sát **nhiều fd** cùng lúc. Khi bất kỳ fd nào ready → xử lý.

```
Application                              Kernel
    │                                      │
    │  epoll_wait(fds=[sock1,sock2,...])    │
    │ ────────────────────────────────────▶ │
    │                                      │  Giám sát tất cả sockets...
    │       BLOCKED on epoll               │
    │   (nhưng giám sát NHIỀU fd           │
    │    cùng lúc!)                         │
    │                                      │  sock3 có data!
    │ ◀──── sock3 ready!                   │
    │                                      │
    │  read(sock3, buf)                    │
    │ ────────────────────────────────────▶ │
    │ ◀──── data                           │  Copy data
    │                                      │
    │  process(data)                       │
    │  epoll_wait(...)  // tiếp tục         │
```

**Đây chính là cách Node.js hoạt động!** — libuv dùng `epoll` (Linux) / `kqueue` (macOS) / `IOCP` (Windows).

## 5. Asynchronous I/O (AIO)

```
Application                        Kernel
    │                                 │
    │  aio_read(socket, buf, callback)│
    │ ──────────────────────────────▶ │
    │ ◀─── OK (submitted)            │  (return ngay lập tức)
    │                                 │
    │  ... làm việc khác ...          │  Kernel đọc data...
    │  ... hoàn toàn free ...         │  Copy data...
    │                                 │
    │ ◀─── SIGNAL: data ready!        │  Notify khi hoàn tất
    │                                 │
    ▼  Xử lý data trong callback      │
```

**Đặc điểm**: Hoàn toàn non-blocking ở cả 2 giai đoạn. Linux AIO (`io_uring` từ kernel 5.1+) hỗ trợ true async I/O.

> **Node.js**: Dù API `fs.readFile()` trông "async", thực tế nó dùng **thread pool** (libuv) + blocking read ở worker thread, KHÔNG phải kernel AIO. Network I/O mới dùng epoll thật sự non-blocking.

## 6. So Sánh Tổng Hợp

| Mô hình           | Wait            | Copy      | Thread cần    | Performance         |
| ----------------- | --------------- | --------- | ------------- | ------------------- |
| **Blocking**      | Block           | Block     | 1/client      | O(n) threads        |
| **Non-blocking**  | Return (poll)   | Block     | 1 (busy loop) | CPU waste           |
| **I/O Multiplex** | Block on select | Block     | 1 cho tất cả  | ✅ Scalable         |
| **Async I/O**     | Non-block       | Non-block | 1 cho tất cả  | ✅ Best (lý thuyết) |

## 7. epoll vs select vs poll — Chi Tiết

### 7.1 select

```c
// Pseudocode
fd_set read_fds;
FD_SET(sock1, &read_fds);
FD_SET(sock2, &read_fds);
FD_SET(sock3, &read_fds);

select(max_fd + 1, &read_fds, NULL, NULL, timeout);
// Returns: số fd ready

// Phải loop qua TẤT CẢ fd để tìm fd nào ready
for (int fd = 0; fd <= max_fd; fd++) {
  if (FD_ISSET(fd, &read_fds)) {
    // fd ready → xử lý
  }
}
```

**Nhược điểm**:

- Giới hạn **FD_SETSIZE = 1024** fd
- Mỗi lần gọi phải **copy toàn bộ fd_set** từ user space → kernel space
- Return → phải **scan toàn bộ** O(n) để tìm fd ready
- Mỗi lần gọi lại phải **khởi tạo lại** fd_set

### 7.2 poll

```c
struct pollfd fds[3];
fds[0] = { .fd = sock1, .events = POLLIN };
fds[1] = { .fd = sock2, .events = POLLIN };
fds[2] = { .fd = sock3, .events = POLLIN };

poll(fds, 3, timeout);
// Không giới hạn số fd (dùng array thay vì bitmap)
// Nhưng vẫn O(n) scan mỗi lần gọi
```

**Cải thiện**: Không có giới hạn 1024. Nhưng vẫn O(n) scan.

### 7.3 epoll (Linux) / kqueue (macOS)

```c
// 1. Tạo epoll instance (chỉ 1 lần)
int epfd = epoll_create(1);

// 2. Đăng ký fd quan tâm (chỉ khi có fd mới)
struct epoll_event ev = { .events = EPOLLIN, .data.fd = sock1 };
epoll_ctl(epfd, EPOLL_CTL_ADD, sock1, &ev);  // thêm sock1
epoll_ctl(epfd, EPOLL_CTL_ADD, sock2, &ev);  // thêm sock2

// 3. Chờ events (gọi lặp lại)
struct epoll_event events[MAX_EVENTS];
int n = epoll_wait(epfd, events, MAX_EVENTS, timeout);

// CHỈ trả về fd READY — không cần scan O(n)!
for (int i = 0; i < n; i++) {
  handle(events[i].data.fd);  // Chỉ xử lý fd ready
}
```

### 7.4 So sánh hiệu năng

|                          | select       | poll          | epoll/kqueue                |
| ------------------------ | ------------ | ------------- | --------------------------- |
| Max fd                   | 1024         | Unlimited     | Unlimited                   |
| Mỗi lần gọi              | Copy all fds | Copy all fds  | Không copy (kernel track)   |
| Tìm fd ready             | O(n) scan    | O(n) scan     | O(1) — chỉ return ready fds |
| fd thêm/xóa              | Rebuild set  | Rebuild array | `epoll_ctl` — incremental   |
| **10,000 fds, 10 ready** | Scan 10,000  | Scan 10,000   | **Return 10**               |

> **Kết luận**: Với 10,000+ connections, `epoll` nhanh hơn `select/poll` hàng trăm lần vì O(active) thay vì O(total).

## 8. Liên Hệ Trực Tiếp Với Node.js Event Loop

### Kiến trúc Node.js I/O

```
┌──────────────────────────────────────────────────────────┐
│                       Node.js                             │
│                                                           │
│  ┌──────────────────┐                                    │
│  │  V8 Engine       │  JavaScript execution              │
│  │  (Call Stack)     │                                    │
│  └────────┬─────────┘                                    │
│           │                                               │
│  ┌────────▼─────────────────────────────────────────┐    │
│  │              libuv Event Loop                      │    │
│  │                                                    │    │
│  │  Network I/O:    epoll/kqueue (kernel) ── NO threads│    │
│  │  File I/O:       Thread Pool (4 threads) ── blocking│    │
│  │  DNS lookup:     Thread Pool                        │    │
│  │  Crypto/Zlib:    Thread Pool                        │    │
│  └────────────────────────────────────────────────────┘    │
│                                                           │
│  Khi epoll_wait return "socket ready":                    │
│    → libuv đọc data                                      │
│    → tạo callback                                         │
│    → đẩy vào Event Loop queue                             │
│    → V8 call stack rảnh → pick callback → execute         │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### Tại sao Node.js handle 10,000+ concurrent connections trên 1 thread?

```
Câu trả lời đầy đủ:

1. Network I/O dùng epoll (Linux) / kqueue (macOS):
   - KHÔNG tạo thread cho mỗi connection
   - 1 epoll instance giám sát hàng nghìn sockets
   - epoll_wait block cho đến khi ≥1 socket ready
   - Chỉ xử lý sockets CÓ data → O(active), không O(total)

2. Mỗi request xử lý rất nhanh (non-CPU-bound):
   - Parse request (~μs)
   - Gửi DB query (async, non-blocking)
   - Gửi response khi DB callback (~μs)
   - Tổng CPU time per request: < 1ms

3. Nếu 10,000 connections, mỗi request 0.1ms CPU:
   - 1 thread xử lý: 10,000 × 0.1ms = 1 second
   - Throughput: ~10,000 RPS trên SINGLE THREAD

4. File I/O và crypto dùng thread pool (default 4 threads)
   vì OS APIs cho file thường blocking.
   → UV_THREADPOOL_SIZE=128 nếu heavy file I/O
```

### Code demo: epoll-style thinking in Node.js

```javascript
const http = require("http");

// Node.js tương đương với:
// while (true) {
//   events = epoll_wait(eventfd); // chờ socket ready
//   for (event of events) {
//     if (event == new_connection) accept();
//     if (event == data_ready) handleRequest();
//     if (event == write_ready) sendResponse();
//   }
// }

const server = http.createServer((req, res) => {
  // Callback này chạy khi epoll báo "socket có data"
  // Xử lý nhanh (non-blocking) → return → tiếp tục epoll_wait
  res.end("Hello");
});

server.listen(3000);
// Bên dưới: libuv gọi epoll_create → bind socket → epoll_wait loop
```

## 9. Câu Hỏi Phỏng Vấn

**Q: `epoll` khác `select` và `poll` ra sao? Tại sao `epoll` hiệu quả hơn ở scale lớn?**

> A: `select` giới hạn 1024 fds, mỗi lần gọi copy toàn bộ fd_set vào kernel và phải O(n) scan để tìm fd ready. `poll` bỏ giới hạn 1024 nhưng vẫn O(n) scan. `epoll` dùng cách tiếp cận khác: (1) đăng ký fd một lần bằng `epoll_ctl`, kernel tự track; (2) `epoll_wait` CHỈ trả về danh sách fd ready, không cần scan → O(active) thay vì O(total). Với 10,000 connections nhưng chỉ 10 active, select scan 10,000 còn epoll chỉ return 10.

**Q: Liên hệ epoll với Event Loop của Node.js?**

> A: libuv (engine bên dưới Node.js) dùng epoll (Linux) / kqueue (macOS) / IOCP (Windows) cho network I/O. Event loop gọi `epoll_wait` → nhận danh sách sockets ready → đọc data → tạo callbacks → đẩy vào event queue → V8 thực thi. Vì mỗi callback xử lý nhanh (microseconds), 1 thread đủ handle tens of thousands connections. File I/O dùng thread pool vì `read()` syscall là blocking.

**Q: Blocking vs Non-blocking vs Async — phân biệt rõ ràng?**

> A: **Blocking**: thread ngủ cho đến khi I/O hoàn tất. **Non-blocking**: system call return ngay (EWOULDBLOCK nếu chưa ready), nhưng khi data copy vẫn block. **Async**: hoàn toàn non-blocking — kernel thực hiện cả chờ + copy, notify khi xong. Trong Node.js: network I/O = non-blocking + epoll (gần async), file I/O = blocking trên thread pool worker (giả async — vì từ góc nhìn main thread, nó non-blocking).
