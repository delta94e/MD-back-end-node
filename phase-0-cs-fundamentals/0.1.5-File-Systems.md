## 1. File System Là Gì?

File system là **lớp phần mềm** quản lý cách data được tổ chức, lưu trữ và truy xuất trên storage device (SSD/HDD).

```
Application (Node.js)
    │  fs.readFile('/data/users.json')
    ▼
VFS (Virtual File System) ─── abstraction layer
    │
    ├──▶ ext4    (Linux)
    ├──▶ APFS    (macOS)
    ├──▶ NTFS    (Windows)
    └──▶ tmpfs   (RAM-based)
    │
    ▼
Block Device Driver
    │
    ▼
Physical Storage (SSD / HDD)
```

**VFS (Virtual File System)**: Linux cung cấp VFS = API chung cho mọi file system. Application gọi `open()`, `read()`, `write()` → VFS chuyển đến file system cụ thể. Đây là ví dụ điển hình của **Adapter pattern**.

### Các File System phổ biến

| File System | OS            | Đặc điểm                                        |
| ----------- | ------------- | ----------------------------------------------- |
| **ext4**    | Linux         | Journaling, extents, mature, default Linux      |
| **XFS**     | Linux         | Tốt cho large files, parallel I/O               |
| **Btrfs**   | Linux         | Copy-on-write, snapshots, checksums             |
| **APFS**    | macOS         | Encryption, snapshots, space sharing            |
| **NTFS**    | Windows       | ACL, compression, encryption                    |
| **ZFS**     | FreeBSD/Linux | Enterprise: RAID + volume manager + file system |

## 2. Block Storage — Lưu Trữ Theo Khối

### 2.1 Disk không lưu từng byte — mà lưu theo BLOCK

SSD/HDD chia thành các **blocks** (thường 4KB). Đọc/ghi 1 byte cũng phải đọc/ghi nguyên 1 block 4KB.

```
Physical Disk:
┌────────┬────────┬────────┬────────┬────────┬────────┐
│Block 0 │Block 1 │Block 2 │Block 3 │Block 4 │Block 5 │  ...
│ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │
│SuperBlk│Inode   │Inode   │Data    │Data    │Data    │
│        │Table   │Table   │Block   │Block   │Block   │
└────────┴────────┴────────┴────────┴────────┴────────┘
```

### 2.2 Disk Layout (ext4)

```
┌───────────┬──────────────┬─────────────┬──────────────────┐
│ Super     │   Inode      │   Block     │                  │
│ Block     │   Table      │   Bitmap    │   Data Blocks    │
│(metadata) │(file info)   │(free/used)  │(actual content)  │
└───────────┴──────────────┴─────────────┴──────────────────┘
  - Super Block: FS info (block size, total blocks, free blocks)
  - Inode Table: metadata cho mỗi file/directory
  - Block Bitmap: track block nào đang dùng/free
  - Data Blocks: nội dung file thực sự
```

## 3. Inodes — "Căn Cước" Của File

### 3.1 Inode là gì?

Mỗi file/directory có **đúng 1 inode** — chứa tất cả metadata **trừ tên file**.

```
┌──────────────────────────────────────────┐
│            Inode #12345                   │
├──────────────────────────────────────────┤
│ File Type:      Regular file             │
│ Permissions:    rwxr-xr-x (755)          │
│ Owner UID:      1000                     │
│ Group GID:      1000                     │
│ Size:           13,847 bytes             │
│ Timestamps:                              │
│   Access (atime): 2026-03-01 10:30:00    │
│   Modify (mtime): 2026-02-28 14:15:00   │
│   Change (ctime): 2026-02-28 14:15:00   │
│ Link Count:     2                        │
│ Block Count:    4                        │
│                                          │
│ Data Block Pointers:                     │
│   Direct[0]:     block 1024             │
│   Direct[1]:     block 1025             │
│   Direct[2]:     block 2048             │
│   Direct[3]:     block 5000             │
│   ...                                    │
│   Indirect:      block 6000 ──▶ [ptrs]  │
│   Double Indirect: block 7000           │
│   Triple Indirect: block 8000           │
└──────────────────────────────────────────┘

Tên file KHÔNG nằm trong inode!
Tên file nằm trong directory entry: { "users.json": inode #12345 }
```

### 3.2 Hard Links vs Symbolic Links

```
Hard Link: Nhiều tên → cùng 1 inode (cùng data blocks)
  "file_a.txt"  ─┐
                  ├──▶ Inode #100 ──▶ Data Blocks
  "file_b.txt"  ─┘
  Link count = 2. Xóa file_a → file_b vẫn hoạt động.

Symbolic Link: Tên → trỏ đến path string (shortcut)
  "shortcut" ──▶ Inode #200 ──▶ data = "/path/to/file_a.txt"
  Xóa file_a → shortcut bị broken (dangling symlink)
```

```bash
# Terminal commands
ln file_a.txt file_b.txt        # Hard link
ln -s file_a.txt shortcut       # Symbolic link
ls -li                           # Xem inode numbers
stat file_a.txt                  # Xem chi tiết inode
```

## 4. File Descriptors — "Tay Cầm" Để Thao Tác File

### 4.1 File Descriptor (fd) là gì?

Khi process mở file, OS trả về một **integer** gọi là file descriptor — đó là "tay cầm" để đọc/ghi file đó.

```
Process file descriptor table:
┌──────┬─────────────────────────────┐
│  fd  │ Trỏ đến                    │
├──────┼─────────────────────────────┤
│  0   │ stdin  (Standard Input)    │  ← luôn mở sẵn
│  1   │ stdout (Standard Output)   │  ← luôn mở sẵn
│  2   │ stderr (Standard Error)    │  ← luôn mở sẵn
│  3   │ /data/users.json           │  ← mới mở
│  4   │ TCP socket (port 3000)     │  ← network connection
│  5   │ /var/log/app.log           │  ← log file
└──────┴─────────────────────────────┘
```

> **"Everything is a file" (Unix philosophy)**: Files, directories, sockets, pipes, devices — tất cả đều access qua file descriptors. Đây là abstraction cực kỳ mạnh.

### 4.2 File Descriptor Limit & Leak

```bash
# Xem giới hạn fd
ulimit -n        # Thường 1024 hoặc 65535

# Mỗi connection = 1 fd. Server 10,000 connections = 10,000 fd.
# Nếu không đóng fd → FD LEAK → "Too many open files" error
```

```javascript
// Node.js fd leak example
const fs = require("fs");

// BUG: mở file nhưng không đóng
function readData() {
  const fd = fs.openSync("/data/file.txt", "r");
  const buffer = Buffer.alloc(1024);
  fs.readSync(fd, buffer);
  // fs.closeSync(fd);  ← MISSING! fd bị leak
  return buffer.toString();
}

// FIX: dùng fs.readFileSync() hoặc try/finally
function readDataFixed() {
  const fd = fs.openSync("/data/file.txt", "r");
  try {
    const buffer = Buffer.alloc(1024);
    fs.readSync(fd, buffer);
    return buffer.toString();
  } finally {
    fs.closeSync(fd); // Luôn đóng, kể cả khi error
  }
}

// BEST: dùng high-level API (tự quản lý fd)
const data = fs.readFileSync("/data/file.txt", "utf-8");
```

## 5. Directory Structure

Directory là **file đặc biệt** chứa danh sách entries: `{ tên_file → inode_number }`.

```
Directory "/home/user/project":
┌──────────────────┬───────────┐
│ Filename         │ Inode #   │
├──────────────────┼───────────┤
│ .                │ 5001      │  (self)
│ ..               │ 5000      │  (parent)
│ package.json     │ 5002      │
│ src/             │ 5003      │
│ node_modules/    │ 5004      │
│ README.md        │ 5005      │
└──────────────────┴───────────┘
```

### Path Resolution

```
Truy cập "/home/user/project/src/app.js":

  1. Inode 2 (root "/")        → tìm entry "home"    → inode 100
  2. Inode 100 ("/home")       → tìm entry "user"    → inode 5000
  3. Inode 5000 ("/home/user") → tìm entry "project" → inode 5001
  4. Inode 5001 ("project")    → tìm entry "src"     → inode 5003
  5. Inode 5003 ("src")        → tìm entry "app.js"  → inode 5010
  6. Đọc inode 5010 → data block pointers → đọc data

  Mỗi "/" = 1 lần đọc directory = 1+ disk I/O
  Path dài = chậm! (OS cache giúp giảm thiểu)
```

## 6. Journaling — Bảo Vệ Tính Toàn Vẹn

### 6.1 Vấn đề: Crash giữa chừng

Ghi file cần update nhiều thứ: data blocks, inode, bitmap. Nếu **mất điện** giữa chừng → file system **inconsistent** (corrupted).

### 6.2 Journal = Write-Ahead Log

```
Thay vì ghi trực tiếp:

  1. GHI VÀO JOURNAL:    "Sẽ update inode #100 và block #500"
  2. GHI VÀO DISK:       Thực hiện update inode #100 và block #500
  3. XÓA JOURNAL ENTRY:  Đánh dấu hoàn tất

Nếu crash ở bước 2 → khi reboot, đọc journal → redo các thao tác chưa hoàn tất
Nếu crash ở bước 1 → chưa ghi gì → safe, chỉ mất data mới nhất
```

> **Liên hệ**: Đây là chính xác nguyên lý **WAL (Write-Ahead Log)** trong PostgreSQL! Database và file system dùng cùng strategy để đảm bảo durability.

### 6.3 Journaling Modes

| Mode          | Journal chứa                            | Performance | Safety                  |
| ------------- | --------------------------------------- | ----------- | ----------------------- |
| **Journal**   | Metadata + Data                         | Chậm nhất   | An toàn nhất            |
| **Ordered**   | Metadata only (data ghi trước metadata) | Cân bằng    | Tốt                     |
| **Writeback** | Metadata only (data ghi bất kỳ lúc nào) | Nhanh nhất  | Rủi ro file content sai |

## 7. Liên Hệ Với Node.js & Backend

| File System Concept | Node.js / Backend                                              |
| ------------------- | -------------------------------------------------------------- |
| File Descriptors    | `fs.open()` trả về fd, socket connections = fd                 |
| Block I/O           | `fs.read()` đọc theo buffer size, Streams dùng `highWaterMark` |
| Journaling / WAL    | PostgreSQL WAL, Redis AOF, MongoDB journal                     |
| Inode timestamps    | `fs.stat()` trả về `atime`, `mtime`, `ctime`                   |
| fd limit            | Server nhiều connections → cần tăng `ulimit -n`                |
| tmpfs               | `/tmp` thường mount trên RAM → nhanh cho temp files            |

```javascript
// Xem file stats (inode info) trong Node.js
const fs = require("fs");
const stats = fs.statSync("package.json");

console.log({
  inode: stats.ino, // Inode number
  size: stats.size, // bytes
  blocks: stats.blocks, // số 512-byte blocks
  hardLinks: stats.nlink, // hard link count
  permissions: stats.mode.toString(8), // octal (e.g., "100644")
  created: stats.birthtime, // creation time
  modified: stats.mtime, // last data modification
  accessed: stats.atime, // last access
  changed: stats.ctime, // last inode change (permissions, etc.)
});
```

## 8. Câu Hỏi Phỏng Vấn

**Q: Inode là gì? Tại sao tên file không nằm trong inode?**

> A: Inode chứa metadata (permissions, size, timestamps, data block pointers) và tên file nằm trong directory entry. Tách biệt vì: (1) cho phép hard links — nhiều tên trỏ đến cùng inode, (2) rename file chỉ cần update directory entry, không cần sửa inode.

**Q: File descriptor là gì? Tại sao "Too many open files"?**

> A: fd là integer mà OS trả về khi process mở file/socket. Mỗi process có giới hạn fd (thường 1024-65535). Server với nhiều connections, mỗi connection = 1 fd. Nếu kết nối bị leak (không close) → hết fd → "EMFILE: too many open files". Fix: đóng connection đúng, tăng `ulimit -n`, dùng connection pooling.

**Q: Journaling giải quyết vấn đề gì? Liên hệ với database WAL?**

> A: Journaling giải quyết crash consistency — khi system crash giữa chừng write, file system vẫn recoverable. Ghi journal (intent) trước, ghi data sau. Nếu crash → replay journal khi boot. PostgreSQL WAL cùng nguyên lý: ghi WAL record trước khi modify data pages, crash → replay WAL records. Cả hai đều là dạng write-ahead logging.
