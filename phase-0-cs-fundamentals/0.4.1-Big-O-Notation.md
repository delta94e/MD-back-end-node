## 1. Big-O Là Gì — Ý Nghĩa Thực Sự

Big-O mô tả **tốc độ tăng trưởng** của thời gian/bộ nhớ khi input tăng, **bỏ qua hằng số và bậc thấp**.

```
Ý tưởng: Khi n rất lớn, chỉ số hạng lớn nhất quan trọng.

f(n) = 3n² + 5n + 100
  Khi n = 10:    300 + 50 + 100 = 450
  Khi n = 1000:  3,000,000 + 5,000 + 100 = 3,005,100
  Khi n = 10^6:  3×10^12 + 5×10^6 + 100 ≈ 3×10^12

→ 5n và 100 trở nên VÔ NGHĨA so với 3n²
→ Hệ số 3 cũng bỏ (Big-O chỉ quan tâm TỐC ĐỘ TĂNG)
→ f(n) = O(n²)
```

**Big-O = upper bound** — "thuật toán KHÔNG BAO GIỜ chậm hơn mức này".

## 2. Các Mức Complexity Phổ Biến

```
Thời gian chạy khi n = 1,000,000 (1 triệu):

O(1)        →  1              ← Hash table lookup
O(log n)    →  20             ← Binary search
O(n)        →  1,000,000      ← Linear scan
O(n log n)  →  20,000,000     ← Merge sort
O(n²)       →  10^12          ← Bubble sort (1 TRIỆU GIÂY!)
O(2^n)      →  10^301029      ← Brute force combinations (VŨ TRỤ CHẾT!)
O(n!)       →  ∞              ← Traveling Salesman brute force

Biểu đồ tăng trưởng:
  │
  │                                        O(2^n)
  │                                      /
  │                               O(n²) /
  │                             /      /
  │                           /      /
  │                    O(n log n)  /
  │                   /        /
  │              O(n)        /
  │            /          /
  │       O(log n)     /
  │     ··────────···
  │  O(1)
  └──────────────────────────────── n
```

### Bảng tham chiếu nhanh

| Complexity | Tên          | Ví dụ                     | n = 10^6 khả thi? |
| ---------- | ------------ | ------------------------- | :---------------: |
| O(1)       | Constant     | Array access, hash lookup |        ✅         |
| O(log n)   | Logarithmic  | Binary search             |        ✅         |
| O(√n)      | Square root  | Prime check               |        ✅         |
| O(n)       | Linear       | Array scan                |        ✅         |
| O(n log n) | Linearithmic | Merge sort, heap sort     |        ✅         |
| O(n²)      | Quadratic    | Nested loops, bubble sort |     ⚠️ (chậm)     |
| O(n³)      | Cubic        | Floyd-Warshall            |   ❌ (n ≤ 500)    |
| O(2^n)     | Exponential  | Subsets, recursive fib    |    ❌ (n ≤ 25)    |
| O(n!)      | Factorial    | Permutations              |    ❌ (n ≤ 12)    |

> **Interview tip**: Khi interviewer nói "n up to 10^5" → solution phải ≤ O(n log n). "n up to 10^3" → O(n²) OK. "n up to 20" → O(2^n) OK.

## 3. Rules Phân Tích Big-O

### Rule 1: Bỏ hằng số

```javascript
// O(3n) = O(n)
for (let i = 0; i < n; i++) {} // n iterations
for (let i = 0; i < n; i++) {} // n iterations
for (let i = 0; i < n; i++) {} // n iterations
// Total: 3n → O(n)
```

### Rule 2: Bỏ bậc thấp hơn

```javascript
// O(n² + n) = O(n²)
for (let i = 0; i < n; i++) {
  // n
  for (let j = 0; j < n; j++) {} // × n = n²
}
for (let k = 0; k < n; k++) {} // + n
// Total: n² + n → O(n²)
```

### Rule 3: Nối tiếp → cộng. Lồng nhau → nhân.

```javascript
// Nối tiếp: O(A) + O(B)
for (let i = 0; i < A; i++) {} // O(A)
for (let j = 0; j < B; j++) {} // O(B)
// Total: O(A + B)

// Lồng nhau: O(A) × O(B)
for (let i = 0; i < A; i++) {
  // O(A)
  for (let j = 0; j < B; j++) {} //  × O(B)
}
// Total: O(A × B)
// ⚠️ CHỈ = O(n²) khi A = B = n!
```

### Rule 4: Log xuất hiện khi chia đôi

```javascript
// O(log n) — mỗi bước chia đôi search space
let i = n;
while (i > 1) {
  i = Math.floor(i / 2); // n → n/2 → n/4 → ... → 1
}
// Số bước: log₂(n)

// O(n log n) — n elements, mỗi element xử lý log n bước
// → Merge Sort: chia log n levels × merge n elements mỗi level
```

### Rule 5: Recursive — dùng recurrence relation

```javascript
// T(n) = T(n/2) + O(1) → O(log n)  (Binary Search)
// T(n) = 2T(n/2) + O(n) → O(n log n)  (Merge Sort)
// T(n) = T(n-1) + O(n) → O(n²)  (Selection Sort)
// T(n) = 2T(n-1) + O(1) → O(2^n)  (Naive Fibonacci)
```

## 4. Best / Worst / Average Case

```
Quick Sort example:
  Best case:    O(n log n)  — pivot luôn chia đều
  Average case: O(n log n)  — hầu hết trường hợp
  Worst case:   O(n²)       — pivot luôn min/max (sorted input!)

Phỏng vấn: luôn hỏi WORST CASE trừ khi nói rõ otherwise.

Ví dụ:
  Hash Table lookup:
    Best/Average: O(1)
    Worst: O(n) — tất cả keys cùng hash → linked list

  BST search:
    Best/Average: O(log n) — balanced
    Worst: O(n) — skewed (thành linked list)
```

## 5. Space Complexity

Đo lượng **bộ nhớ bổ sung** mà thuật toán sử dụng (không tính input).

```javascript
// O(1) space — in-place
function bubbleSort(arr) {
  for (let i = 0; i < arr.length; i++)
    for (let j = 0; j < arr.length - 1; j++)
      if (arr[j] > arr[j + 1]) [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
  // Chỉ dùng thêm vài biến → O(1) extra space
}

// O(n) space — tạo array mới
function mergeSort(arr) {
  if (arr.length <= 1) return arr;
  const mid = Math.floor(arr.length / 2);
  const left = mergeSort(arr.slice(0, mid)); // tạo copy!
  const right = mergeSort(arr.slice(mid)); // tạo copy!
  return merge(left, right); // tạo result!
  // Extra space: O(n) cho arrays + O(log n) cho call stack
}

// O(n) space — recursion call stack
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
  // Call stack depth = n → O(n) space (dù time = O(2^n)!)
}
```

| Algorithm         | Time           | Space                    |
| ----------------- | -------------- | ------------------------ |
| Bubble Sort       | O(n²)          | **O(1)**                 |
| Merge Sort        | O(n log n)     | **O(n)**                 |
| Quick Sort        | O(n log n) avg | **O(log n)** call stack  |
| BFS               | O(V + E)       | **O(V)** queue           |
| DFS               | O(V + E)       | **O(V)** stack/recursion |
| Hash Map solution | O(n)           | **O(n)** map storage     |

## 6. Amortized Analysis

Một operation đôi khi đắt, nhưng **trung bình over n operations** thì rẻ.

```
Dynamic Array push:
  Push 1-4: O(1) mỗi lần (có chỗ trống)
  Push 5:   O(4) — resize! copy 4 elements + push 1
  Push 6-8: O(1) mỗi lần
  Push 9:   O(8) — resize! copy 8 elements + push 1
  ...

  n pushes: n + (1 + 2 + 4 + 8 + ... + n/2) copies ≈ n + n = 2n
  Amortized cost per push = 2n / n = O(1) ✓

  "Mỗi push TRẢ TRƯỚC 1 xu cho resize tương lai"
  → Khi resize xảy ra, đã tích đủ xu → chi phí = 0 amortized

Practical examples:
  Array.push()  → amortized O(1)
  Hash table insert → amortized O(1) (rehash đôi khi O(n))
  Splay tree    → amortized O(log n)
```

## 7. Big-Ω và Big-Θ

```
Big-O (O):  Upper bound  — "TỐI ĐA chậm bằng này"
            f(n) ≤ c × g(n) cho n đủ lớn

Big-Ω (Ω): Lower bound  — "TỐI THIỂU nhanh bằng này"
            f(n) ≥ c × g(n) cho n đủ lớn

Big-Θ (Θ): Tight bound  — "CHÍNH XÁC bằng này" (cả O và Ω)
            c₁ × g(n) ≤ f(n) ≤ c₂ × g(n) cho n đủ lớn

Ví dụ: Merge Sort
  O(n log n)  — worst case không vượt quá n log n ✓
  Ω(n log n)  — best case cũng ít nhất n log n ✓
  Θ(n log n)  — chính xác n log n (tight bound) ✓

Phỏng vấn: Hầu hết dùng Big-O. Nếu interviewer hỏi "tight bound?" → dùng Big-Θ.
```

## 8. Phân Tích Code Thực Tế

```javascript
// Ví dụ 1: O(n) hay O(n²)?
function example1(arr) {
  const set = new Set(arr); // O(n) — build set
  for (const item of arr) {
    // O(n) loop
    if (set.has(item * 2)) {
    } // O(1) lookup
  }
}
// O(n) + O(n × 1) = O(n) ✓

// Ví dụ 2: Tricky nested loop
function example2(n) {
  for (let i = 1; i < n; i *= 2) {
    // O(log n) — i doubles
    for (let j = 0; j < n; j++) {
      // O(n)
      // ...
    }
  }
}
// O(n log n) — log n outer iterations × n inner iterations

// Ví dụ 3: String concatenation trap!
function example3(n) {
  let s = "";
  for (let i = 0; i < n; i++) {
    s += "a"; // ⚠️ String immutable! Creates new string mỗi lần!
    //          Copy 1 + 2 + 3 + ... + n chars = O(n²)!
  }
}
// O(n²) — NOT O(n)! Fix: dùng Array.join() hoặc StringBuilder

// Ví dụ 4: Recursive analysis
function example4(n) {
  if (n <= 0) return;
  example4(n - 1);
  example4(n - 1);
}
// T(n) = 2T(n-1) + O(1)
// Mỗi level gấp đôi calls: 1 → 2 → 4 → 8 → ... → 2^n
// Total: O(2^n)
```

## 9. Câu Hỏi Phỏng Vấn

**Q: O(n) vs O(n log n) — khác biệt thực tế bao nhiêu?**

> A: Với n = 10^6: O(n) = 10^6 operations, O(n log n) = 2×10^7 (≈20x chậm hơn). Với n = 10^9: O(n) = 10^9, O(n log n) = 3×10^10 (≈30x). Trong thực tế, O(n log n) vẫn rất nhanh. Sự khác biệt thực sự quan trọng là O(n) vs O(n²): với n = 10^6, O(n²) = 10^12 — chênh 1 TRIỆU lần!

**Q: Amortized O(1) khác O(1) worst case thế nào?**

> A: O(1) worst case = LUÔN O(1), mọi operation. Amortized O(1) = trung bình O(1) over n operations, nhưng 1 operation CÓ THỂ là O(n) (ví dụ: array resize). Vấn đề: real-time systems (latency-sensitive) không chấp nhận amortized — 1 request chậm 100ms trong khi 99 requests nhanh 1μs vẫn là BUG.

**Q: Tại sao phỏng vấn hỏi time + space complexity?**

> A: Trade-off! Nhiều bài: giảm time complexity → tăng space (dùng hash map). Interviewer muốn biết bạn hiểu trade-off và chọn phù hợp context. Ví dụ: mobile app (ít RAM) → ưu tiên space. Backend batch job → ưu tiên time.
