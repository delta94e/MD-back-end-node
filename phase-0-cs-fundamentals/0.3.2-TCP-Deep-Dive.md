## 1. TCP Là Gì

TCP cung cấp **reliable, ordered, error-checked** delivery giữa 2 applications qua network.

```
TCP Segment Header (20 bytes minimum):
┌───────────────────────────────────────────────────┐
│  Source Port (16-bit)  │  Dest Port (16-bit)      │
├───────────────────────────────────────────────────┤
│            Sequence Number (32-bit)               │
├───────────────────────────────────────────────────┤
│         Acknowledgment Number (32-bit)            │
├───────────────────────────────────────────────────┤
│ Data │ Reserved │ Flags │   Window Size (16-bit)  │
│Offset│          │ UAPRSF│                         │
├───────────────────────────────────────────────────┤
│   Checksum (16-bit)    │  Urgent Pointer (16-bit) │
├───────────────────────────────────────────────────┤
│              Options (variable)                   │
└───────────────────────────────────────────────────┘

Flags: URG, ACK, PSH, RST, SYN, FIN
```

## 2. 3-Way Handshake

Thiết lập TCP connection:

```
  Client                                    Server
    │                                         │
    │  ──── SYN (seq=x) ──────────────────▶  │  Step 1: "Tôi muốn kết nối"
    │                                         │           Client chọn seq number ngẫu nhiên
    │  ◀──── SYN-ACK (seq=y, ack=x+1) ────  │  Step 2: "OK, tôi cũng muốn kết nối"
    │                                         │           Server chọn seq riêng, ACK seq client
    │  ──── ACK (ack=y+1) ────────────────▶  │  Step 3: "Xác nhận, bắt đầu gửi data"
    │                                         │
    │  ════ CONNECTION ESTABLISHED ═══════   │
    │                                         │

Tại sao 3-way, không phải 2-way?
  → 2-way: Server không biết Client đã nhận SYN-ACK chưa
  → 3-way: Cả hai bên xác nhận khả năng gửi VÀ nhận
  → Ngăn chặn "stale connections" (SYN cũ đến trễ)
```

### RTT và Latency

```
1 RTT = Round Trip Time = thời gian packet đi + về

3-way handshake = 1.5 RTT (SYN → SYN-ACK → ACK)
Nếu RTT = 50ms → mất 75ms CHỈ ĐỂ KẾT NỐI (chưa gửi data!)
HTTP/1.1: 1 connection / request → handshake mỗi lần = chậm!
HTTP/1.1 keep-alive: reuse connection → handshake 1 lần
HTTPS: TCP handshake (1.5 RTT) + TLS handshake (1-2 RTT) = ~150-200ms!
```

## 3. 4-Way Teardown

```
  Client                                    Server
    │                                         │
    │  ──── FIN (seq=u) ──────────────────▶  │  Step 1: Client muốn đóng
    │                                         │
    │  ◀──── ACK (ack=u+1) ────────────────  │  Step 2: Server ACK
    │                                         │  (Server vẫn có thể gửi data!)
    │    HALF-CLOSED state                    │
    │                                         │
    │  ◀──── FIN (seq=v) ────────────────── │  Step 3: Server cũng muốn đóng
    │                                         │
    │  ──── ACK (ack=v+1) ────────────────▶  │  Step 4: Client ACK
    │                                         │
    │  ════ CONNECTION CLOSED ════════════   │

Tại sao 4-way, không phải 3-way?
  → TCP là full-duplex (gửi/nhận độc lập 2 chiều)
  → Client đóng chiều gửi ≠ Server đóng chiều gửi
  → Cần 2 FIN + 2 ACK riêng biệt
```

### TIME_WAIT State

```
Sau khi gửi ACK cuối, Client vào TIME_WAIT (2×MSL ≈ 60-120 giây).

Tại sao?
  1. Đảm bảo ACK cuối cùng đến Server (nếu mất → Server gửi lại FIN)
  2. Ngăn chặn delayed packets từ connection cũ lẫn vào connection mới

Vấn đề thực tế:
  Server đóng nhiều connections → hàng nghìn TIME_WAIT sockets
  → Fix: SO_REUSEADDR option, tcp_tw_reuse (Linux)
```

## 4. Reliability Mechanisms

### 4.1 Sequence Numbers & ACK

```
Mỗi byte data được đánh số (sequence number).
Receiver gửi ACK = "Tôi đã nhận đến byte X, gửi tiếp byte X+1 đi"

Sender:  [seq=1000, 500 bytes data] ──────▶
Receiver: ◀────── [ack=1500]              "Đã nhận 1000-1499, gửi 1500 tiếp"
Sender:  [seq=1500, 500 bytes data] ──────▶
Receiver: ◀────── [ack=2000]              "Đã nhận 1500-1999, gửi 2000 tiếp"
```

### 4.2 Retransmission — Gửi lại khi mất

```
Sender:  [seq=1000] ──────▶  ✓ nhận
Sender:  [seq=1500] ──────▶  ✗ MẤT!
Sender:  [seq=2000] ──────▶  ✓ nhận
Receiver: ◀── [ack=1500]               "Vẫn chờ 1500!"
Receiver: ◀── [ack=1500]               "Duplicate ACK!"
Receiver: ◀── [ack=1500]               "Triple duplicate ACK!"

Sender nhận 3 duplicate ACKs → FAST RETRANSMIT [seq=1500]
(Không cần đợi timeout — nhanh hơn nhiều!)
```

### 4.3 Checksum — Phát hiện data bị hỏng

TCP tính checksum trên toàn bộ segment. Receiver verify → nếu sai → drop silently → Sender timeout → retransmit.

## 5. Flow Control — Sliding Window

Ngăn sender gửi quá nhanh **làm overwhelm receiver**.

```
Receiver thông báo: "Window Size = 4KB" (tôi còn 4KB buffer trống)

                    Window = 4KB
          ┌──────────────────────────────┐
  ────────┤ Sent, ACKed │ Sent, not ACKed │ Can send │ Cannot send│─────
          └──────────────────────────────┘
                 ◀─── Sliding ──▶

Khi receiver xử lý data → buffer trống → gửi ACK với window lớn hơn
Khi receiver bận → window = 0 → SENDER DỪNG! (Zero Window)

→ Sender periodically probe (Window Probe) để check khi nào gửi tiếp
```

## 6. Congestion Control

Ngăn sender gửi quá nhanh **làm overwhelm NETWORK** (routers, links).

### 4 Phases:

```
Throughput
 │                                    ┌─── ssthresh
 │                               ┌───┤
 │                          ┌───┘    │  Congestion Avoidance
 │                     ┌───┘         │  (tăng linear: +1 MSS/RTT)
 │                ┌───┘              │
 │           ┌───┘                   │
 │      ┌───┘                        ▼ Packet Loss!
 │  ┌──┘  Slow Start                 │
 │ ┘   (tăng exponential:            │ cwnd giảm 1/2 (Fast Recovery)
 │      doubling mỗi RTT)           │  hoặc reset = 1 MSS (Timeout)
 └───────────────────────────────────── Time

Phase 1: SLOW START
  cwnd = 1 MSS → 2 → 4 → 8 → 16 ... (exponential)
  Cho đến khi cwnd ≥ ssthresh → chuyển sang Congestion Avoidance

Phase 2: CONGESTION AVOIDANCE
  cwnd += 1 MSS mỗi RTT (linear)
  Tiếp tục cho đến khi packet loss

Phase 3: FAST RETRANSMIT / FAST RECOVERY (3 duplicate ACKs)
  ssthresh = cwnd / 2
  cwnd = ssthresh (không reset về 1!)
  TCP Reno: quay lại Congestion Avoidance

Phase 4: TIMEOUT (nghiêm trọng hơn)
  ssthresh = cwnd / 2
  cwnd = 1 MSS (reset hoàn toàn!)
  Quay lại Slow Start
```

## 7. Nagle's Algorithm & TCP_NODELAY

```
Vấn đề: Gửi nhiều packets nhỏ (ví dụ: 1 byte data + 40 bytes header)
         = rất lãng phí bandwidth ("small packet problem")

Nagle's Algorithm (default ON):
  Nếu có data chưa ACK → buffer data mới
  Chỉ gửi khi: (1) tích đủ MSS, hoặc (2) nhận ACK cho data trước

Vấn đề: tăng latency cho interactive applications!
  Keystroke 1: gửi ngay (1 byte)
  Keystroke 2: PHẢI ĐỢI ACK của keystroke 1! (+ RTT delay)

Fix: TCP_NODELAY = tắt Nagle → gửi ngay mọi data
```

```javascript
// Node.js: tắt Nagle cho real-time apps
const net = require("net");
const server = net.createServer((socket) => {
  socket.setNoDelay(true); // TCP_NODELAY = true
});

// HTTP server cũng có thể set:
const http = require("http");
const server = http.createServer((req, res) => {
  req.socket.setNoDelay(true);
  // ...
});
```

## 9. Liên Hệ Backend

| TCP Concept         | Backend Impact                                            |
| ------------------- | --------------------------------------------------------- |
| 3-way handshake     | Connection pooling (reuse connections, giảm latency)      |
| TIME_WAIT           | High-traffic servers: `SO_REUSEADDR`, tune `tcp_tw_reuse` |
| Nagle + Delayed ACK | Real-time apps: `TCP_NODELAY` bắt buộc                    |
| Congestion control  | SSH/Database connections over WAN = slow start penalty    |
| Window size         | Upload/download large files → tune `tcp_rmem`, `tcp_wmem` |
| Keep-alive          | HTTP keep-alive, database connection pools                |

## 10. Câu Hỏi Phỏng Vấn

**Q: Tại sao TCP 3-way handshake, không phải 2-way?**

> A: 2-way không đủ vì: (1) Server không biết Client đã nhận SYN-ACK → có thể allocate resources cho connection "ma". (2) Stale SYN cũ (từ connection trước) có thể trigger connection mới sai. 3-way đảm bảo cả hai bên verify khả năng gửi VÀ nhận, đồng bộ sequence numbers.

**Q: Slow Start có thực sự "slow" không?**

> A: KHÔNG — "Slow Start" là tên gọi sai lệch. Nó tăng **exponential** (1→2→4→8→16 MSS), rất nhanh. Gọi "slow" vì so với thời kỳ trước khi TCP gửi full bandwidth ngay lập tức → gây congestion collapse. Slow Start vẫn ảnh hưởng performance: mỗi TCP connection mới bắt đầu từ cwnd=1 → cần ~log₂(BDP) RTTs để đạt full throughput. Connection pooling/keep-alive giải quyết vấn đề này.

**Q: TCP_NODELAY giải quyết vấn đề gì?**

> A: Tắt Nagle's algorithm — gửi data ngay lập tức mà không buffer. Cần cho: WebSocket messages, game server, real-time chat, database protocol (mỗi query cần gửi ngay). Tổ hợp Nagle + Delayed ACK gây "200ms delay" nổi tiếng.
